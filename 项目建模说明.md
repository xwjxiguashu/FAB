# FAB 机台组动态滚动调度项目建模说明

## 1. 项目定位与当前实现范围

本项目面向半导体 FAB 机台组调度问题，构建了一个能够生成 Lot 级和 Wafer 级排程结果的动态滚动调度原型。当前实现重点解决以下问题：

1. Lot 随时间动态到达；
2. 每个 Lot 在可加工 Machine 集合中选择设备；
3. 在选定 Machine 后选择对应 PPID 工艺路径；
4. 在 PPID 的各 Stage 中选择具体 Chamber/Side 资源；
5. 在滚动调度过程中冻结近期操作，保持计划稳定；
6. 同时评价 Q-time 违背、拖期、优先级违背和机台利用率等多目标指标；
7. 使用 NSGA-II 对策略权重进行多目标进化搜索。

当前代码实现的是一个**确定性动态滚动调度 + 策略权重进化优化**框架。项目说明中涉及的多场景 CVaR 鲁棒优化、神经网络策略和强化学习训练，目前应作为后续扩展方向，而不是当前已完成实现。

---

## 2. 核心对象与输入数据

### 2.1 Lot

Lot 是外层排产对象，表示一批需要完成相同或相近工艺的晶圆集合。每个 Lot 在模型中包含：

- Lot 编号；
- Wafer 数量；
- 到达时间 `arrival_times`；
- 交期 `due_dates`；
- 优先级 `priorities`；
- 可加工 Machine 集合 `feasible_machines`。

Lot 的实际开工时间不得早于自身到达时间，也不得早于当前滚动调度时刻。

### 2.2 Machine

Machine 表示机台组中的加工设备。每个 Lot 可以在一个或多个可行 Machine 上加工。当前模型在 Lot 级别维护 Machine 日历，同一台 Machine 在同一时刻最多被一个 Lot 占用。

### 2.3 PPID

PPID 表示某个 Lot 在某台 Machine 上可选的工艺路径。对于同一个 Lot：

- 不同 Machine 可以对应不同 PPID 集合；
- 同一 Machine 下也可以存在多条 PPID；
- PPID 由多个顺序执行的 Stage 组成。

当前输入通过 `feasible_ppids[(lot, machine)]` 给出可选 PPID，通过 `ppid_steps[(lot, machine, ppid)]` 给出具体 Stage 结构。

### 2.4 Stage 与 Chamber/Side

每条 PPID 由若干 Stage 组成，每个 Stage 对应多个候选 Chamber/Side 资源。候选资源格式为：

```text
[chamber, side, process_time]
```

其中 `process_time` 是该 Stage 在该 Chamber/Side 上的确定性加工时间。当前模型在 Wafer 级别维护 Chamber/Side 日历，同一 `(machine, chamber, side)` 资源在同一时刻最多处理一个 Wafer 工序。

### 2.5 Wafer

Wafer 是 Lot 内部的实际加工单元。一个 Lot 包含多片 Wafer，每片 Wafer 按所选 PPID 的 Stage 顺序加工，并逐 Stage 占用具体 Chamber/Side 资源。

因此项目同时输出两类排程：

1. **Lot 级排程**：`[lot, machine, ppid, start_time, end_time]`；
2. **Wafer 级排程**：`[lot, wafer_id, machine, ppid, stage_id, chamber, side, start_time, end_time]`。

## 3. 四层决策结构

当前调度决策可以理解为四层结构：

1. **Lot Sequence / Lot Priority 层**：决定当前窗口内下一个优先安排哪个 Lot；
2. **Machine Selection 层**：为该 Lot 选择具体 Machine；
3. **PPID Selection 层**：在选定 Machine 后选择具体 PPID；
4. **Chamber/Side Selection 层**：为每片 Wafer 的每个 Stage 选择 Chamber/Side 资源。

与传统显式染色体编码不同，当前实现采用的是**策略权重编码**：

- `priority_gene` 用于 Lot 选择打分；
- `resource_gene["plan_weights"]` 用于 Machine/PPID 方案选择打分；
- `resource_gene["stage_weights"]` 用于 Chamber/Side 资源选择打分。

NSGA-II 进化的是这些策略权重，而不是直接进化完整排程表。每个候选策略在解码阶段根据当前资源状态逐步生成可执行排程。

---

## 4. 策略特征与打分机制

### 4.1 Lot 选择特征

当前 Lot 选择使用以下特征：

1. `due_urgency`：交期紧迫度；
2. `priority`：Lot 优先级；
3. `waiting_time`：已等待时间；
4. `arrival_readiness`：是否已经到达；
5. `wafer_count`：Wafer 数量；
6. `fastest_process_time`：估计最快总加工时间；
7. `machine_load`：可选 Machine 中的较优负载状态；
8. `qtime_pressure`：估计 Q-time 压力。

特征会先归一化，然后与 `priority_gene` 点乘得到 Lot 分数。分数最高的 Lot 被优先安排。若某个 Lot 已经有冻结操作，则其 Machine/PPID 决策被锁定，并在后续选择中获得极高优先级，保证已冻结计划能够延续。

### 4.2 Machine/PPID 方案选择特征

Machine/PPID 方案选择使用以下特征：

1. `machine_available_time`：Machine 可插入时间；
2. `machine_calendar_load`：Machine 当前负载；
3. `total_process_time`：估计总加工时间；
4. `predicted_completion`：估计完成时间；
5. `stage_count`：Stage 数量；
6. `qtime_risk`：估计 Q-time 风险。

每个候选 Machine/PPID 方案形成一个特征向量，归一化后与 `resource_gene["plan_weights"]` 点乘，选择得分最高的方案。

### 4.3 Chamber/Side 资源选择特征

Stage 级资源选择使用以下特征：

1. `resource_earliest_start`：该资源最早可开工时间；
2. `resource_finish_time`：预计完成时间；
3. `process_time`：加工时间；
4. `chamber_load`：资源负载；
5. `balance`：资源当前已安排区间数量。

每个候选 Chamber/Side 资源归一化后与 `resource_gene["stage_weights"]` 点乘，选择得分最高的资源。

## 5. 日历解码与可行排程生成

当前项目使用基于资源日历的解码器生成可执行排程。

### 5.1 Machine 日历

Machine 日历记录每台 Machine 已被 Lot 占用的时间区间。模型通过插入区间并检查重叠，保证同一台 Machine 不会同时处理多个 Lot。

### 5.2 Chamber/Side 日历

Chamber/Side 日历记录每个 `(machine, chamber, side)` 资源已被 Wafer Stage 占用的时间区间。模型通过查找最早可插入空隙，保证同一细粒度资源不会发生时间冲突。

### 5.3 Lot 与 Wafer 的一致性

对一个 Lot 解码时，模型先选择 Machine 和 PPID，然后对该 Lot 内所有 Wafer 逐片、逐 Stage 安排 Chamber/Side 资源。Lot 的开始时间为该 Lot 所有 Wafer Stage 的最早开始时间，结束时间为所有 Wafer Stage 的最晚结束时间。

由于 Lot 级 Machine 占用区间需要覆盖该 Lot 内所有 Wafer 的实际加工区间，解码器会迭代调整 Lot 的 Machine 插入位置，直到 Machine 日历和 Chamber/Side 日历之间达到一致。

## 6. 动态到达与滚动调度机制

### 6.1 动态到达

每个 Lot 具有到达时间 `arrival_times[lot]`。滚动调度初始时刻为所有 Lot 中最早到达时间。每个 Lot 只有满足以下条件时才可能被纳入当前窗口：

```text
arrival_time <= current_time + lookahead_window
```

但即使未来到达的 Lot 进入前瞻窗口，其实际开工时间仍不得早于自身到达时间。

### 6.2 滚动调度流程

每一轮滚动调度包含以下步骤：

1. 根据当前时间和冻结窗口确定 `freeze_until = current_time + freeze_window`；
2. 将上一轮计划中进入冻结窗口的 Wafer 操作固化；
3. 根据固定 Lot/Wafer 排程恢复 Machine 与 Chamber/Side 资源状态；
4. 根据前瞻窗口选择当前待优化 Lot 集合；
5. 使用 NSGA-II 搜索一组策略权重并生成 Pareto Front；
6. 从 Pareto Front 中选择一个执行方案；
7. 保存本轮历史记录；
8. 当前时间推进 `rolling_interval`，进入下一轮。

### 6.3 冻结机制

冻结机制用于保持滚动调度稳定性。一旦某些 Wafer Stage 进入冻结窗口，其开始时间、结束时间、Machine、PPID、Chamber 和 Side 均固定。若一个 Lot 已经存在冻结操作，则后续轮次会锁定该 Lot 原有 Machine/PPID 决策，只继续安排未冻结部分。

---

## 7. 约束条件

当前实现显式处理以下约束：

1. **到达时间约束**：Lot 开工时间不得早于到达时间和当前滚动时刻；
2. **Machine 容量约束**：同一 Machine 时间区间不得重叠；
3. **Chamber/Side 容量约束**：同一 `(machine, chamber, side)` 时间区间不得重叠；
4. **Wafer 工艺顺序约束**：同一 Wafer 的 Stage 必须按 PPID 顺序依次加工；
5. **工艺可行性约束**：Lot 只能选择可行 Machine、可行 PPID 和对应 Stage 的候选 Chamber/Side；
6. **冻结约束**：进入冻结窗口的 Wafer 操作不可移动；
7. **锁定约束**：已有冻结操作的 Lot 后续必须沿用原 Machine/PPID；
8. **Q-time 评价约束**：支持对指定 `(lot, machine, ppid, from_stage, to_stage)` 计算 Q-time 违背。

其中 Q-time 当前作为目标函数中的违背指标进行评价，而不是在解码阶段作为硬约束强制禁止。

---

## 8. 目标函数

当前目标函数返回 6 维最小化目标向量：

```text
[
  q_time_violation_count,
  total_q_time_violation,
  tardy_count,
  total_tardiness,
  priority_violation,
  -avg_machine_utilization
]
```

各目标含义如下：

1. **Q-time 违背次数**：超过 Q-time 限制的 Wafer 约束数量；
2. **Q-time 总违背时间**：所有 Q-time 超限时间之和；
3. **拖期 Lot 数**：完成时间超过交期的 Lot 数量；
4. **总拖期时间**：所有 Lot 拖期时间之和；
5. **优先级违背程度**：高优先级 Lot 被低优先级 Lot 延后的程度；
6. **负平均机台利用率**：利用率越高越好，因此取负值转化为最小化目标。

机台利用率当前基于 Lot 级 Machine 占用区间计算，Chamber/Side 利用率当前主要通过资源日历保证可行性，并未单独作为优化目标。

---

## 9. NSGA-II 多目标策略搜索

当前项目使用 NSGA-II 对策略权重进行多目标优化。

### 9.1 个体表示

每个个体包含：

- `priority_gene`：Lot 选择权重；
- `resource_gene["plan_weights"]`：Machine/PPID 选择权重；
- `resource_gene["stage_weights"]`：Chamber/Side 选择权重；
- 解码得到的 Lot 级排程；
- 解码得到的 Wafer 级排程；
- 6 维目标向量；
- 决策轨迹 `decision_trace`。

### 9.2 进化操作

当前实现包含：

1. 随机初始化策略权重；
2. 解码并评价每个个体；
3. 快速非支配排序；
4. 拥挤距离计算；
5. 锦标赛选择；
6. 权重交叉；
7. 权重变异；
8. 环境选择生成下一代。

### 9.3 Pareto 解选择

每一轮滚动优化后，系统从 Pareto Front 中选择一个实际执行方案。当前支持：

1. `lexicographic`：按目标向量字典序选择；
2. `tardy_then_utilization`：优先拖期相关指标，再考虑利用率；
3. `weighted_sum`：按用户给定权重做加权和选择。

---

## 10. 输出结果与实验脚本

当前项目提供小、中、大三类示例脚本：

1. `run_small_instance_gantt.py`：小规模手工实例，包含显式 Q-time 限制；
2. `run_medium_uncertainty_gantt.py`：中等规模随机生成实例，名称中包含 uncertainty，但当前代码仍为确定性加工时间评价；
3. `run_large_instance_gantt.py`：较大规模随机生成实例，用于测试滚动调度和并行评价。

每次运行会输出：

- Lot 级 CSV 排程；
- Wafer 级 CSV 排程；
- Lot 级甘特图；
- Wafer 级紧凑甘特图；
- 运行时间、滚动轮数、目标函数值等摘要信息。

---

## 11. 当前实现边界

为了避免混淆，当前项目尚未实现以下功能：

1. **多场景加工时间扰动评价**：当前 `process_time` 作为确定性加工时间使用；
2. **CVaR 鲁棒目标聚合**：当前目标函数未对多场景结果做 mean + CVaR 聚合；
3. **神经网络策略**：当前 Lot、Machine/PPID、Chamber/Side 选择均为手工特征加线性权重打分；
4. **强化学习训练**：当前没有环境交互式 RL 训练过程；
5. **局部搜索/VNS**：`local_search.py` 中相关方法目前显式标记为未实现；
6. **Q-time 硬约束修复**：当前 Q-time 作为评价目标，未在解码时强制禁止违背；
7. **Chamber/Side 利用率目标**：当前只优化 Machine 平均利用率。

这些内容可以作为后续算法创新方向。

## 12. 建议的后续扩展方向

结合当前项目结构，后续最自然的创新方向是：

### 12.1 学习增强策略调度

保留当前日历解码器和可行性检查机制，将线性策略权重替换为神经网络策略：

- Lot 选择网络；
- Machine/PPID 选择网络；
- Chamber/Side 资源选择网络。

可以先利用当前 NSGA-II 产生的优秀 `decision_trace` 做模仿学习，再通过强化学习微调策略。

### 12.2 多场景鲁棒评价

在 `evaluate_policy_objectives` 外层增加多场景加工时间扰动，对同一策略在多个场景下重复解码，然后将目标聚合为：

```text
robust_objective = mean(objective) + lambda * CVaR_alpha(objective)
```

这样可以把当前确定性策略搜索扩展为加工时间不确定下的鲁棒调度。

### 12.3 Q-time 硬约束与修复机制

当前 Q-time 是软目标。后续可以在资源选择或解码回溯阶段加入 Q-time 可行性过滤，使严重 Q-time 违背的候选资源不被选择，或在违背后触发局部修复。

### 12.4 更完整的实验体系

建议后续实验增加：

1. 不同 Lot 数量：10、30、50、100；
2. 不同 Machine/Chamber/Side 规模；
3. 不同到达强度；
4. 不同 Q-time 紧张程度；
5. 不同加工时间扰动幅度；
6. 与 FIFO、EDD、SPT、CR、当前线性策略 NSGA-II、学习策略等方法对比。

---

## 13. 总结

当前项目已经形成了一个较完整的半导体 FAB 机台组动态滚动调度原型。其核心特点是：

1. 面向 Lot、Machine、PPID、Chamber/Side 的层次化调度建模；
2. 同时输出 Lot 级和 Wafer 级可执行排程；
3. 使用资源日历保证 Machine 与 Chamber/Side 不冲突；
4. 支持动态到达、前瞻窗口和冻结窗口；
5. 支持 Q-time、拖期、优先级和利用率的多目标评价；
6. 使用 NSGA-II 搜索可解释的线性策略权重。

从论文创新角度看，当前项目的下一步重点不应再只是增加基础约束，而应在现有框架上引入**神经网络/强化学习策略、鲁棒多场景评价和 Q-time 可行性修复机制**，形成学习增强的半导体 FAB 动态滚动调度方法。
