# FAB_RL 阶段2：规则触发 SAS-PPO 项目方案

## 1. 目标与边界

本方案对应 [报告.md](报告.md) 第10部分的“阶段2：规则触发的 SAS-PPO”代码实现，目标是在现有阶段1资源日历环境之上，完成一个**最小闭环版**的强化学习调度原型。

本阶段只验证以下闭环是否成立：

1. 规则触发当前 Machine 的 SAS 决策；
2. 基于固定长度候选池选择 `action_index`；
3. 通过 action mask 过滤结构性不可行动作；
4. 调用资源日历环境执行 `sas_step`；
5. 收集 rollout；
6. 使用 PPO 更新 Actor-Critic；
7. 在小规模实例上跑通训练与推理。

本阶段明确**不做**：

- DDT 决策时间智能体；
- 单注意力 / 双注意力结构；
- 大规模实验对比框架；
- 完整论文级评估系统；
- 复杂 reward 设计的全面展开。

因此，阶段2的核心是：

> 先打通“候选池 + mask + 资源日历 + 简化 reward + PPO”的稳定闭环，而不是一步做到最终 DDA-PPO。

### 1.1 与报告第10段的对齐说明

本方案严格对齐 [报告.md](报告.md) 第10段"阶段 2：规则触发的 SAS-PPO"的约束边界。以下列出报告原文要点与本方案的映射：

| 报告原文要点 | 本方案对应位置 |
|---|---|
| 当某台 Machine 空闲且候选池非空时立即触发 | 5.1 节 |
| 不学习"是否等待" | 5 节整体 |
| Actor 输出候选池索引 `action_index` | 7.1 节 |
| 训练阶段保持采样轨迹一致性，插入失败只返回惩罚 | 9 节第 7-8 步 |
| 推理阶段按概率从高到低尝试，取第一个可执行方案作 fallback | 10.8 节 |
| 评估指标：Q-time 违背、拖期 Lot 数、总拖期时间、机器利用率、优先级违背 | 13.2 节 |
| 拖期、Q-time、priority、进度只作为轻量 shaping，基础策略收敛后逐步加入 | 8.2 节 |

补充对齐要求：

- `RolloutStep` 必须记录 `machine_id` 与 `info`，否则无法从训练轨迹追溯当前动作对应的 Machine、Lot、PPID、插入结果和 reward 分解。
- Machine 选择规则采用报告第9.1节推荐的确定性受限优先规则：`machine_t = argmin_m (next_available_time_m, num_feasible_candidates_m, machine_id_m)`。
- Episode driver 必须显式维护 `total_wait_steps_per_episode`、`consecutive_failed_actions` 和 `unrecoverable_error`，并在终止时给出明确 `termination_reason`。
- Buffer 内部可以存储拆分后的 `candidate_features`、`candidate_mask` 和 `global_features`，但文档上必须把这些字段视为 `obs_t` 的展开形式，并保留 `next_observation` 对齐报告中的 `obs_{t+1}`。

## 3. 总体架构

阶段2建议拆成五个逻辑模块：

```text
问题实例 / Encoder
        ↓
ResourceCalendarEnv
        ↓
Phase2EpisodeDriver
        ↓
ObservationEncoder
        ↓
SAS Actor-Critic
        ↓
RolloutBuffer + PPOTrainer
        ↓
Train / Inference Entry
```

各模块职责如下。

### 3.1 环境层

由 [FAB_RL/FABenv/rl_environment.py](FAB_RL/FABenv/rl_environment.py) 提供。

职责：

- 管理 Machine / Chamber / Side 资源日历；
- 生成候选池与 mask；
- 执行动作 dry-run / commit；
- 计算 SAS 单步 reward；
- 输出状态、schedule 与校验结果。

### 3.2 调度驱动层

新增 `phase2_sas_driver.py`。

职责：

- 决定当前是否存在需要触发 SAS 决策的 Machine；
- 决定在多个可调度 Machine 中先处理哪一台；
- 在无动作时推进时间到下一事件；
- 驱动一个完整 episode 的执行流程。

### 3.3 观测编码层

新增 `phase2_sas_observation.py`。

职责：

- 将候选池和环境摘要状态编码为网络输入；
- 生成固定形状的 `candidate_features`、`candidate_mask`、`global_features`；
- 为 PPO 训练提供稳定的输入格式。

### 3.4 策略与价值网络层

新增 `phase2_sas_policy.py`。

职责：

- 使用共享 MLP 对候选动作打分；
- 基于 mask 生成合法动作概率；
- 输出状态价值估计；
- 为 PPO 更新提供 `log_prob`、`entropy`、`value`。

### 3.5 训练层

新增 `phase2_ppo_buffer.py`、`phase2_ppo_trainer.py`。

职责：

- 存储 episode 轨迹；
- 计算 return 和 advantage；
- 执行 PPO clipped objective 更新；
- 管理训练循环与日志输出。

## 5. 规则触发机制设计

阶段2必须遵守 [报告.md](报告.md) 中“规则触发的 SAS-PPO”的边界：

- 当前阶段**不学习是否等待**；
- 只有当某台 Machine 需要派工时，SAS 才进行候选池动作选择；
- SAS 只负责决定“当前候选池中选择哪个 `(Lot, PPID)`”。

### 5.1 触发规则

报告原文触发条件："当某台 Machine 空闲且候选池非空时立即触发 SAS 决策。"

具体实现：

1. 当前时刻该 Machine 空闲（无在加工 Lot）；
2. 基于 `build_candidate_pool(machine)` 能生成至少一个有效候选动作（即候选池非空）。

### 5.2 多台 Machine 同时可调度时的规则

为避免阶段2引入组合动作空间，阶段2仍然使用固定规则逐台决策，但该规则应与报告第9.1节保持一致：

```text
machine_t = argmin_m (next_available_time_m, num_feasible_candidates_m, machine_id_m)
```

含义如下：

1. `next_available_time_m` 优先保证最早空闲的 Machine 先处理；
2. `num_feasible_candidates_m` 在同一空闲时刻优先处理候选动作更少、更受限的 Machine；
3. `machine_id_m` 仅作为 tie-breaker，保证训练和推理可复现。

该规则比单纯按 Machine 编号排序更贴近报告中的 SAS 单训设定，因为它优先处理当前调度状态中更容易失去可行窗口的 Machine，同时仍然避免组合动作空间。

### 5.3 无动作时的处理

如果当前时刻所有 Machine 都没有有效候选动作，则不调用 Actor，而是将环境推进到下一个事件时刻：

- 下一个 Lot arrival time；
- 或下一个资源释放相关时间点。

因此，阶段2的时间推进不由学习控制，而由规则控制。

### 5.4 Episode 终止条件

阶段2的 episode 必须显式返回 `done` 和 `termination_reason`。终止条件与报告第9.8节保持一致：

1. 所有 Lot 均已完成 Lot-level 与 wafer-level 排程；
2. `current_time` 超过当前 planning horizon，且没有可继续调度的已到达 Lot；
3. 候选池为空、未来无 Lot 到达、未来无 Machine 释放，环境无法继续推进；
4. `total_wait_steps_per_episode > max_total_wait_steps_per_episode`；
5. `consecutive_failed_actions > max_failed_actions`，其中 `max_failed_actions` 推荐默认取 `3 × K_action`；
6. 出现资源日历不可恢复错误，例如已提交区间冲突、Lot / wafer stage 缺失或状态回滚失败。

`Phase2EpisodeDriver` 应维护以下计数与状态：

```text
total_wait_steps_per_episode
consecutive_failed_actions
unrecoverable_error
termination_reason
```

每次成功插入动作后，`consecutive_failed_actions` 归零；每次 mask invalid、dry-run 失败或 commit rollback 后，`consecutive_failed_actions` 加一；每次 wait / no-op 后，`total_wait_steps_per_episode` 加一。

## 6. 观测设计

阶段2最小闭环版不需要完整图结构表示，直接使用**候选池特征 + 全局摘要特征**即可。

### 6.1 候选动作特征

候选动作特征直接复用 [FAB_RL/FABenv/rl_environment.py](FAB_RL/FABenv/rl_environment.py) 中 `CandidatePool.features` 的输出格式。

当前已有特征名：

```text
is_real
is_wait
score
arrival_time
waiting_time
machine_slot_start
machine_load
total_process_time
predicted_completion
stage_count
qtime_risk
wafer_count
priority
due_slack
priority_rank_norm
due_slack_rank_norm
is_best_priority
is_most_urgent_due
```

其中前 14 个为阶段1已有候选特征，后 4 个为阶段2建议新增的候选池内相对排序特征。它们不改变动作空间，也不改变资源日历执行逻辑，只增强 MLP Actor 对候选之间相对关系的感知能力。

### 6.1.1 候选池 rank / best 标记

阶段2 Actor 仍使用共享 MLP，对每个候选动作独立打分。由于 MLP 本身不显式比较候选之间的相对关系，建议在 `CandidatePool.features` 中新增 4 个轻量 rank 特征：

```text
priority_rank_norm
due_slack_rank_norm
is_best_priority
is_most_urgent_due
```

计算规则：

1. 只对真实有效候选动作计算 rank，即 `is_real = 1`、`is_wait = 0`、`candidate_mask = True` 的候选；
2. `wait / no-op` 不参与 rank，四个新增特征均为 `0.0`；
3. padding 不参与 rank，整行特征保持 `0.0`；
4. `priority_rank_norm` 按 priority 从高到低排序，最高优先级候选为 `1.0`，最低为 `1 / n_valid_real`；
5. `due_slack_rank_norm` 按 due_slack 从小到大排序，最紧急候选为 `1.0`，最不紧急为 `1 / n_valid_real`；
6. `is_best_priority = 1.0` 仅标记 priority 最高的真实候选；如并列，则所有并列最高候选均可标记为 `1.0`；
7. `is_most_urgent_due = 1.0` 仅标记 due_slack 最小的真实候选；如并列，则所有并列最紧急候选均可标记为 `1.0`。

示例：

```text
候选 A: priority=10, due_slack=300 -> priority_rank_norm=1.00, due_slack_rank_norm=0.50, is_best_priority=1, is_most_urgent_due=0
候选 B: priority=8,  due_slack=20  -> priority_rank_norm=0.75, due_slack_rank_norm=1.00, is_best_priority=0, is_most_urgent_due=1
候选 C: priority=3,  due_slack=50  -> priority_rank_norm=0.50, due_slack_rank_norm=0.75, is_best_priority=0, is_most_urgent_due=0
候选 D: priority=1,  due_slack=500 -> priority_rank_norm=0.25, due_slack_rank_norm=0.25, is_best_priority=0, is_most_urgent_due=0
```

该优化属于低成本状态增强：只增加候选特征列，不修改 PPO、Actor-Critic、action mask、`sas_step(...)` 或 reward 语义。

### 6.2 全局摘要特征

建议新增少量稳定的全局特征：

- `current_time_norm`
- `completed_ratio`
- `remaining_ratio`
- `machine_id_norm`
- `machine_busy_time`
- `valid_action_count_norm`
- `candidate_score_mean`
- `candidate_waiting_time_max`
- `candidate_due_slack_min`

### 6.3 Observation 输出结构

建议统一编码为：

```python
@dataclass
class Phase2Observation:
    machine_id: int
    current_time: float
    candidate_features: np.ndarray   # [K, F]
    candidate_mask: np.ndarray       # [K]
    global_features: np.ndarray      # [G]
    action_indices: np.ndarray       # [K]
    valid_action_count: int
```

### 6.4 状态、观测、动作与信息边界

为了避免后续阶段把环境状态、网络输入和诊断信息混在一起，阶段2统一采用以下边界：

- `s_t`：环境完整状态，包括 `current_time`、Machine / Chamber / Side 日历、未完成 Lot 集合、已完成 Lot 集合和当前候选池生成所需的上下文；
- `obs_t = featurize(s_t)`：供策略网络输入的特征化观测，仅包含 `candidate_features`、`candidate_mask`、`global_features`、`action_indices`、`valid_action_count` 等摘要字段；
- `a_t`：策略对候选池索引 `action_index` 的选择，不直接暴露为 `(lot, machine, ppid)`；
- `info_t`：环境执行后的诊断结果与 reward 分解，例如 `selected_lot`、`selected_ppid`、`insertion_success`、`insertion_failed`、`mask_invalid`、`wait_or_noop`、`reward_*` 等，不作为策略输入；
- `s_{t+1}`：资源日历 commit / rollback 后得到的下一环境状态，`obs_{t+1}` 则是对 `s_{t+1}` 的再次特征化结果。

训练轨迹中保存的是 `obs_t -> a_t -> reward_t -> obs_{t+1}` 的闭环，`info_t` 仅用于日志与分析，不参与 PPO 概率比计算。

## 7. 策略网络设计

阶段2最小闭环版采用**最小 MLP Actor-Critic**，不引入注意力结构。

### 7.1 Actor

输入：

- `candidate_features: [K, F]`
- `candidate_mask: [K]`

结构：

1. 对每个 candidate 使用共享 MLP；
2. 输出每个 candidate 一个 logit；
3. 对 `mask=False` 的位置填充极小值；
4. 通过 softmax 得到合法动作分布。

### 7.1.1 Actor 公式

```text
h_i = MLP(candidate_features[i])
logit_i = Linear(h_i)
π(a_i | s) = MaskedSoftmax(logit_i, mask_i)
```

### 7.2 Critic

输入：

- `candidate_features`
- `candidate_mask`
- `global_features`

结构：

1. 对有效候选 embedding 做 masked pooling；
2. 与 `global_features` 拼接；
3. 经 MLP 输出标量 value。

### 7.2.1 Critic 公式

```text
candidate_repr = MaskedMeanPool(h_i)
state_repr = concat(candidate_repr, global_features)
V(s) = MLP(state_repr)
```

## 8. Reward 设计

阶段2 reward 必须保持简单，优先复用 [FAB_RL/FABenv/rl_environment.py](FAB_RL/FABenv/rl_environment.py) 中已有 `RewardConfig` 与 `compute_sas_reward(...)`。

### 8.1 总体公式与阶段开关

Reward 采用“执行结果为主、调度目标逐步加入”的 curriculum 设计。总公式为：

```text
r_t =
  r_execute
+ r_wait
+ lambda_tardy    * r_tardy
+ lambda_qtime    * r_qtime
+ lambda_priority * r_priority
+ lambda_progress * r_progress
+ r_terminal
```

阶段2实现时必须通过 `RewardConfig` 控制各项是否启用，而不是在训练代码中写死 reward 分支。推荐分三阶段启用：

```text
R0 合法性学习：r = r_execute + r_wait
R1 轻量调度偏好：r = r_execute + r_wait + light shaping
R2 最终指标对齐：r = r_execute + r_wait + light shaping + terminal reward
```

默认训练入口使用 R0，即 `use_light_shaping = False`、`use_terminal_reward = False`。只有当最小闭环可以稳定完成多个 episode、推理 schedule 可以通过 `validate_schedule()`、失败动作比例明显下降后，才开启 R1。R2 只用于后续对比实验，不作为阶段2最小闭环的默认目标。

### 8.2 R0：合法性学习 reward

先只开执行结果与 wait 惩罚：

```text
成功插入      +0.20
插入失败      -0.40
mask 无效动作 -0.50
wait / no-op  -0.02
```

其中，`wait / no-op` 只指策略主动选择等待或空操作；环境由于“当前无可调度 Machine”而推进到下一个事件时刻，不应额外视为策略错误，也不应重复叠加等待惩罚。

R0 的目标不是优化最终调度指标，而是让策略先学会：

1. 不选择 padding / masked action；
2. 优先选择资源日历可提交的候选动作；
3. 减少主动 wait / no-op；
4. 保持 PPO 采样轨迹与真实执行结果一致。

### 8.3 R1：轻量 shaping reward

初期默认：

- `use_light_shaping = False`
- `use_terminal_reward = False`

等最小闭环稳定后，再开启轻量 shaping。

在启用 shaping 后，各项建议保持如下语义：

- `tardy_norm`：结果性目标，只针对当前成功插入的 Lot 计算，表示本次动作后该 Lot 相对 due date 的归一化拖期量；
- `qtime_norm`：工艺风险项，表示本次动作新增的 Q-time 违背强度，建议用 `new_qtime_violation / selected_lot_process_time` 裁剪到 `[0, 1]`；
- `priority_rank_penalty`：局部派工偏好项，只比较当前候选池内的相对优先级，避免把已经完成或当前不可选 Lot 混入惩罚；
- `progress_norm`：非常小的稳定项，只鼓励成功推进排程，不应主导策略。

`progress_norm` 应始终保持比其它 shaping 项更弱的权重，避免模型偏向“加工更长但不一定更优”的 Lot。

推荐初始权重：

```text
tardy_weight    = -0.05
qtime_weight    = -0.08
priority_weight = -0.03
progress_weight = +0.01
```

注意：现有 `rl_environment.py` 中 `priority_rank_penalty` 目前基于已完成 Lot 计算，代码实现阶段应改为“当前候选池内相对优先级惩罚”。如果无法在 `compute_sas_reward(...)` 内访问候选池，则应在 `sas_step(...)` 组装 `info` 时预先写入 `priority_rank_penalty`。

### 8.4 R2：terminal reward

Terminal reward 只在 episode 结束时加入，用于让最终目标与报告第10段评估指标对齐。推荐公式：

```text
r_terminal =
 - 0.20 * tardy_lot_count_norm
 - 0.10 * total_tardiness_norm
 - 0.15 * qtime_violation_count_norm
 + 0.05 * machine_utilization_norm
 - 0.05 * priority_violation_norm
```

R2 不作为默认训练设置，原因是该信号延迟较长，早期容易造成 advantage 方差过大。开启条件建议为：

1. R0 可以稳定跑完多个 episode；
2. R1 下 reward 曲线不崩溃；
3. 推理 fallback 生成的 schedule 能稳定通过校验；
4. 已能输出 Q-time 违背、拖期 Lot 数、总拖期时间、机器利用率和优先级违背指标。

### 8.5 Reward 分解字段

每次 `sas_step(...)` 必须在 `info` 中记录 reward 分解，便于训练诊断和后续消融实验：

```text
reward_execute
reward_wait
reward_tardy
reward_qtime
reward_priority
reward_progress
reward_shape
reward_terminal
reward_total
```

其中：

- `reward_execute` 只记录成功插入、插入失败、mask invalid 这类执行结果项；
- `reward_wait` 只记录策略主动 wait / no-op 惩罚；
- `reward_shape = reward_tardy + reward_qtime + reward_priority + reward_progress`；
- `reward_total` 必须等于最终返回给 PPO 的 clipped reward；
- `info` 只用于日志、debug 和验收，不作为策略输入，不参与 PPO 概率比计算。

## 9. 训练与推理数据流

阶段2训练闭环建议如下：

```text
1. 创建 encoder
2. 创建 ResourceCalendarEnv
3. 创建 Phase2EpisodeDriver
4. 构建 obs_t：candidate_features、candidate_mask、global_features、action_indices、valid_action_count
5. policy 输出 masked action distribution
6. 训练时 sample action_index，并记录 logπ_old 与 value
7. 调用 env.sas_step(machine, action_index, pool, reward_config)
8. 记录 transition：
   - machine_id / current_time
   - obs_t 展开字段：candidate_features、candidate_mask、global_features、action_indices、valid_action_count
   - action_index / logπ_old / value / reward / done
   - obs_{t+1}：next_observation
   - info：selected_lot、selected_ppid、insertion_success、insertion_failed、mask_invalid、wait_or_noop、selected_lot_start、selected_lot_end、selected_lot_process_time、new_qtime_violation、priority_rank_penalty、reward_execute、reward_wait、reward_tardy、reward_qtime、reward_priority、reward_progress、reward_shape、reward_terminal、reward_total
9. 插入失败只返回失败惩罚，不临时替换为另一个动作
10. 若无调度动作则规则推进时间
11. episode 结束后计算 advantage / return
12. PPO update
13. 定期 greedy inference 验证
```

> **关键约束**（来自报告第10段）：训练阶段保持采样轨迹一致性——即使资源日历插入失败，也只返回失败惩罚，不临时替换为另一个动作。这保证了 PPO advantage 估计的无偏性。

### 10.5 `phase2_ppo_buffer.py`

```python
from dataclasses import dataclass, field
from typing import Any


@dataclass
class StepInfo:
    selected_lot: int | None = None
    selected_ppid: int | None = None
    insertion_success: bool = False
    insertion_failed: bool = False
    mask_invalid: bool = False
    wait_or_noop: bool = False
    selected_lot_start: float | None = None
    selected_lot_end: float | None = None
    selected_lot_process_time: float | None = None
    new_qtime_violation: float = 0.0
    priority_rank_penalty: float = 0.0
    reward_execute: float = 0.0
    reward_wait: float = 0.0
    reward_tardy: float = 0.0
    reward_qtime: float = 0.0
    reward_priority: float = 0.0
    reward_progress: float = 0.0
    reward_shape: float = 0.0
    reward_terminal: float = 0.0
    reward_total: float = 0.0
    raw: dict[str, Any] = field(default_factory=dict)


@dataclass
class RolloutStep:
    machine_id: int
    current_time: float
    candidate_features: object
    candidate_mask: object
    global_features: object
    action_indices: object
    valid_action_count: int
    action: int
    log_prob: float
    value: float
    reward: float
    done: bool
    next_observation: object | None
    info: StepInfo


class Phase2RolloutBuffer:
    def __init__(self, gamma: float = 0.99, gae_lambda: float = 0.95) -> None:
        self.gamma = gamma
        self.gae_lambda = gae_lambda
        self.steps: list[RolloutStep] = []
        self.returns: list[float] = []
        self.advantages: list[float] = []

    def add(self, step: RolloutStep) -> None:
        self.steps.append(step)

    def finish_episode(self, last_value: float = 0.0) -> None:
        self.compute_returns_and_advantages(last_value=last_value)

    def compute_returns_and_advantages(self, last_value: float = 0.0) -> None:
        returns: list[float] = []
        advantages: list[float] = []
        gae = 0.0
        next_value = last_value
        for step in reversed(self.steps):
            non_terminal = 0.0 if step.done else 1.0
            delta = step.reward + self.gamma * next_value * non_terminal - step.value
            gae = delta + self.gamma * self.gae_lambda * non_terminal * gae
            advantages.append(gae)
            returns.append(gae + step.value)
            next_value = step.value
        self.advantages = list(reversed(advantages))
        self.returns = list(reversed(returns))

    def get_training_batches(self, batch_size: int):
        raise NotImplementedError('Batch collation is implemented in Task 6 of the Phase 2 code plan')

    def clear(self) -> None:
        self.steps.clear()
        self.returns.clear()
        self.advantages.clear()
```

说明：

- `candidate_features`、`candidate_mask`、`global_features`、`action_indices` 和 `valid_action_count` 是报告中 `obs_t` 的展开形式；
- `next_observation` 对应报告中的 `obs_{t+1}`，terminal step 可记录为 `None`；
- `machine_id` 对应报告中的 `machine_t`；
- `info` 保留 reward 分解与资源日历执行结果，用于日志、debug 和验收；

```

### 推理 fallback 机制

报告第10段明确要求推理阶段支持 fallback：

> "推理阶段则可以按照 SAS 概率从高到低尝试候选动作，选择第一个资源日历可执行方案作为 fallback。"

具体流程：

```text
1. policy.greedy_action(...) 返回动作概率分布
2. 按概率从高到低排序候选动作
3. 依次调用 env.sas_step(machine, action_index, pool)
4. 若 sas_step 返回 committed=True，则采用该动作
5. 若所有候选均失败，则使用 wait/no-op 兜底
```

注意：**训练阶段不使用 fallback**，只有推理 / 部署阶段才使用此降级策略。这是"采样轨迹一致性"与"推理可用性"之间的明确分工。

## 13. 测试与验收标准

阶段2最小闭环版的验收重点不是最终调度最优，而是系统闭环稳定。

### 13.1 必做测试

1. **候选池 + mask 测试**
   - padding 不会被提交；
   - masked 位置不会被 greedy 或 sample 选中；
   - 当所有候选都被 mask 时，不执行 SAS softmax，而是由 driver 规则推进时间。

2. **Machine 选择规则测试**
   - 多台 Machine 同时可调度时，选择 `(next_available_time_m, num_feasible_candidates_m, machine_id_m)` 字典序最小的 Machine；
   - 当两台 Machine 空闲时间相同，优先选择真实有效候选数更少的 Machine；
   - 当前两项都相同，使用 Machine 编号作为稳定 tie-breaker。

3. **规则触发测试**
   - 有可调度 Machine 时能触发 SAS；
   - 无可调度 Machine 时能推进到下一事件；
   - 时间推进后 `current_time` 必须严格增加，除非 episode 已经终止。

4. **Transition 字段测试**
   - `RolloutStep` 记录 `machine_id`、`current_time`、`action_indices` 和 `valid_action_count`；
   - `RolloutStep.info` 记录 selected Lot / PPID、插入成功失败、mask invalid、wait/no-op 和 reward 分解；
   - `next_observation` 与下一次 SAS 决策观测一致，terminal step 可为 `None`。

5. **Episode 终止条件测试**
   - 所有 Lot 完成时终止；
   - 超过 planning horizon 且没有可继续调度 Lot 时终止；
   - 候选池为空且未来无 arrival / Machine release 时终止；
   - 超过 `max_total_wait_steps_per_episode` 时终止；
   - 超过 `max_failed_actions` 时终止；
   - 资源日历不可恢复错误时终止，并返回明确 `termination_reason`。

6. **单 episode 闭环测试**
   - 可以从初始状态执行到完成；
   - 不出现死循环或状态损坏；
   - driver 返回 episode reward、完成 Lot 数、step 数、wait 数、失败动作数和 `termination_reason`。

7. **PPO 烟雾测试**
   - rollout 后 advantage 可正确计算；
   - 一次 update 能成功 backward；
   - PPO 更新只使用训练阶段真实采样的 action，不使用推理 fallback 替换动作。

8. **最终 schedule 校验**
   - `validate_schedule()` 返回通过；
   - 不产生 machine / chamber 时间冲突；
   - Lot / Wafer 完整性校验通过。

### 13.2 成功标志

若满足以下条件，则可认为阶段2实现成功：

- 训练脚本可稳定运行多个 episode；
- 推理脚本能生成一个合法 schedule；
- reward 曲线不崩溃；
- 最终 schedule 能通过校验；
- 现有阶段1环境逻辑未被破坏；
- Machine 选择规则与报告第9.1节一致；
- rollout 中每个 SAS transition 都能追溯 `machine_id`、`obs_t`、`action_index`、`logπ_old`、`reward`、`obs_{t+1}` 和 `info`；
- episode 结束时总能给出明确 `termination_reason`；
- 训练阶段不使用推理 fallback 替换采样动作。

### 13.3 评估指标（来自报告第10段）

最小闭环稳定后，应能在 small 实例上输出以下指标并与随机基线对比：

| 指标 | 说明 |
|---|---|
| Q-time 违背 | wafer Q-time 约束违反次数 |
| 拖期 Lot 数 | 超过 due date 的 Lot 数量 |
| 总拖期时间 | 所有拖期 Lot 的拖期时间之和 |
| 平均机器利用率 | Machine 加工时间 / 总可用时间 |
| 优先级违背 | 高优先级 Lot 晚于低优先级 Lot 完成的次数 |

## 14. 风险点与控制策略

### 风险1：规则触发与时间推进写错，导致死循环

控制策略：

- 将 `advance_to_next_event()` 独立实现；
- 在 driver 中增加 `max_steps` 防御；
- 每轮循环都显式检查 `done` 和 `current_time` 是否前进。

### 风险2：Critic 输入不稳定

控制策略：

- 阶段2先不做注意力；
- 使用固定 `top_k` + masked pooling。

### 风险3：环境接口不足，driver 逻辑过重

控制策略：

- 在 `rl_environment.py` 只补少量标准辅助接口；
- 不在 driver 中重复资源日历内部逻辑。

### 风险4：reward 过于复杂导致训练不稳定

控制策略：

- 初期只开 `r_execute + r_wait`；
- 最小闭环跑通后再考虑 light shaping。

### 风险5：推理阶段缺少 fallback，生成 schedule 失败

控制策略：

- 推理 fallback（按概率降级尝试）在 `run_phase2_sas_inference_demo.py` 中实现，不混入训练逻辑；
- fallback 最终兜底为 wait/no-op，确保 episode 不会因无可用动作而中断；
- fallback 只在推理时使用，训练阶段保持采样轨迹一致性。

## 16. 结论

阶段2最小闭环版不应追求一步到位的复杂神经结构，而应基于现有阶段1资源日历环境，优先完成：

```text
规则触发
+ 候选池索引动作空间
+ action mask
+ 资源日历执行
+ 简化 reward
+ PPO Actor-Critic 更新
```

在这个闭环稳定之后，后续阶段3再将 Actor 从 MLP 升级为单注意力，阶段4再升级为双注意力，阶段5再引入 DDT 智能体，整体路线才是稳健的。

此外，推理阶段的 fallback 机制（按概率降级尝试候选动作）是阶段2闭环的重要组成部分。它确保了即使策略尚未充分收敛，推理也能产出合法 schedule，为后续阶段的基线对比提供可靠参考。

为了让后续升级更平滑，阶段2还应尽量冻结以下接口语义：

- `Phase2Observation` 继续作为候选池与全局摘要的统一观测结构；
- `RolloutStep` 继续保留 `machine_id`、`next_observation` 和 `info`，方便后续替换 Actor 结构时不改轨迹格式；
- `build_candidate_pool` 和 `build_action_mask` 的候选池 / mask 语义保持稳定，只在评分器与策略网络层升级；
- `sas_step` 继续承担资源日历可行性校验、状态转移和 reward 组装，不把这些逻辑分散到策略网络里；
- `termination_reason` 继续作为 episode 结束的统一出口，便于阶段3以后复用训练与评估代码。

这也是当前项目从“可执行环境原型”走向“可训练调度策略”的最合理过渡方案。
