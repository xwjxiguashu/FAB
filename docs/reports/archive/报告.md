# 面向半导体 FAB 机台组调度问题的项目报告

## 0. 报告边界说明

本报告以 `FAB厂机台组调度问题说明.pptx` 作为问题定义的主要依据。报告中关于调度对象、资源层级、输入信息、决策变量和优化目标的定义，只采用 PPT 中已经出现的概念。

论文资料和当前代码只作为**算法实现方案**、**技术路线设计**和**后续扩展方向**的参考，不把 PPT 中没有出现的概念写成原始问题定义。也就是说：

- PPT 中出现的内容，可以作为本项目的问题定义；
- 论文中出现但 PPT 未定义的内容，只能作为后续方法设计或可选扩展；
- 当前代码中已有但 PPT 未明确提出的内容，应说明为“实现方式”，而不是“PPT 原始定义”；
- 当前报告提出的 DDA-PPO、注意力网络、强化学习训练等内容，是后续算法升级方案，不是当前代码已经完成的功能。

因此，本报告的阅读逻辑是：先明确 PPT 定义的问题边界，再说明当前代码如何对应这一问题，然后分析当前实现不足，最后在已有资源日历和多目标评价基础上提出循序渐进的算法升级方案。

## 1. PPT 中定义的 FAB 机台组调度问题

根据 PPT，本项目研究的是一个面向半导体 FAB 的**机台组调度问题**。它不是简单的单机排序问题，而是一类带有多层资源结构、工艺路径选择和 wafer-level 加工安排的特殊并行机调度问题。

### 1.1 调度对象

PPT 中出现的主要调度对象包括：

- **Lot**：待排产对象；
- **Wafer**：Lot 内部包含的晶圆；
- **Recipe**：Lot 对应的工艺需求；
- **Step**：Recipe 或 PPID 中包含的加工步骤；
- **PPID**：在具体 Machine 上加工某个 Recipe 时可用的工艺路径；
- **Chamber / Tank / Side**：Machine 内部用于执行加工的细粒度资源。

因此，本项目的调度决策不能只停留在 Lot 先后顺序上，还必须进一步决定：Lot 分配到哪台 Machine、使用哪个 PPID、wafer 在哪些 Chamber / Side 上加工，以及每个加工步骤的开始和结束时间。

### 1.2 资源层级

PPT 给出的资源层级为：

```text
FAB Area -> Machine Group -> Machine -> Chamber / Tank -> Side
```

本项目关注的是 **Machine Group 内部**的调度问题。也就是说，所有候选 Machine 均属于同一个机台组，调度算法需要在该机台组内完成 Machine 选择、PPID 选择和 Chamber / Side 级别的 wafer 排程。

跨 FAB Area、跨 Machine Group 的全厂级联动调度不作为当前模型的基本范围。

### 1.3 输入信息

PPT 中与 Lot 相关的关键信息包括：

- wafer 数量；
- Recipe；
- 局部 due date；
- 预计到达时间；
- 加工优先级相关属性。

PPT 中与 Machine 相关的关键信息包括：

- Machine 结构信息；
- Chamber / Tank 数量及模式；
- 可加工 Recipe 集合；
- 加工每个 Recipe 时可用的 PPID。

PPT 中与 PPID 相关的关键信息包括：

- 加工步骤数量；
- 每个步骤内加工每片 wafer 的 process time；
- 其它生产约束所需信息。

这些信息共同决定了某个 Lot 是否能够在某台 Machine 上加工、有哪些 PPID 可选、对应 PPID 是否需要特定 Chamber / Side，以及加工完成后会产生怎样的交期、Q-time 和资源利用率结果。

### 1.4 决策变量

PPT 中明确给出的决策变量包括：

1. Lot 的加工机台；
2. Lot 在 Machine 上的加工顺序；
3. Lot 在 Machine 上的上机和下机时刻；
4. Lot 在 Machine 上加工所使用的 PPID；
5. wafer 在各 Chamber / Side 上的加工顺序；
6. wafer 上下 Chamber / Side 的时刻。

因此，本项目的输出至少需要包含两层排程结果：

```text
Lot 层：Lot -> Machine -> PPID -> 上机时间 / 下机时间
Wafer 层：Wafer -> Step -> Chamber / Side -> 进入时间 / 离开时间
```

Lot 层结果反映机台组层面的派工与机台占用，Wafer 层结果反映 PPID 内部步骤在底层资源上的实际执行。

### 1.5 优化目标

PPT 中给出的优化目标包括：

1. 减少拖期 Lot 数；
2. 减少总拖期时间；
3. 保持与工艺时限相关的加工约束；
4. 提高机台组平均机器利用率；
5. 尽量按照加工优先级顺序上机加工。

为避免概念混用，本报告将这些目标拆分为三类：

- **工艺约束 / 工艺风险**：Q-time 相关限制，反映工序之间允许等待的上限；
- **交期目标**：due date 相关的拖期 Lot 数和总拖期时间；
- **派工偏好**：priority 相关的加工顺序偏好。

对应的评价指标为：

```text
Q-time 违背次数
Q-time 总违背量
tardy Lot 数
总拖期时间
机台组平均机器利用率
加工优先级顺序违背
```

其中，Q-time 是否作为硬约束，取决于业务定义：如果生产规则要求绝对不可违反，则应在候选动作过滤或资源日历校验阶段直接禁止违背；如果业务允许少量越界并通过目标函数惩罚，则可将其作为软约束进入 reward 和最终评价。当前报告后续算法方案默认采用后一种处理方式，即 Q-time 主要作为风险特征、训练惩罚和评价指标，而不是一开始就屏蔽所有高风险动作。

## 2. 建模假设与约束边界

为了避免把问题无限扩大为全厂级复杂调度，本报告将研究范围限定为**单个 Machine Group 内部的动态派工与资源排程问题**。

### 2.1 调度范围假设

本项目默认研究对象为一个给定 Machine Group 内部的调度问题，而不是跨多个 FAB Area 或跨多个 Machine Group 的全厂级联动调度。所有候选 Machine 均属于同一个 Machine Group，Lot 在进入该 Machine Group 前的上游工序，以及离开该 Machine Group 后的下游工序，不在本模型中显式展开。

因此，本项目关注的是：在当前机台组内，给定已经到达或即将到达的 Lot，如何选择 Machine、PPID、加工顺序和 wafer-level Chamber / Side 时间安排。跨机台组搬运、跨区域 WIP 平衡、全厂瓶颈迁移和跨工序重入流控制，可作为后续扩展方向。

### 2.2 工艺与资源假设

本报告采用以下工艺与资源假设：

1. **非抢占式加工**：一旦 wafer 的某个 stage 在 Chamber / Side 上开始加工，该 stage 不被中断；Lot 在 Machine 上的占用区间也不被拆分。
2. **确定性加工时间**：PPID 中每个 stage 的 process time 在调度时已知，暂不考虑随机加工时间、设备波动和批次间 processing time 分布。
3. **Machine 可加工性由 Recipe / PPID 决定**：某个 Lot 只能被分配到支持其 Recipe 且存在可用 PPID 的 Machine 上。
4. **Chamber / Side 是细粒度容量资源**：同一 `(machine, chamber, side)` 在同一时间只能加工一个 wafer stage。
5. **Machine 是 Lot-level 容量资源**：同一 Machine 在同一时间只能承载一个 Lot 的上机到下机区间。
6. **PPID 决定 wafer-level stage 结构**：选择 PPID 后，Lot 内所有 wafer 需要按照该 PPID 定义的 stage 结构完成加工。

这些假设使问题从抽象并行机调度扩展为具有 Machine / PPID / Chamber / Side 多层资源结构的 FAB 机台组调度问题。

### 2.3 当前暂不纳入的约束

为保证模型可实现和算法训练可控，当前报告不把以下因素作为基本约束：

- 跨 Machine Group 的搬运时间和 AMHS 路径拥堵；
- operator / 人员操作能力约束；
- 设备维护、停机、PM、故障恢复和机况退化；
- setup time、recipe 切换时间和 chamber 清洗时间；
- 批处理、合批、拆批和 lot merge / split；
- re-entrant flow 下同一 Lot 多次回到同一机台组的完整全流程建模；
- 随机加工时间、多场景不确定性和鲁棒 / CVaR 优化；
- 全厂级 WIP 控制、bottleneck shift 和跨区域负载均衡。

这些因素并非不重要，而是会显著增加状态空间、动作空间和仿真复杂度。若后续扩展到更接近真实 FAB 的工业场景，应逐步加入上述约束，并重新定义状态特征、action mask、资源日历和 reward。

### 2.4 硬约束、软约束与评价指标

本报告将约束和目标区分为三类：

1. **硬约束**：必须由候选动作过滤、action mask 或资源日历强制保证，包括 Machine 可加工性、PPID 可用性、PPID stage 完整性、Machine 时间不重叠、Chamber / Side 时间不重叠，以及 Lot / Wafer 加工完整性。
2. **软约束 / 训练惩罚**：可以通过 reward 或评分函数引导优化，包括 Q-time 风险、拖期风险、优先级违背、机台空闲和资源冲突风险。
3. **最终评价指标**：用于实验比较和结果汇报，包括 Q-time 违背次数和总违背量、拖期 Lot 数、总拖期时间、平均机器利用率、优先级违背程度。

其中，Q-time、due date 和 priority 的作用不同：

- **Q-time** 是工艺等待时限，主要用于判断工艺风险或工艺可行性；
- **due date** 是交期结果，反映 Lot 是否按时完成；
- **priority** 是派工偏好，反映候选 Lot 在同一时间窗内的排序偏好。

因此，三者不应合并写成一个笼统的“违背目标”，而应在状态特征、reward 和最终评价中分别建模、分别统计、分别解释。

## 3. 当前代码与 PPT 定义的对应关系

当前项目代码已经实现了一个基于动态调度、资源日历和多目标评价的原型系统。核心类为 `TwoPopulationScheduler`，由多个 mixin 组成。当前实现可以作为后续强化学习环境的基础，但它本身仍是确定性动态滚动调度加 NSGA-II 搜索线性策略权重，而不是已经实现的 DDA-PPO。

### 3.1 输入数据对应关系

当前代码中的输入与 PPT 定义可以对应如下：

| PPT 概念 | 当前代码中的对应变量 |
|---|---|
| Lot | `lot` 编号 |
| wafer 数量 | `wafer_counts[lot]` |
| Recipe / 可加工关系 | 通过 `feasible_machines`、`feasible_ppids` 间接表示 |
| Machine | `machine` 编号 |
| PPID | `ppid` 编号 |
| PPID 加工步骤 | `ppid_steps[(lot, machine, ppid)]` |
| Chamber / Side | stage 候选资源中的 `[chamber, side, process_time]` |
| process time | stage 候选资源中的 `process_time` |
| 预计到达时间 | `arrival_times[lot]` |
| due date | `due_dates[lot]` |
| 加工优先级 | `priorities[lot]` |
| Q-time 相关限制 | `q_time_limits` |

需要注意的是，当前代码中没有显式建立独立的 `Recipe` 对象和 `Machine Group` 对象，而是通过 Lot、Machine、PPID 之间的可行关系间接表达。因此，在报告和论文表述中，应避免把当前代码中的隐式结构说成 PPT 已经单独定义的对象。

### 3.2 输出结果对应关系

当前代码输出两类排程。

Lot schedule：

```text
[lot, machine, ppid, start_time, end_time]
```

对应 PPT 中的：

- 加工机台；
- 在 Machine 上的加工顺序；
- 上机和下机时刻；
- 所用 PPID。

Wafer schedule：

```text
[lot, wafer_id, machine, ppid, stage_id, chamber, side, start_time, end_time]
```

对应 PPT 中的：

- wafer 在各 Chamber / Side 的加工顺序；
- wafer 上下 Chamber / Side 的时刻。

因此，当前代码的输出结构与 PPT 中的决策变量是匹配的。

### 3.3 资源日历的作用

当前代码使用资源日历保证排程可执行：

- Machine 日历用于避免同一 Machine 上的 Lot 加工时间重叠；
- Chamber / Side 日历用于避免同一 `(machine, chamber, side)` 上的 wafer stage 时间重叠；
- earliest slot 搜索用于确定可插入的最早加工时间；
- rollback 用于在候选方案不可行时撤销临时插入。

资源日历不是 PPT 中单独给出的定义，而是当前代码为实现 PPT 中资源约束和决策变量而采用的工程方法。后续 PPO 或 DDA-PPO 方案不应绕开资源日历，而应把资源日历作为不可导的调度环境和可行性校验模块。

### 3.4 当前目标函数对应关系

当前代码的目标函数包括：

```text
[
  q_time_violation_count,
  total_q_time_violation,
  tardy_count,
  total_tardiness,
  priority_violation,
  -avg_machine_utilization
]
```

这些指标可以对应到 PPT 中的目标：

- `tardy_count` 对应减少拖期 Lot 数；
- `total_tardiness` 对应减少总拖期时间；
- `q_time_violation_count` 和 `total_q_time_violation` 对应 Q-time 工艺时限违背；
- `-avg_machine_utilization` 对应提高机台组平均机器利用率；
- `priority_violation` 对应尽量按照加工优先级顺序上机加工。

当前代码通过 NSGA-II 搜索线性评分策略的权重，并使用上述指标进行多目标评价。这里的“多目标”是指评价向量包含多个目标，而不是后续 DDA-PPO 训练已经在直接搜索 Pareto 前沿。

## 4. 当前实现的主要不足与算法升级动机

在 PPT 定义范围内，当前代码已经完成了基础调度结构，但如果希望形成更有研究创新性的智能调度算法，仍存在以下不足。

### 4.1 Machine Group 与 Recipe 语义不够显式

PPT 明确包含 Machine Group 和 Recipe 概念，但当前代码主要直接使用 Machine 编号、可行 Machine 集合、可行 PPID 集合和 PPID stage 数据，没有显式建立 Machine Group 对象和 Recipe 对象。

这并不影响当前原型运行，但在报告和论文表达中需要说明：当前调度范围默认是某个 Machine Group 内部，Recipe 通过 Lot 与 PPID 可行关系间接表达。后续若要更贴近 PPT，可在数据结构或输入说明中显式加入 Machine Group、Recipe、Machine-Recipe-PPID 的层级关系。

### 4.2 Q-time 主要作为评价指标，尚未充分前移

当前代码能够统计 Q-time 违背次数和总违背量，但 Q-time 主要在目标函数中评价，还没有充分前移到候选动作评分、风险特征或显式约束处理阶段。

后续可以在不改变 PPT 定义的前提下，把 Q-time 从“事后评价”前移到“候选动作特征、reward 惩罚或硬约束过滤”阶段。若 Q-time 被定义为软约束，则进入风险特征和惩罚项；若被定义为硬约束，则进入 action mask 或资源日历插入校验。

### 4.3 当前策略表达能力有限

当前代码通过手工特征和线性权重为 Lot、Machine / PPID、Chamber / Side 打分。这种方法可解释、易实现，并适合做基础原型，但表达能力有限：

- 线性权重难以表达 Lot 紧迫度、PPID 加工结构和底层资源冲突之间的非线性关系；
- 线性策略主要依赖人工设计特征，难以自动学习不同调度场景下的策略变化；
- NSGA-II 搜索权重适合小规模可解释优化，但面对更大规模、动态到达和复杂候选动作时，搜索效率和泛化能力会受到限制。

因此，后续升级方向不是推翻当前资源日历，而是在保留资源日历可行性校验的基础上，将线性评分策略升级为可学习的神经网络策略。

### 4.4 为什么选择强化学习与 PPO

FAB 机台组调度具有明显的序贯决策特征：每一次派工都会改变 Machine 日历、Chamber / Side 日历、后续候选 Lot 的等待时间、Q-time 风险和拖期风险。因此，单次动作的好坏不能只看当前局部收益，还会影响后续状态。

强化学习适合处理这种“动作改变环境、环境再影响后续决策”的问题。与直接用监督学习模仿规则相比，强化学习可以通过完整排程结果和 step-level reward 学习长期收益；与只用进化算法搜索固定权重相比，神经网络策略可以根据当前候选池和资源状态动态调整决策。

在强化学习算法中，PPO 适合作为本项目的起点，原因是：

1. PPO 可以处理离散动作选择，适合候选池索引动作空间；
2. PPO 使用 clipped objective，训练稳定性通常优于普通 policy gradient；
3. 资源日历是不可导环境，PPO 不要求对资源日历本身求导；
4. PPO 可以结合 Actor-Critic 结构，把候选动作选择和状态价值估计放在统一训练闭环中。

因此，本项目的升级路线应当是：保留当前资源日历和多目标评价，先构建候选池资源日历 PPO，再逐步加入注意力机制、决策时间智能体和双智能体 DDA-PPO。

## 5. 论文依据与方法选择

本节只说明论文对算法实现的启发，不把论文中的额外概念写成 PPT 问题定义。PPT 定义仍然是项目边界；论文资料主要用于支撑后续算法升级方案。

### 5.1 动态候选动作池用于处理动作空间过大

`lunwen/10/` 和 `lunwen/15/` 中的多智能体调度、半导体 FAB 强化学习、自监督强化学习和 cluster tool 调度相关论文都说明，调度问题中的动作空间通常随状态变化，不适合构造固定全局动作表。

对应到本项目，如果直接定义全局动作：

```text
a = (Lot, Machine, PPID)
```

动作规模约为：

```text
|A| ≈ |Lots| × |Machines| × |PPIDs|
```

在大规模 Lot 和 PPID 场景下，这会导致大量动作几乎不可行或很少被采样，策略梯度方差变大，训练难以收敛。因此，本项目应采用动态候选动作池：在当前 Machine `m` 空闲时，只生成该 Machine 上当前结构可行或高质量的局部候选动作。

### 5.2 Attention 可用于可变规模候选动作评分

DANIEL、FAB 调度强化学习和相关 attention / graph embedding 论文强调，调度模型需要处理不同规模实例，并在不同候选数量下保持策略输出稳定。

对应到本项目，可以把当前空闲 Machine 作为上下文，将候选 `(Lot, PPID)` 动作编码为 embedding，再通过带 mask 的 attention 输出候选动作概率。这样可以同时解决两个问题：

- 候选动作数量随时间变化；
- 不同候选动作之间存在竞争关系，例如交期紧迫度、Q-time 风险、加工时间和资源冲突之间的权衡。

Attention 不直接替代资源约束；它只负责学习候选动作排序或概率分布。资源日历仍然负责最终可行性检查。

### 5.3 双智能体思想支持“何时决策 + 如何派工”

多智能体调度论文提供了一个重要启发：复杂调度问题可以拆成多个职责更清晰的子决策。对应到本项目，可以将调度拆成两层：

```text
DDT：判断当前是 dispatch_now 还是 wait / time advance
SAS：在当前 Machine 的候选池中选择 (Lot, PPID)
```

这种拆分符合 FAB 场景的事件驱动特征。Machine 空闲并不一定意味着立即派工最优；如果短时间内会到达更高优先级或更紧急的 Lot，适度等待可能改善全局排程质量。因此，单独学习“何时派工”有助于提升策略灵活性。

这里需要明确一个边界：**未来尚未到达的 Lot 不应进入当前 SAS 候选动作池，也不应通过 action mask 伪装成当前可执行动作**。它们在当前时刻还不能被资源日历 commit，因此只能作为 DDT 或全局状态的前瞻信息，用来估计当前派工的机会成本。换言之，SAS 解决“当前能派谁”，DDT 解决“现在是否值得派”。

### 5.4 DeepSets + Attention Pooling 适合编码未来未到达 Lot

未来未到达 Lot 的数量随时间变化，且它们没有像图像一样的规则网格结构，也不构成严格的自然语言序列。因此，直接使用 CNN 或普通 RNN 作为主要特征提取器并不贴合该对象。更合适的建模方式是把未来窗口内的未到达 Lot 看作一个变长集合：

```text
F_t = {l | arrival_time_l > t_now, arrival_time_l <= t_now + H_future}
```

DeepSets 类方法先用共享 MLP 对每个未来 Lot 独立编码，再通过 permutation-invariant pooling 得到固定长度表示，天然适合处理变长集合。为了避免简单 mean / max pooling 把所有未来 Lot 视为同等重要，可以进一步使用 machine-aware attention pooling：以当前 Machine 状态作为上下文，让模型自动学习哪些未来 Lot 对当前是否等待最关键。

对应到本项目，每个未来 Lot 可编码为：

```text
u_j = [
  time_to_arrival_j,
  priority_j,
  due_slack_at_arrival_j,
  estimated_process_time_j,
  wafer_count_j,
  compatible_machine_count_j,
  is_compatible_with_current_machine_j,
  is_only_compatible_with_current_machine_j
]
```

共享编码器和注意力聚合可以写为：

```text
h_j = MLP_future(u_j)
q_m = MLP_machine(machine_features_m)
e_j = MLP_attn(concat(h_j, q_m))
α_j = MaskedSoftmax(e_j, future_lot_mask)
future_context = Σ_j α_j h_j
```

其中 `future_lot_mask` 只用于屏蔽 padding 的未来 Lot，不表示该 Lot 当前可执行。`future_context` 是固定长度前瞻表示，可与当前 Machine 特征、当前候选池摘要特征一起输入 DDT；在更高阶版本中，也可以作为全局上下文输入 SAS Critic，帮助价值函数评估当前派工对未来的影响。但第一版不建议让未来未到达 Lot 直接参与 SAS Actor 的动作打分，以免混淆“当前可执行动作”和“未来机会成本”。

该结构比人工统计聚合更灵活，比完整 Transformer 更轻量，适合作为 DDT 阶段的第一版神经特征提取器。完整 Transformer、Lot-Machine 二部图 GNN 或候选动作感知 cross-attention 可作为后续增强，而不应作为最小闭环的起点。

### 5.5 双注意力结构支持 Lot 紧迫性与资源竞争建模

柔性作业车间和 FAB 调度中的双注意力、图注意力和机器注意力方法说明，调度决策通常同时涉及需求侧和资源侧两类信息。

对应到本项目，可以将 SAS Actor 设计为两个注意力分支：

```text
Lot-PPID 注意力：
  学习 due date、Q-time、priority、waiting time、PPID process time 等候选动作竞争关系

Machine-Chamber 竞争注意力：
  学习 Machine、Chamber、Side 短窗口利用率、资源冲突风险和可插入性
```

这样比单一 MLP 或单一注意力结构更贴合本项目的 Lot / Wafer 两层资源日历结构。

### 5.6 Action masking 用于结构性不可行动作过滤

相关调度论文强调 action masking 的重要性。对应到当前项目，mask 应主要用于过滤结构性不可行动作，例如：

- Lot 尚未到达或已经完成；
- Machine 不支持该 Recipe；
- Machine 上没有可用 PPID；
- PPID stage 定义缺失或不完整；
- Chamber / Side 不满足候选加工要求；
- padding action。

Q-time 风险、due date 紧迫度和 priority 偏好通常不作为同一类 hard mask，而是分别进入状态特征、step-level reward 和终局评价。除非生产规则明确规定某类 Q-time 风险绝对不可接受，否则不应仅因风险高而屏蔽动作。

### 5.7 资源日历下更适合使用 step-level PPO

半导体调度环境具有长周期、动态到达、多资源约束和不可导资源日历。直接用黑盒进化算法搜索大规模神经网络参数效率较低，而完全依赖普通反向传播又无法穿透资源日历。

因此，本项目更适合采用折中方式：把资源日历视为不可导环境，神经网络 Actor 输出候选动作概率，Critic 估计状态价值，资源日历执行派工并返回 step-level reward，最后通过 PPO 更新 Actor-Critic 网络。

## 6. 算法升级总体框架

在明确问题边界、当前不足和论文依据之后，后续算法升级可以概括为：

> **以资源日历为环境，以候选池控制动作规模，以 PPO 训练可学习策略，并逐步升级为双智能体双注意力 DDA-PPO。**

### 6.1 总体流程

整体调度流程如下：

```text
事件触发或当前时刻推进
      ↓
读取资源日历状态 s_t
      ↓
确定当前需要决策的 Machine m
      ↓
生成 Machine m 的局部候选动作池 A_t^m
      ↓
Top-K / padding / action mask 得到固定长度候选池
      ↓
Actor 在候选池中选择 action_index
      ↓
环境映射为真实动作 (lot, machine, ppid)
      ↓
资源日历 dry-run / commit
      ↓
计算 step-level reward 和下一状态
      ↓
PPO 使用 rollout 更新 Actor-Critic
```

该流程中各模块职责清晰：

```text
候选池：控制动作规模
Action mask：过滤结构性不可行动作
Actor：学习候选动作选择概率
Critic：估计状态价值
资源日历：执行可行性校验和状态转移
Reward：把交期、Q-time、利用率、优先级等目标转化为训练信号
PPO：更新策略网络和价值网络
```

### 6.2 MDP 形式化定义

环境状态 `s_t` 包含完整资源日历和当前调度上下文：

```text
s_t = {
  当前时刻 t_now,
  未完成 Lot 集合及其 arrival time / due date / priority / wafer count,
  空闲 Machine 集合 M_idle,
  每台 Machine 的 Lot-level calendar,
  每个 (machine, chamber, side) 的 wafer-level calendar,
  已经提交的 Lot / Wafer 操作,
  当前窗口内已完成和未完成 Lot 状态
}
```

神经网络不直接输入完整日历明细，而是输入由环境提取出的特征化表示 `z_t = featurize(s_t)`，例如：

```text
z_t = {
  当前时刻特征,
  当前 Machine 的负载率、下一空闲时间、已排 Lot 数,
  当前已到达候选 Lot 的 due date 紧迫度、Q-time 风险、优先级、wafer 数量、等待时间,
  当前候选 PPID 的步骤数量、process time 统计量、Chamber / Side 需求,
  未来未到达 Lot 的前瞻上下文:
    - 未来窗口内 Lot 的到达时间、优先级、due slack 和加工规模
    - 与当前 Machine 的可加工关系和不可替代性
    - 由 DeepSets + machine-aware attention pooling 得到的 future_context,
  资源日历摘要特征:
    - Machine 短窗口占用率
    - Chamber / Side 短窗口占用率
    - 预计最早开始时间
    - 预计完成时间
    - 当前平均机台利用率
}
```

完整资源日历保留在环境中，网络只使用可计算的状态摘要特征。需要特别区分的是，当前已到达 Lot 可以进入 SAS 候选池；未来未到达 Lot 只进入状态表示，帮助 DDT 判断是否等待，不能作为当前可执行动作。

### 6.3 局部动作空间

在单台 Machine `m` 空闲时，动作定义为：

```text
a_t^m = (lot, ppid)
```

此时 `machine = m` 是当前状态的一部分，不需要在动作中重复出现。环境执行时将其还原为完整动作：

```text
execute(m, a_t^m) = execute(lot, m, ppid)
```

如果多台 Machine 同时空闲，则按固定顺序逐台决策，例如按最早空闲时间、候选动作数量或 Machine 受限程度排序。每台 Machine 决策一次后立即更新资源日历和候选 mask，再处理下一台 Machine。这样可以避免构造组合动作空间。

## 7. 基础方案：候选池资源日历 SAS-PPO

DDA-PPO 不应一步到位实现。更合理的起点是先构建一个基础版候选池资源日历 PPO，其中只学习“当前 Machine 应该选择哪个 Lot / PPID”，暂不学习“是否等待”。本报告将该基础方案称为 **SAS-PPO**。

### 7.1 候选动作池生成

对当前空闲 Machine `m`，先生成结构上可行的候选动作：

```text
A_t^m = {(l, p) | l 已到达且未完成，m 可加工 l，p 是 (l, m) 的可用 PPID}
```

结构性过滤包括：

1. Lot 尚未到达或已经完成；
2. Machine 不支持该 Lot 对应的 Recipe；
3. `(lot, machine)` 没有可用 PPID；
4. PPID 的步骤定义不存在或不完整；
5. PPID 所需 Chamber / Side 在当前 Machine 上不存在。

如果候选动作仍然过多，则使用启发式评分保留 Top-K：

```text
CandidateScore(l, m, p)
  = due_date_urgency(l)
  + qtime_risk(l)
  + priority(l)
  + waiting_time(l)
  - estimated_process_time(l, m, p)
  - resource_conflict_risk(l, m, p)
```

最终只保留：

```text
A_top^m = TopK(A_t^m, CandidateScore, K_action)
```

其中 `K_action` 是候选池大小。训练初期可取 30 到 50，中等规模可取 50 到 100，大规模场景可取 100 到 200。候选池只是控制搜索规模，不替代资源日历的最终可行性校验。

### 7.2 可变动作维度的处理

实际调度过程中，每个决策时刻可选 Lot 数量和每个 Lot 可用 PPID 数量都可能变化。因此，本项目不构造固定全局动作空间，也不让 Actor 直接输出固定 Lot 编号或 PPID 编号，而是采用**候选池索引动作空间**：

```text
Actor 输出: action_index ∈ {0, 1, ..., K_action - 1}
环境映射: candidate_actions[action_index] = (lot_id, ppid_id)
执行动作: execute(lot_id, machine_id, ppid_id)
```

不同决策时刻的候选动作数通过 Top-K、padding 和 action mask 对齐到固定长度 `K_action`：

```text
如果 |A_t^m| > K_action:
    A_fixed^m = TopK(A_t^m, CandidateScore, K_action)

如果 |A_t^m| < K_action:
    A_fixed^m = Pad(A_t^m, K_action)

如果 |A_t^m| = K_action:
    A_fixed^m = A_t^m
```

Actor 始终输出固定维度的 logits：

```text
logits ∈ R^{K_action}
```

随后使用 masked softmax：

```text
π(a_i | s_t, A_fixed^m) =
    exp(logit_i) / Σ_{j: mask_j=1} exp(logit_j),  if mask_i = 1
    0,                                             if mask_i = 0
```

如果某一时刻候选池为空，或所有候选动作都被 mask 屏蔽，则不执行 masked softmax。此时环境直接执行 wait / no-op，或将时间推进到下一次 Lot 到达 / Machine 释放事件，避免 softmax 分母为 0。

### 7.3 SAS Actor-Critic 结构

基础 SAS-PPO 可以先使用简单 MLP Actor；在此基础上再升级为机台条件化注意力 Actor。

每个候选动作被编码为动作特征向量：

```text
x_i = [lot_i_features, ppid_i_features, machine_m_features, calendar_i_features]
```

其中：

- `lot_i_features` 包括 due date 紧迫度、Q-time 风险、优先级、wafer 数量和等待时间；
- `ppid_i_features` 包括步骤数量、process time 统计量和 Chamber / Side 需求；
- `machine_m_features` 包括当前 Machine 的下一空闲时间、短窗口负载率和已排 Lot 数；
- `calendar_i_features` 包括预计最早开始时间、预计完成时间和资源冲突风险。

机台条件化注意力结构可以写为：

```text
e_i = Linear(x_i)
q_m = Linear(machine_m_features)

Q = q_m
K = [e_1, e_2, ..., e_K]
V = [e_1, e_2, ..., e_K]

h_m = MultiHeadAttention(Q, K, V, mask)
score_i = MLP([h_m, e_i])
π_SAS(a_i | z_t, A_fixed^m) = MaskedSoftmax(score_i)
```

Critic 可以共享候选动作 embedding 和注意力表示，并通过 pooling 得到当前候选池的整体表示：

```text
g_t = Pooling(h_m, e_1, e_2, ..., e_K)
V_SAS(z_t) = MLP([g_t, global_features])
```

### 7.4 资源日历执行与失败处理

Actor 选择动作后，环境将动作交给资源日历执行：

```text
TryInsert(lot, machine, ppid, C_t)
```

资源日历负责检查：

1. Machine 时间是否可插入；
2. PPID 对应步骤是否完整；
3. Chamber / Side 是否存在可用时间；
4. 插入后 Lot / Wafer 时间表是否与已有操作冲突。

如果动作可执行，则提交 Machine 日历和 Chamber / Side 日历，得到下一状态 `s_{t+1}`。

如果动作不可执行，训练阶段不能把实际执行动作替换为候选池中的下一个动作。原因是 PPO 更新需要记录被策略采样的动作及其 `logπ_t`，如果 Actor 采样的是动作 A，而环境实际执行的是动作 B，则轨迹中的动作和策略概率不一致，会导致 PPO 的概率比和梯度估计偏离真实策略。

因此，训练阶段的处理方式应为：保留原始采样动作 A，返回失败惩罚，不提交任何日历修改，并把环境推进到明确的下一状态，例如 wait / no-op 或下一事件时刻。

推理阶段不涉及 PPO 梯度更新，可以采用工程 fallback：按 Actor 概率从高到低尝试候选动作，提交第一个资源日历可插入的动作；若全部不可行，再执行 wait / no-op。

### 7.5 基础 SAS-PPO 的简化奖励设计

基础阶段只单独训练 SAS，不训练 DDT。此时 reward 不应同时承担“是否等待”“全局多目标权衡”“双智能体协同”等任务，而应只回答一个问题：

```text
在当前 Machine 和当前候选池中，SAS 选择的 (Lot, PPID) 是否是一个稳定、可执行、相对合理的派工动作？
```

因此，基础 SAS-PPO 的 reward 应当尽量小、尽量局部、尽量容易由代码直接计算。推荐先采用如下形式：

```text
r_SAS = clip(r_execute + r_wait + r_shape + r_terminal, -1.0, 1.0)
```

其中，第一版训练可以令：

```text
r_shape = 0
r_terminal = 0
```

也就是先只训练：

```text
r_SAS = clip(r_execute + r_wait, -1.0, 1.0)
```

#### 7.5.1 执行结果奖励

执行结果奖励是 SAS 单训阶段的主信号：

```text
r_execute =
  +0.20,  if insertion_success = True
  -0.40,  if insertion_failed = True
  -0.50,  if mask_invalid = True
   0.00,  otherwise
```

其中：

- `insertion_success` 表示资源日历成功插入 `(lot, machine, ppid)`，并提交 Lot-level 和 Wafer-level schedule；
- `insertion_failed` 表示候选动作理论上来自候选池，但资源日历 dry-run 或 commit 失败；
- `mask_invalid` 表示 Actor 采样到了 padding action 或 `candidate_mask[action_index] = 0` 的动作。

这部分 reward 的作用是让 SAS 先学会避开无效动作和不可插入动作，而不是一开始就学习复杂全局目标。

#### 7.5.2 wait / no-op 惩罚

如果基础环境中仍然保留 wait / no-op，例如候选池为空、所有候选都不可插入，或当前时间需要推进，则 wait 只给很小惩罚：

```text
r_wait =
  -0.02,  if wait_or_noop = True
   0.00,  otherwise
```

wait 惩罚不能太大。若 wait 惩罚过大，策略会倾向于强行选择不可行候选；若 wait 惩罚过小，策略可能长期等待。因此基础阶段建议固定为 `-0.02`，并通过 `max_consecutive_wait` 或 episode 终止条件限制无意义等待。

#### 7.5.3 轻量 shaping 项

当执行结果奖励已经能让 SAS 生成基本完整 schedule 后，再加入轻量 shaping：

```text
r_shape =
  -0.05 × tardy_norm
  -0.03 × qtime_norm
  -0.02 × priority_rank_penalty
  +0.03 × progress_norm
```

所有分量都必须裁剪到 `[0, 1]`。

```text
tardy_norm = clip(max(0, selected_lot_end - due_date[selected_lot]) / H, 0, 1)
```

`selected_lot_end` 来自本次成功插入后的 Lot schedule 行 `[lot, machine, ppid, start_time, end_time]`，`H` 为当前 planning window 长度。

```text
qtime_norm = clip(new_qtime_violation / max(1, selected_lot_process_time), 0, 1)
```

`new_qtime_violation` 只统计本次动作新增的 selected lot wafer schedule 中产生的 Q-time 违背量，不在每一步重新计算全局 Q-time 总违背。

```text
priority_rank_penalty =
  count(priority(l') > priority(selected_lot) for l' in real_candidates)
  / max(1, len(real_candidates) - 1)
```

它只衡量当前候选池内是否跳过了更高优先级 Lot，不使用全局 `priority_violation` 逆序总量作为 step reward。

```text
progress_norm = clip((selected_lot_end - selected_lot_start) / H, 0, 1)
```

`progress_norm` 是很小的进度奖励，用于鼓励成功排入实际加工任务。它的权重不能过大，否则策略可能偏向加工时间长的 Lot。

#### 7.5.4 终局奖励

SAS 单独训练初期建议不使用终局奖励：

```text
r_terminal = 0
```

原因是终局目标通常尺度更大、延迟更长，容易压过单步执行结果奖励，使 PPO 在早期训练中不稳定。若基础策略已经能稳定完成 episode，可以再加入很小的终局辅助项：

```text
r_terminal = -0.05 × final_score_norm
```

其中：

```text
final_score_norm =
  0.40 × tardy_count / num_lots
  + 0.30 × clip(total_tardiness / (num_lots × H), 0, 1)
  + 0.20 × q_time_violation_count / max(1, num_qtime_checks)
  + 0.10 × clip(priority_violation / priority_scale, 0, 1)
```

该终局奖励只在 episode 结束时加入一次，并且只作为辅助信号。训练是否收敛，优先看 SAS 是否能稳定生成无资源冲突、Lot / Wafer 完整的 schedule，而不是一开始追求所有评价指标最优。

#### 7.5.5 推荐训练顺序

基础 SAS-PPO 的 reward 建议按以下顺序逐步打开：

```text
阶段 A：只用执行结果
r_SAS = r_execute + r_wait

阶段 B：加入轻量局部 shaping
r_SAS = r_execute + r_wait + r_shape

阶段 C：加入极小终局辅助
r_SAS = r_execute + r_wait + r_shape + r_terminal
```

每一阶段都应保持：

```text
r_SAS = clip(r_SAS, -1.0, 1.0)
```

如果训练出现明显振荡，优先关闭 `r_terminal` 和 `qtime_norm`，退回阶段 A 或阶段 B，而不是继续增加 reward 分量。

### 7.6 基础 SAS-PPO 算法流程

```text
Algorithm: Candidate-pool Calendar SAS-PPO

Input:
    Waiting lots L
    Idle machines M_idle
    Feasible PPIDs P
    Resource calendars C
    Candidate size K_action
    SAS Actor π_SAS
    Critic network V_SAS

Output:
    Dispatch decisions and updated calendars

1. Observe environment state s_t and extract features z_t
2. Select an idle machine m according to a fixed rule
3. Build structurally feasible candidate pool A_t^m = {(l, p)}
4. Use Top-K / padding / mask to obtain fixed-size candidate pool A_fixed^m
5. Encode the current machine and fixed-size candidate actions
6. SAS Actor computes π_SAS(a | z_t, A_fixed^m)
7. Sample action_index during training, or rank action_index during inference
8. If action_index is padding or mask=0:
       do not call resource calendar
       set reward = -0.50
       store transition and continue
9. If candidate pool is empty:
       execute wait / no-op
       set reward = -0.02
       store transition and advance time
10. Resource calendar tries to insert (lot, m, ppid)
11. If insertion succeeds:
       commit calendars
       compute small SAS reward using r_execute + optional r_shape
12. If insertion fails during training:
       keep the sampled action in the trajectory
       do not replace it with another action
       set reward = -0.40
13. If insertion fails during inference:
       try the next highest-probability candidate or execute wait / no-op
14. Store (z_t, m_t, A_fixed^m, action_index, logπ_t, reward, z_{t+1})
15. Repeat until the planning window is completed
16. Use PPO to update SAS Actor and Critic
```

基础 SAS-PPO 的价值在于先验证候选池、mask、资源日历执行、简化 reward 和 PPO 更新能否形成稳定闭环。只有该基础闭环稳定后，才适合继续加入终局奖励、DDT 决策时间智能体和双注意力结构。

## 8. 创新方案：DDA-PPO 双智能体双注意力资源日历调度

在基础 SAS-PPO 稳定后，进一步升级为：

> **DDA-PPO：Dual-agent Dual-attention Calendar-constrained PPO Dispatching**

中文可概括为：

> **双智能体双注意力资源日历约束 PPO 派工算法**

该方案不改变 PPT 中定义的 FAB 机台组调度问题，也不替代现有资源日历解码器，而是在“何时决策”和“如何派工”两个层面增强基础 PPO 框架。

### 8.1 双智能体协同决策

DDA-PPO 将调度过程拆成两个协同智能体：

```text
资源日历状态
      ↓
DDT 决策时间智能体
判断：立即派工 / 等待下一个 Lot / 等待下一台 Machine / 等待短时间步
      ↓
SAS 派工动作智能体
在当前 Machine 的 Top-K 候选池中选择 (Lot, PPID)
      ↓
资源日历 dry-run / commit
      ↓
step-level reward + episode-level reward
      ↓
PPO 更新；稳定后进行联合微调
```

DDT agent 负责学习“当前是否应该立即派工”。在 FAB 调度中，机台空闲并不总是意味着马上派工最优；如果短时间内会到达更高优先级、Q-time 风险更高或 due date 更紧的 Lot，适度等待可能提升全局排程质量。

DDT 的状态特征可以包括：

```text
当前时刻 t_now
空闲 Machine 数量
当前最高 Q-time 风险
当前最高 due date 紧迫度
当前平均机台利用率
当前候选池是否为空
下一次 Lot 到达时间
下一次 Machine 释放时间
连续 wait 次数
future_context：未来未到达 Lot 的 DeepSets + Attention 表示
```

其中 `future_context` 由未来窗口内未到达 Lot 的变长集合编码得到：

```text
F_t = {l | arrival_time_l > t_now, arrival_time_l <= t_now + H_future}

u_l = [
  time_to_arrival_l,
  priority_l,
  due_slack_at_arrival_l,
  estimated_process_time_l,
  wafer_count_l,
  compatible_machine_count_l,
  is_compatible_with_machine_l,
  is_only_compatible_with_machine_l
]

h_l = MLP_future(u_l)
q_m = MLP_machine(machine_features_m)
α_l = MaskedSoftmax(MLP_attn([h_l, q_m]))
future_context_m = Σ_l α_l h_l
```

该前瞻表示的作用是估计当前 Machine 被未来紧急 Lot 竞争的程度。例如，如果短时间内会到达高优先级、低 due slack 且只能在当前 Machine 加工的 Lot，DDT 应能够学习到等待的潜在价值。反过来，如果未来窗口内没有紧急且依赖当前 Machine 的 Lot，则当前空闲 Machine 更适合立即派工。

DDT 的动作空间定义为：

```text
d_t ∈ {
  dispatch_now,
  wait_next_lot,
  wait_next_machine,
  wait_delta
}
```

SAS agent 负责在 DDT 选择 `dispatch_now` 后执行具体派工决策。其动作仍然是固定长度候选池索引：

```text
a_t ∈ {0, 1, ..., K_action - 1}
candidate_actions[a_t] = (lot_t, ppid_t)
execute_action = (lot_t, machine_t, ppid_t)
```

这种双智能体结构使本项目不再只是学习“当前机台选哪个 Lot”，而是同时学习“什么时候值得派工”和“派哪个 Lot / PPID 更好”。

### 8.2 双注意力 SAS Actor

SAS Actor 在基础机台条件化注意力的基础上升级为双注意力结构，包含两个并行分支。

Lot-PPID 注意力分支输入候选动作本身的紧迫性和工艺特征：

```text
x_i = [
  lot_due_date_urgency,
  lot_qtime_risk,
  lot_priority,
  lot_waiting_time,
  wafer_count,
  ppid_step_count,
  ppid_process_time_mean,
  ppid_process_time_sum
]
```

其注意力表示为：

```text
E_lot = SelfAttention(x_1, x_2, ..., x_{K_action}, mask)
```

Machine-Chamber 分支输入候选动作与资源日历的匹配特征：

```text
y_i = [
  machine_next_available_time,
  machine_short_window_utilization,
  chamber_side_short_window_utilization,
  estimated_earliest_start,
  estimated_finish_time,
  chamber_conflict_risk,
  insertion_slack,
  ppid_required_chamber_side_count
]
```

其注意力表示为：

```text
E_res = CrossAttention(machine_context, y_1, y_2, ..., y_{K_action}, mask)
```

两个注意力分支融合后输出候选动作概率：

```text
h_i = concat(E_lot_i, E_res_i, x_i, y_i)
score_i = MLP(h_i)
π_SAS(a_i | s_t, A_fixed^m) = MaskedSoftmax(score_i)
```

这里的 `mask` 仍然只屏蔽结构性不可行动作和 padding 动作。Q-time 风险、due date 紧迫度和 priority 偏好不合并为 hard mask，而是分别进入特征和 reward。

Critic 可以共享双注意力表示，并结合全局状态特征估计状态价值：

```text
g_t = Pooling(E_lot, E_res, global_features)
V_SAS(s_t) = MLP(g_t)
```

其中 `global_features` 可以包括当前平均机台利用率、未完成 Lot 数、最大 Q-time 风险、最大 due date 紧迫度、空闲 Machine 数和短窗口内预计到达 Lot 数。

### 8.3 DDA-PPO 训练流程

DDA-PPO 可采用共享环境、双策略头、双 value head、联合 PPO 更新的训练方式：

```text
Algorithm: DDA-PPO

Input:
    Resource calendars C
    Waiting lots L
    Machines M
    Feasible PPIDs P
    Candidate size K_action
    DDT policy π_DDT
    SAS policy π_SAS
    Value networks V_DDT and V_SAS

Output:
    Dispatch decisions and updated calendars

1. Observe full calendar state s_t
2. Extract current dispatch features and future-lot context z_t^DDT
3. DDT selects decision-time action d_t ~ π_DDT(d | z_t^DDT)
4. If d_t != dispatch_now:
       advance environment time according to d_t
       compute DDT reward r_DDT
       store DDT transition
       continue
5. If d_t = dispatch_now:
       select idle machine m
       build structurally feasible candidate pool A_t^m
       obtain A_fixed^m by Top-K / padding / mask
       compute candidate features x_i and resource features y_i
6. If candidate pool is empty:
       do not run SAS softmax
       return to DDT for wait / time advance
7. SAS selects action_index ~ π_SAS(action_index | z_t^SAS)
8. Resource calendar tries TryInsert(lot, machine, ppid)
9. If insertion succeeds, commit calendars and compute r_SAS
10. If insertion fails during training:
       keep the sampled action in the trajectory
       give failure penalty
       execute wait / no-op or event advance
11. Store SAS transition with A_fixed^m, action_index and logπ_SAS
12. Repeat until the episode ends
13. Compute GAE advantages for DDT and SAS transitions
14. Update π_DDT, π_SAS, V_DDT and V_SAS with PPO clipped objective
```

在该流程中，未来未到达 Lot 的特征提取发生在 DDT 选择动作之前；SAS 的候选池仍只包含当前已到达且结构可行的 `(Lot, PPID)`。因此，future context 改变的是“是否等待”的状态判断，而不是当前资源日历可以执行的动作集合。

训练阶段必须保持 PPO 轨迹一致性：SAS 采样了哪个动作，轨迹中就记录哪个动作。即使资源日历插入失败，也不能在训练轨迹中替换成下一个可行动作。推理阶段则可以采用工程 fallback：按照 SAS 概率从高到低尝试候选动作，提交第一个资源日历可插入的动作；若全部失败，再执行 wait / no-op。

### 8.4 可解释性输出

为了增强算法汇报和实际应用可信度，DDA-PPO 可以在每次决策后输出解释信息：

```text
DDT 解释:
  为什么立即派工或等待
  等待前后候选池质量如何变化
  等待造成的 idle / Q-time 风险 / due date 拖期风险变化

SAS 解释:
  当前 Machine 的 Top-K 候选动作概率
  Lot-PPID 注意力主要关注哪些候选
  Machine-Chamber 注意力主要关注哪些资源冲突
  被 mask 的动作及其原因
  最终 reward 中拖期、Q-time、利用率、优先级各项贡献
```

因此，DDA-PPO 的创新点可以总结为：在候选池和资源日历可行性校验的基础上，用 DDT agent 学习动态决策时机，用 SAS agent 学习具体 Lot / PPID 派工动作，并通过 Lot-PPID 与 Machine-Chamber 双注意力同时建模交期紧迫性和底层资源竞争关系。

### 8.5 失败场景与替代方案

DDA-PPO 是最终研究目标，但实际训练中可能遇到失败场景，因此需要给出保底改法：

1. **候选池过小或经常为空**：先退化为规则触发的派工机制，只让 SAS 在可行动作中做选择，暂时不启用 DDT。
2. **双注意力训练不稳定**：先退化为单注意力 SAS-PPO，只保留 Lot-PPID 注意力分支，把 Machine-Chamber 竞争信息作为手工特征输入。
3. **reward 尺度难以统一**：先使用更简单的加权标量化 reward，只保留拖期、Q-time、利用率和失败惩罚四类核心项。
4. **PPO 收敛缓慢或振荡**：先采用规则派工或启发式策略作为 warm start，再进行强化学习微调。
5. **调度时机学习过于复杂**：先不引入 DDT，把研究重点放在 SAS 的候选动作选择上。

因此，DDA-PPO 不是唯一实现路径，而是一个分阶段推进的研究目标：先做候选池动作选择，再做单注意力策略，再做双注意力策略，最后加入 DDT 形成完整双智能体框架。

## 9. SAS 单独训练阶段的训练闭环、Reward 与评价定义

本章只描述基础阶段 **SAS 单独训练** 的闭环。此阶段不训练 DDT，也不设计双智能体联合 reward。DDT 的“何时派工”奖励应等 SAS 单 PPO 能稳定生成完整 schedule 后，再作为下一阶段单独加入。

SAS 单训阶段的目标是：在当前规则选定的 Machine 和固定长度候选池中，学习选择一个可执行、稳定、局部较优的 `(Lot, PPID)`。因此，reward 设计应坚持三个原则：

1. **小尺度**：单步 reward 尽量控制在 `[-1, 1]`；
2. **局部性**：优先使用当前动作直接产生的信息，不在每一步计算复杂全局目标；
3. **渐进式**：先用成功 / 失败信号训练可行派工，再逐步加入拖期、Q-time 和 priority shaping。

需要特别说明：最终实验评价仍然保留多目标指标；但 SAS 训练 reward 不是直接把所有目标完整相加，而是使用小权重、裁剪后的加权标量化信号。

### 9.1 Episode、rollout 与规则触发

SAS 单独训练时，episode 仍可定义为一个 planning window 内的完整派工过程：

- **planning window**：从当前时刻 `t_now` 到 `t_now + H` 的调度窗口；
- **episode**：一个 planning window 内从初始资源日历到窗口结束或全部 Lot 完成的过程；
- **rollout**：PPO 收集的一段 SAS transition 序列。

此阶段不由 DDT 决定是否派工，而是由环境规则触发 SAS：

```text
如果存在空闲 Machine 且候选池非空：
    选择当前决策 Machine
    调用 SAS 选择 action_index
否则：
    执行 wait / no-op 或推进到下一事件时刻
```

多台 Machine 同时空闲时，建议先使用固定规则选择一个当前决策 Machine：

```text
machine_t = argmin_m (next_available_time_m, num_feasible_candidates_m)
```

这样可以让 SAS 只学习“当前 Machine 选哪个候选动作”，避免一开始同时学习 Machine 选择、是否等待和 Lot / PPID 选择。

### 9.2 SAS transition 字段

每一步 SAS transition 建议记录为：

```text
SAS_transition = (
  obs_t,
  machine_t,
  candidate_features_t,
  candidate_mask_t,
  action_index_t,
  logπ_old_t,
  reward_t,
  obs_{t+1},
  done_t,
  info_t
)
```

其中 `info_t` 至少包含以下字段，便于 reward 计算和日志分析：

```text
info_t = {
  selected_lot,
  selected_ppid,
  insertion_success,
  insertion_failed,
  mask_invalid,
  wait_or_noop,
  selected_lot_start,
  selected_lot_end,
  selected_lot_process_time,
  new_qtime_violation,
  priority_rank_penalty,
  reward_execute,
  reward_wait,
  reward_shape,
  reward_terminal
}
```

这些字段都可以由候选池、资源日历 dry-run / commit 结果、Lot schedule 和 Wafer schedule 直接得到，不要求网络或环境额外推断复杂全局状态。

### 9.3 基础 reward 公式

基础 SAS reward 写为：

```text
r_SAS = clip(r_execute + r_wait + r_shape + r_terminal, -1.0, 1.0)
```

第一版训练建议使用：

```text
r_shape = 0
r_terminal = 0
```

也就是：

```text
r_SAS = clip(r_execute + r_wait, -1.0, 1.0)
```

执行结果奖励为：

```text
r_execute =
  +0.20,  if insertion_success = True
  -0.40,  if insertion_failed = True
  -0.50,  if mask_invalid = True
   0.00,  otherwise
```

wait / no-op 奖励为：

```text
r_wait =
  -0.02,  if wait_or_noop = True
   0.00,  otherwise
```

这样设计的目的，是先让 SAS 学会两个最基础的行为：

1. 尽量选择资源日历可以插入的动作；
2. 避免采样 padding、mask=0 或明显不可执行的动作。

### 9.4 可选轻量 shaping 项

当基础版本已经能稳定生成完整 schedule 后，再加入轻量 shaping：

```text
r_shape =
  -0.05 × tardy_norm
  -0.03 × qtime_norm
  -0.02 × priority_rank_penalty
  +0.03 × progress_norm
```

各项定义如下。

**拖期项**只针对本次 selected lot：

```text
tardy_norm = clip(max(0, selected_lot_end - due_date[selected_lot]) / H, 0, 1)
```

**Q-time 项**只统计本次动作新增的 selected lot wafer schedule：

```text
qtime_norm = clip(new_qtime_violation / max(1, selected_lot_process_time), 0, 1)
```

其中 `new_qtime_violation` 的计算方式与当前代码 `compute_q_time_violation()` 一致，但只对本次动作新增的 selected lot wafer rows 计算，不在每一步重新计算全局 Q-time 总违背。

**priority 项**只使用当前候选池内的局部优先级排序惩罚：

```text
priority_rank_penalty =
  count(priority(l') > priority(selected_lot) for l' in real_candidates)
  / max(1, len(real_candidates) - 1)
```

它不使用最终 schedule 的全局 `priority_violation` 逆序计数作为 step reward，因为全局逆序计数尺度较大、延迟较长，容易导致 PPO 更新不稳定。

**进度项**使用本次成功插入的 Lot 占用时长：

```text
progress_norm = clip((selected_lot_end - selected_lot_start) / H, 0, 1)
```

该项只作为很小的正向信号，避免策略过度 wait。若发现策略偏向长加工时间 Lot，可以将 `progress_norm` 权重降为 0。

### 9.5 终局奖励的使用时机

SAS 单训初期建议不使用终局奖励：

```text
r_terminal = 0
```

当满足以下条件后，才加入很小的终局辅助奖励：

1. 大多数 episode 能生成完整 Lot-level 和 Wafer-level schedule；
2. 资源日历冲突和插入失败明显减少；
3. PPO loss 和 episode return 不再大幅振荡。

终局奖励可写为：

```text
r_terminal = -0.05 × final_score_norm
```

其中：

```text
final_score_norm =
  0.40 × tardy_count / num_lots
  + 0.30 × clip(total_tardiness / (num_lots × H), 0, 1)
  + 0.20 × q_time_violation_count / max(1, num_qtime_checks)
  + 0.10 × clip(priority_violation / priority_scale, 0, 1)
```

该项只在 episode 结束时加入一次，不能作为每一步 reward 重复加入。若加入后训练不稳定，应立即关闭终局奖励，退回只使用 step reward 的版本。

### 9.6 代码实现伪代码

基础 SAS reward 可以实现为一个独立函数：

```text
function compute_sas_reward(info, config):
    reward = 0.0

    if info.mask_invalid:
        reward += config.mask_invalid_penalty      # -0.50
        return clip(reward, -1.0, 1.0)

    if info.wait_or_noop:
        reward += config.wait_penalty              # -0.02
        return clip(reward, -1.0, 1.0)

    if info.insertion_failed:
        reward += config.insert_fail_penalty       # -0.40
        return clip(reward, -1.0, 1.0)

    if info.insertion_success:
        reward += config.insert_success_reward     # +0.20

        if config.use_light_shaping:
            tardy_norm = clip(
                max(0, info.selected_lot_end - due_date[info.selected_lot]) / H,
                0,
                1
            )

            qtime_norm = clip(
                info.new_qtime_violation / max(1, info.selected_lot_process_time),
                0,
                1
            )

            progress_norm = clip(
                (info.selected_lot_end - info.selected_lot_start) / H,
                0,
                1
            )

            reward += -0.05 * tardy_norm
            reward += -0.03 * qtime_norm
            reward += -0.02 * info.priority_rank_penalty
            reward += +0.03 * progress_norm

    return clip(reward, -1.0, 1.0)
```

推荐配置为：

```text
RewardConfig:
  insert_success_reward = +0.20
  insert_fail_penalty = -0.40
  mask_invalid_penalty = -0.50
  wait_penalty = -0.02
  tardy_weight = -0.05
  qtime_weight = -0.03
  priority_weight = -0.02
  progress_weight = +0.03
  terminal_weight = -0.05
  reward_clip_min = -1.00
  reward_clip_max = +1.00
  use_light_shaping = False at first, True after stable insertion
  use_terminal_reward = False at first, True only after stable schedule completion
```

### 9.7 推荐训练顺序

SAS 单训建议采用三阶段 reward，而不是一开始使用完整目标：

```text
阶段 A：可行性训练
r_SAS = r_execute + r_wait
目标：学会选择可插入动作，减少 mask invalid 和 insertion failed。

阶段 B：局部质量训练
r_SAS = r_execute + r_wait + r_shape
目标：在保持可行的基础上，轻微偏向低拖期、低 Q-time 风险、高优先级和有效进度。

阶段 C：终局微调
r_SAS = r_execute + r_wait + r_shape + r_terminal
目标：让 episode-level 多目标评价略微参与训练，但不压过 step-level 主信号。
```

如果阶段 B 或阶段 C 出现不收敛，应优先关闭新增项，退回上一阶段。不要通过继续增加 reward 分量解决不收敛问题。

### 9.8 终止条件与失败处理

SAS 单训阶段的 episode 在以下任一条件满足时终止：

1. 所有 Lot 均已完成 Lot-level 与 wafer-level 排程；
2. `t_now` 超过当前 planning horizon，且没有可继续调度的已到达 Lot；
3. 候选池为空、未来无 Lot 到达、未来无 Machine 释放，环境无法继续推进；
4. `total_wait_steps_per_episode > max_total_wait_steps_per_episode`；
5. 连续插入失败次数超过 `max_failed_actions`，例如 `max_failed_actions = 3 × K_action`；
6. 出现资源日历不可恢复错误，例如已提交区间冲突、Lot / wafer stage 缺失或状态回滚失败。

训练阶段的失败处理规则为：

1. **采样到 padding 或 mask=0 动作**：不调用资源日历插入，返回 `-0.50`；
2. **候选池为空**：不执行 SAS softmax，执行 wait / no-op，返回 `-0.02`；
3. **资源日历 dry-run 失败**：保留原始 `(state, action, logπ)`，返回 `-0.40`，不提交任何日历修改；
4. **资源日历 commit 中途失败**：执行 rollback，恢复到动作前状态，返回 `-0.40`；
5. **连续失败过多**：触发 episode 终止条件，标记为 infeasible episode。

推理阶段不涉及 PPO 梯度更新，可以按 SAS 概率从高到低尝试候选动作，提交第一个资源日历可插入动作；若全部候选失败，再执行 wait / no-op。

### 9.9 Rollout 采样与 PPO 更新

SAS 单训阶段只维护一条策略轨迹：

```text
(z_t^SAS, action_index_t, logπ_SAS_old, r_SAS, z_{t+1}^SAS, done, candidate_mask_t)
```

PPO 概率比为：

```text
ρ_t^SAS(θ) =
  π_SAS,θ(action_index_t | z_t^SAS, candidate_mask_t)
  / π_SAS,old(action_index_t | z_t^SAS, candidate_mask_t)
```

Actor 使用 clipped objective：

```text
L_actor^SAS = - E_t [ min(
  ρ_t^SAS(θ) A_t,
  clip(ρ_t^SAS(θ), 1 - ε, 1 + ε) A_t
) ]
```

Critic 只估计 SAS 状态价值：

```text
V_SAS(s_t) = ValueHead_SAS(global_features, pooled_candidate_features)
```

总损失为：

```text
L_SAS-PPO =
  L_actor^SAS
  + c_v L_value^SAS
  - c_e H(π_SAS)
```

DDT 的 transition、DDT reward、双 value head 和 DDA-PPO 联合损失不在基础 SAS 单训阶段实现。等 SAS 单 PPO 稳定后，再进入 DDT 单独训练或 DDA-PPO 联合训练。

### 9.10 最终评价指标

无论 SAS 训练阶段使用多简化的 reward，最终实验评价都应分别报告以下指标：

```text
F(S) = [
  q_time_violation_count,
  total_q_time_violation,
  tardy_count,
  total_tardiness,
  -average_machine_utilization,
  priority_violation
]
```

其中平均机器利用率在目标向量中取负，是为了保持所有目标统一为最小化形式；在结果展示时应仍以正向利用率报告。这样可以清楚区分：训练阶段使用小尺度、可收敛的 SAS reward，实验阶段仍使用完整多目标指标向量评价。

## 10. 分阶段实现路线

为了降低实现风险，后续工作应按“环境稳定优先、策略复杂度逐步增加”的顺序推进。

### 阶段 1：资源日历环境 + 候选池 + mask

本阶段先不引入强化学习训练，而是把调度环境本身做稳定。核心工作是保留并完善 Machine、Chamber / Side 两层资源日历，使每个候选 `(Lot, Machine, PPID)` 都能够通过 dry-run 检查其可插入性，并在 commit 后生成一致的 Lot-level 和 Wafer-level 排程结果。

同时，需要建立固定长度候选池机制：在某台 Machine 需要决策时，从当前已到达且未完成的 Lot 中生成局部候选动作，再经过 Top-K、padding 和 action mask 得到 `A_fixed^m`。其中 mask 主要负责过滤结构性不可行动作，例如 Lot 未到达、Machine 不支持对应 Recipe、PPID 不可用、PPID stage 缺失或资源日历无法插入。该阶段的目标是得到一个可重复、可验证、不会产生资源冲突的基础环境。

### 阶段 2：规则触发的 SAS-PPO

在资源日历环境稳定后，先使用规则触发方式决定何时派工，例如当某台 Machine 空闲且候选池非空时立即触发 SAS 决策。此时还不学习“是否等待”，而是把学习重点集中在“在当前候选池中选择哪个 `(Lot, PPID)`”。

SAS-PPO 的 Actor 输出候选池索引 `action_index`，再由环境映射回真实动作并交给资源日历执行。训练 reward 先采用小尺度的 SAS 单步奖励：成功插入给小正奖励，插入失败、无效 mask 和 wait/no-op 给小惩罚；拖期、Q-time、priority 和进度只作为轻量 shaping 项，待基础策略收敛后再逐步加入。Critic 估计当前候选池状态下的价值。该阶段用于验证候选池动作空间、mask、资源日历执行和 PPO 更新能否形成完整闭环。

### 阶段 3：单注意力 SAS-PPO

在规则触发 SAS-PPO 能运行后，将 Actor 从普通 MLP 或简单打分器升级为单注意力结构。模型以当前 Machine 状态作为上下文，对候选池中的 `(Lot, PPID)` 动作进行 attention 编码，学习不同候选动作之间的相对优先级。

这一阶段的重点是处理不同实例规模下候选数量变化的问题。通过 padding + mask，模型始终面对固定长度输入，但 attention 只在有效候选上计算概率。单注意力 SAS-PPO 应能够学习 due date、Q-time、priority、waiting time、process time 等候选动作特征对派工选择的影响，为后续双注意力结构提供基线。

### 阶段 4：双注意力 SAS-PPO

在单注意力版本稳定后，引入双注意力 SAS-PPO。第一个分支为 Lot-PPID 注意力，主要建模候选 Lot 之间的交期紧迫度、Q-time 风险、优先级和等待时间竞争；第二个分支为 Machine-Chamber 注意力，主要建模候选动作对 Machine、Chamber 和 Side 资源日历的占用压力。

两个注意力分支融合后输出候选动作概率，Critic 可以共享融合后的状态表示。该阶段的目标是验证双注意力是否比单注意力更适合 FAB 机台组调度，因为它同时考虑了 Lot 层面的派工价值和底层资源日历的可插入性压力。实验上可以将其与阶段 3 进行消融对比。

### 阶段 5：加入 DDT 决策时间智能体与未来 Lot 前瞻编码

在 SAS 派工策略稳定后，再加入 DDT 决策时间智能体。DDT 不直接决定派哪个 Lot，而是判断当前应该 `dispatch_now` 还是等待，例如等待下一个 Lot 到达、等待下一台 Machine 空闲，或推进一个短时间步。

加入 DDT 后，调度过程被拆成两层：DDT 负责“何时决策”，SAS 负责“如何派工”。只有当 DDT 选择立即派工时，SAS 才进入当前 Machine 的候选池中选择 `action_index`。这一阶段需要单独设计 DDT reward，使其既避免无意义等待，也避免在候选质量较差时过早派工。

为了让 DDT 具备前瞻性，应在该阶段引入 Future Lot Encoder。该编码器以当前时刻为起点，选取未来窗口内尚未到达的 Lot，使用共享 MLP 编码每个 Lot 的到达时间、优先级、due slack、预计加工时间、wafer 数量和 Machine 兼容性，再用当前 Machine 状态作为 query 做 attention pooling，得到固定长度 `future_context`。该表示与当前候选池摘要和资源日历摘要拼接后输入 DDT-PPO，用于学习当前是否值得等待。

需要强调的是，未来未到达 Lot 在该阶段仍不进入 SAS 候选动作池，也不作为 action mask 的有效动作。它们只通过 `future_context` 影响 DDT 对 `dispatch_now / wait` 的判断。

### 阶段 6：完整 DDA-PPO

最后形成完整 DDA-PPO 框架，将 DDT、双注意力 SAS、资源日历环境、候选池、mask 和 PPO 训练流程合并。训练时分别记录 DDT transition 和 SAS transition，并在 rollout 或 episode 结束后统一计算 advantage，再使用 PPO clipped objective 更新两个 Actor 和对应 Critic。

完整版本应支持训练和推理两种模式。训练阶段保持采样轨迹一致性，即使资源日历插入失败也只返回失败惩罚，不临时替换为另一个动作；推理阶段则可以按照 SAS 概率从高到低尝试候选动作，选择第一个资源日历可执行方案作为 fallback。最终评估时，需要在 small、medium 和 large 实例上比较 Q-time 违背、拖期 Lot 数、总拖期时间、平均机器利用率和优先级违背等指标。

完整 DDA-PPO 的实现不宜直接从双智能体联合训练开始。更稳妥的顺序是先训练规则触发 SAS-PPO，再单独训练带 Future Lot Encoder 的 DDT-PPO，最后在同一资源日历环境中进行联合微调。这样可以区分“候选动作选择不稳定”和“等待时机学习不稳定”两类问题，降低调试难度。

## 11. 最终结论

根据 PPT，本项目本质上是一个面向 FAB 机台组的特殊并行机调度问题。其核心任务是在 Machine Group 内，为待排产 Lot 决定加工 Machine、PPID、上机 / 下机时刻，以及 wafer 在 Chamber / Side 上的加工顺序和时间。

当前代码已经较好地实现了这一问题的基础结构，包括 Lot / Wafer 两层排程、Machine 和 Chamber / Side 资源日历、动态滚动调度、NSGA-II 线性策略权重搜索和多目标评价。当前实现可以作为后续智能调度环境的基础，但不应被表述为已经实现了 DDA-PPO、神经网络策略或强化学习训练。

后续优化应在不改变 PPT 问题定义的前提下推进：

1. 显式对齐 Machine Group、Recipe、PPID 和 Chamber / Side 的关系；
2. 将调度触发从固定滚动时间逐步改为机台空闲或事件驱动触发；
3. 将调度过程建模为 MDP，明确完整环境状态、特征化网络输入、局部动作空间和状态转移；
4. 通过候选池把大规模 Lot / PPID 集合压缩为 Top-K 局部候选；
5. 使用 action mask 过滤结构性不可行动作；
6. 使用 SAS 派工动作智能体在候选池中选择 `(Lot, PPID)`；
7. 使用 Lot-PPID 注意力和 Machine-Chamber 竞争注意力共同建模交期紧迫性与底层资源冲突；
8. 使用 DDT 决策时间智能体学习“立即派工还是等待”的动态触发策略；
9. 使用 Machine-aware DeepSets + Attention Pooling 编码未来未到达 Lot，形成固定长度前瞻上下文；
10. 通过资源日历 dry-run / commit 保证候选方案真正可执行；
11. 先分别稳定 SAS-PPO 和 DDT-PPO，再进行 DDA-PPO 联合微调；
12. 输出 action mask、注意力权重、动作概率和 reward 分解，提高算法解释性。

因此，本项目后续方案可以概括为：

> **在 PPT 定义的 FAB 机台组调度问题基础上，保留当前资源日历与多目标评价框架，逐步构建一个以候选池控制、未来 Lot 前瞻编码、双智能体动态决策、双注意力动作选择、资源日历可行性校验和 PPO 训练为核心的智能调度原型。**
