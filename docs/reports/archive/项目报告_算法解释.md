# 面向半导体 FAB 机台组调度的 SAS + VC-MCTS 项目报告

> 本文档按 2026-06-08 的本地代码重写，重点不是“模块说明”，而是把当前实现的**算法流程、每个决策点的数学形式、以及为什么这样设计**写清楚。代码事实以 `FAB_RL/FABenv/` 为准；旧版 `项目报告4_VC-MCTS预留规划替代RMA版(2).md` 的叙事只作为历史参考。

---

## 0. 报告主线

当前系统已经不是“单纯的派工 PPO”了，而是一条完整的两层调度链：

1. 下层用固定规则完成工艺内部批处理和资源日历排程，负责给出完成时间分布和占用区间。
2. 上层 SAS 在候选池中选择当前 `(lot, ppid)`。
3. VC-MCTS 在有预留价值的决策点上，比较 `reserve / dispatch(or delegate_dispatch) / no_op` 的反事实结果，决定是否把机台留给未来 lot。
4. SAS policy 既可单独训练，也可作为 VC-MCTS 的 dispatch delegate、PUCT 先验；叶子估值只在 multi-head critic 打开时可用。

这意味着当前算法的核心不再是“奖励里学 hold”，而是“在真实资源日历上做反事实比较”。

---

## 1. 统一符号

记：

- `l` 为 lot；
- `m` 为 machine；
- `p` 为 PPID；
- `t` 为当前时刻；
- `candidate_set(t,m)` 为时刻 `t`、机台 `m` 的候选动作集；
- `policy_theta(a | obs_t)` 为 SAS policy 在观察 `obs_t` 下选择动作 `a` 的概率；
- `value_c(state)` 为多头 critic 在通道 `c` 上的价值估计；
- `deadline(l)` 为 lot `l` 的 Q-time deadline；
- `arrival_time(l)` 为 lot 到达时刻；
- `mu_finish(l,m,p), sigma_finish(l,m,p)` 为 `(l,m,p)` 的完成时间均值和标准差；
- `epsilon_qtime` 为 Q-time 机会约束容许违规率；
- `z_eps` 为对应分位点，例如 `epsilon_qtime=2%` 时约 `2.05`。

调度状态记为：

```text
s_t = { 当前时刻, 机器/腔体日历, 已完成 lots, 可见 lots, 前瞻 lots, 预留 ledger, ... }
```

## 2. 总体算法流程

当前 episode 的主流程可写为：

```text
while not done:
  1. 选出当前可派工机台 m*
  2. 构造机台 selected_machine 的候选池 candidate_set(t, selected_machine)
  3. 先做 Q-time mask，再做 priority filter，再做候选排序
  4. SAS policy 在固定长度候选池里采样/贪心选动作
  5. 试算 dry-run，再 commit 到真实日历
  6. 记录 reward vector 和 next state
  7. 若出现 ROP，则在 VC-MCTS 中比较 reserve / dispatch / no_op
```

其中第 7 步不是所有时刻都发生，只有 ROP 门控认为“当前存在预留价值”时才建树。

## 3. 下层：完成时间估计与资源日历排程

### 3.1 子批处理

一个 lot 的 wafer 数为 `N`，若 side capacity 为 `c`，则子批数为：

```text
B = ceil(N / c)
```

符号说明：

- `B`：该 lot 被拆成的子批数量；
- `N`：该 lot 的 wafer 总数；
- `c`：单个 chamber side 一次可处理的 wafer 容量；
- `ceil(...)`：向上取整，表示最后不足一批也要占用一个子批。

若 `c` 为空，则等价于单批：

```text
B = N
```

符号说明：

- 这里的 `B=N` 是代码里 `side_capacity` 未设置时的退化口径，表示不再按 side capacity 聚合成更少子批。

为什么要这样做：工艺内部通常是“同进同出”的批处理机，子批是把一个 lot 拆成若干可排的最小批处理单元。

### 3.2 确定性 list scheduling

下层共享核心是 `schedule_deterministic`。对每个子批 `b`、每个 stage `s`，其开始时刻由

```text
start(b,s) = max(ready(b,s), free_selected(s))
```

符号说明：

- `b`：子批编号；
- `s`：stage 编号；
- `start(b,s)`：子批 `b` 在 stage `s` 的开始加工时刻；
- `ready(b,s)`：子批在前一阶段完成后的可开工时间；
- `free_selected(s)`：stage `s` 中被选中资源的空闲时刻；
- `max` 表示必须同时满足“子批已经 ready”和“资源已经空闲”。

对应资源选择是：

```text
selected_inst = argmin over inst in resource_options(s):
    max(ready(b,s), free(inst))
```

符号说明：

- `inst`：stage `s` 的一个可选资源实例，即某个 `(machine, chamber, side)`；
- `selected_inst`：最终选中的资源实例；
- `resource_options(s)`：stage `s` 的可选资源集合；
- `free(inst)`：资源实例 `inst` 的当前空闲时刻；
- `argmin`：选择使最早开工时间最小的资源。

结束时刻为：

```text
end(b,s) = start(b,s) + process_time(b,s)
```

符号说明：

- `end(b,s)`：子批 `b` 在 stage `s` 的结束时刻；
- `process_time(b,s)`：子批 `b` 在 stage `s` 的加工时长；在确定性路径中取均值，在噪声路径中可为采样值。

并更新资源空闲时刻：

```text
free(selected_inst) <- end(b,s)
```

符号说明：

- `free(selected_inst)`：被选中资源实例的下一次可用时刻；
- `<-`：赋值更新，表示该资源被当前子批占用到 `end(b,s)`。

总完成时间：

```text
completion_time(l) = max over b end(b, last_stage)
```

符号说明：

- `completion_time(l)`：lot `l` 的完成时刻；
- `last_stage`：该 PPID 工艺路线的最后一个 stage；
- `max_b`：对所有子批取最晚结束时刻，因为 lot 只有所有子批完成后才算完成。

为什么选这个规则：  
1. 它是贪心但确定的，便于 `estimate` 与 `schedule_on_calendar` 共用同一核心；  
2. 它保证“估计路径”和“提交路径”算法一致，只是 free-time 来源不同；  
3. 对当前问题，内部批处理没有可学习的战略取舍，学习它只会增加方差。

### 3.3 Monte Carlo 完成时间分布

`estimate()` 对每次采样 `k` 生成一个 makespan `completion_sample(l,k)`，然后估计：

```text
mu_finish(l,m,p) =
    (1 / K) * sum over k=1..K completion_sample(l,k)

sigma_finish(l,m,p) =
    sqrt((1 / K) * sum over k=1..K
         squared(completion_sample(l,k) - mu_finish(l,m,p)))
```

符号说明：

- `mu_finish(l,m,p)`：lot `l` 放到 machine `m`、使用 PPID `p` 时完成时间的样本均值；
- `sigma_finish(l,m,p)`：同一动作完成时间的样本标准差；
- `K`：Monte Carlo 采样次数，对应代码里的 `n_mc`；
- `k`：第 `k` 次采样；
- `completion_sample(l,k)`：第 `k` 次采样下 lot `l` 的完成时刻；
- `l,m,p`：分别表示 lot、machine、PPID。

为什么用 Monte Carlo：因为多子批并行会产生 `E[max]` 偏差，简单的均值路径会系统性低估完成时间。

### 3.4 Q-time 机会约束

当前 Q-time mask 的核心判断是：

```text
deadline(l) - mu_finish(l,m,p) < z_eps * sigma_finish(l,m,p)
```

符号说明：

- `deadline(l)`：lot `l` 的 Q-time deadline；
- `mu_finish(l,m,p)`：该候选动作的预计完成时间均值；
- `sigma_finish(l,m,p)`：该候选动作完成时间的不确定性；
- `z_eps`：机会约束分位数，`epsilon_qtime` 越小，`z_eps` 越大，mask 越保守；
- 左侧 `deadline(l) - mu_finish(l,m,p)`：预计完成时间距离 deadline 的安全余量；
- 右侧 `z_eps * sigma_finish(l,m,p)`：考虑随机波动后的安全边际。

若上式成立，则认为该候选在容许风险下不可接受，动作被屏蔽。

等价地，动作可行要求：

```text
deadline(l) - mu_finish(l,m,p) >= z_eps * sigma_finish(l,m,p)
```

符号说明：

- 该式与上一式使用相同符号；
- 它表示“安全余量至少覆盖 `z_eps` 倍标准差”，因此动作通过 Q-time mask。

为什么要把 Q-time 写成机会约束，而不是奖励项：  
Q-time 是硬约束，不应该靠 reward shaping“劝一劝”。把它放进 mask，才能在搜索和 PPO 两边都保持一致的可行动作空间。

### 3.5 链式 Q-time

当前代码还支持链式模式。设 Q-time 链集合为 `qtime_chains`，某候选动作 `action` 在一次 dry-run 下若存在任意链窗违约，则该次样本记为 1。则：

```text
chain_violation_prob(action) =
    (1 / K) * sum over k=1..K
        indicator(sample k has any chain-window violation)
```

符号说明：

- `action`：待评估候选动作；
- `chain_violation_prob(action)`：动作 `action` 在链式 Q-time 检查中的估计违约概率；
- `K`：chain-joint 模式的 dry-run 采样次数，对应 `qtime_chain_mc`；
- `indicator(...)`：指示函数，条件成立取 1，否则取 0；
- “链窗违约”：任意 `(stage_i, stage_j)` 的真实 Q-time 间隔超过 `q_time_limits`。

若

```text
chain_violation_prob(action) > chain_threshold
```

符号说明：

- `chain_threshold`：允许的链式违约样本比例阈值，对应 `qtime_chain_threshold`；
- 当前默认 `chain_threshold=0`，表示只要任一样本中出现链窗违约，就屏蔽该动作。

若上式成立，则屏蔽。

为什么要有 chain 模式：聚合 deadline proxy 会漏掉 stage 间真实窗口，尤其是 `(1,2)`、`(2,3)` 这种链式约束。

## 4. 上层 SAS：候选池、打分、动作选择

### 4.1 候选池构造

对机台 `m`，结构可行动作集为：

```text
candidate_set(t,m) =
    {(l,p) | lot l is compatible with machine m and ppid p}
```

符号说明：

- `candidate_set(t,m)`：时刻 `t`、机台 `m` 的结构可行动作集合；
- `(l,p)`：候选动作，表示选择 lot `l` 和 PPID `p`；
- “兼容”：lot 可用该 PPID，且 machine `m` 支持该 PPID/工艺路线。

然后依次做：

```text
candidate_set(t,m)
  -> qtime_safe_set(t,m)
  -> priority_filtered_set(t,m)
  -> topk_pool(t,m,K)
```

符号说明：

- `qtime_safe_set(t,m)`：经过 Q-time mask 后保留下来的动作集合；
- `priority_filtered_set(t,m)`：再经过 priority filter 后的动作集合；
- `topk_pool(t,m,K)`：最终截断或填充成固定长度 `K` 的候选池；
- `K`：候选池大小，对应环境参数 `top_k`。

为什么要这样分层：  
Q-time 是硬约束，priority 是强偏好，utilization 才是软目标。把它们混成一个分数，会让高优先级但高风险动作和安全但普通动作混在一起，不利于解释。

### 4.2 priority filter

soft 模式下不删动作，只排序。strict 模式保留：

```text
priority(action) >= max_priority_in_pool - priority_gap
```

符号说明：

- `action`：待判断的候选动作；
- `priority(action)`：动作 `action` 对应 lot 的 priority；
- `max_priority_in_pool`：当前候选池中的最高 priority；
- `priority_gap`：priority 容差，对应 `priority_min_gap`；
- strict 模式下，只有 priority 接近最高优先级的动作会保留。

为什么优先级不直接写进 reward：  
priority 不是弱“偏好”，而是强约束式排序条件。用 filter 比用 reward 更稳定，也更符合字典序。

### 4.3 候选打分

对每个真实候选 `a=(l,p)`，代码构造的启发式分数可以写成：

```text
score(action) =
    due_urgency(action)
    + waiting_time(action)
    + 0.1 * qtime_slack(action)
    - 0.001 * total_process_time(action)
    - 0.001 * qtime_risk(action)
```

符号说明：

- `score(action)`：候选动作 `action` 的启发式排序分；
- `due_urgency(action)`：交期紧迫度；
- `waiting_time(action)`：该 lot 已等待时间；
- `qtime_slack(action)`：Q-time 安全余量；
- `total_process_time(action)`：总加工时长；
- `qtime_risk(action)`：Q-time 风险特征。

且：

```text
qtime_slack(action) = max(0, deadline(l) - predicted_completion(action))
```

符号说明：

- `qtime_slack(action)`：动作 `action` 的 Q-time slack 特征；
- `deadline(l)`：动作 `action` 所选 lot `l` 的 Q-time deadline；
- `predicted_completion(action)`：动作 `action` 的预计完成时刻，来自 dry-run 的 `predicted_completion`；
- `max(0,·)`：若预计已经超过 deadline，则 slack 记为 0。

为什么这样打分：  
它并不试图“学会全局最优”，只是在 Q-time safe 且 priority 通过过滤后，给 SAS 一个稳定的局部排序信号。

### 4.4 SAS 动作选择

policy 输出候选池上的 masked softmax：

```text
policy_prob(i | obs_t) =
    exp(logit_i) * valid_mask_i
    / sum over j [exp(logit_j) * valid_mask_j]
```

符号说明：

- `policy_prob(i | obs_t)`：SAS policy 在观察 `obs_t` 下选择候选索引 `i` 的概率；
- `obs_t`：当前 observation，包括候选特征、mask 和全局特征；
- `logit_i`：actor 对第 `i` 个候选输出的 logit；
- `valid_mask_i`：第 `i` 个候选的 mask，`1` 表示有效，`0` 表示 padding/无效；
- `j`：候选池内的索引；
- 分母只对有效动作归一化，因此无效动作概率为 0。

训练时采样，推理时一般取：

```text
selected_index(t) = argmax over i policy_prob(i | obs_t)
```

符号说明：

- `selected_index(t)`：时刻 `t` 最终选中的候选索引；
- `argmax_i`：选择概率最大的候选索引；
- 推理/greedy evaluation 走该式，训练时则从 `policy_prob` 中采样。

为什么使用候选池而不是直接对所有 lot 建模：  
当前问题的动作本质是“在当前机台上选一个可执行 `(lot, ppid)`”，固定长度候选池更容易处理 mask、padding 和 PPO 轨迹一致性。

### 4.5 机台选择

当多个机器同时可派工时，`Phase2EpisodeDriver` 的机台排序是 lexicographic：

```text
selected_machine =
    argmin over m in dispatchable_machines:
        (machine_available_time(m), valid_real_action_count(m), machine_id(m))
```

符号说明：

- `selected_machine`：当前被选中进入决策的 machine；
- `dispatchable_machines`：当前可派工 machine 集合；
- `machine_available_time(m)`：machine `m` 的最早可用时刻；
- `valid_real_action_count(m)`：该机台真实候选数；
- `machine_id(m)`：机台编号平局项；
- `argmin` 对三元组做字典序比较，即先比较可用时刻，再比较候选数，最后比较 machine 编号。

为什么这样选：  
先处理最早空闲的机器，再处理候选较多的机器，再按编号打破平局，可以避免调度顺序不稳定，也减少组合动作空间。

## 5. SAS 奖励与多头 PPO

### 5.1 标量奖励

标量奖励由执行奖励、等待奖励、塑形奖励和终态奖励构成：

```text
scalar_reward(t) =
    reward_exec(t)
    + reward_wait(t)
    + reward_shape(t)
    + reward_terminal(t)
```

符号说明：

- `scalar_reward(t)`：时刻 `t` 的标量 SAS reward；
- `reward_exec(t)`：执行合法性相关奖励；
- `reward_wait(t)`：等待动作奖励/惩罚，当前默认 0；
- `reward_shape(t)`：可选轻量塑形奖励；
- `reward_terminal(t)`：可选终态奖励。

其中执行奖励的逻辑是：

```text
reward_exec(t) =
    mask_invalid_penalty    if mask invalid
    0                       if wait / noop
    insert_fail_penalty     if insertion failed
    insert_success_reward   if insertion success
```

符号说明：

- `mask_invalid_penalty`：选择 masked/padding 动作的惩罚；
- `insert_fail_penalty`：插入失败惩罚；
- `insert_success_reward`：成功插入奖励；
- `wait / noop`：候选池无可执行动作或显式 wait 的情况，SAS 默认不惩罚。

为什么 wait 不罚：  
SAS 不负责全局 wait 决策，wait 的成本属于更高层的时机策略，不应在 SAS 这里重复惩罚。

### 5.2 向量奖励

当前多头路径使用三通道：

```text
reward_vector(t) = (reward_exec(t), reward_qtime(t), reward_util(t))
```

符号说明：

- `reward_vector(t)`：时刻 `t` 的向量 reward；
- `reward_exec(t)`：执行通道；
- `reward_qtime(t)`：Q-time residual 通道；
- `reward_util(t)`：utilization 终局通道；
- 通道顺序对应代码中的 `("exec", "qtime", "util")`。

其中：

```text
reward_exec(t) =
    insert_success_reward + w_pack * total_work / span
```

符号说明：

- `reward_exec(t)`：成功插入时的执行通道奖励；
- `insert_success_reward`：成功插入基础奖励；
- `w_pack`：packing 信号权重；
- `total_work`：该 lot 的总工作量估计；
- `span`：该 lot 从开始到结束的占用跨度；
- `total_work/span`：packing efficiency，跨度越长、拥塞越明显，该值越小。

```text
reward_qtime(t) = - new_qtime_violation(t) / num_lots
```

符号说明：

- `reward_qtime(t)`：Q-time 通道奖励，非正数；
- `new_qtime_violation(t)`：本次 commit 新增的 Q-time 违约数量或增量；
- `num_lots`：实例中的 lot 总数，用于归一化；
- 负号表示违约越多奖励越低。

```text
reward_util(t) =
    is_terminal(t) * avg_machine_utilization
```

符号说明：

- `reward_util(t)`：utilization 通道奖励；
- `is_terminal(t)`：指示函数，只有 episode 终止步为 1；
- `avg_machine_utilization`：当前完整日历下的平均机台利用率。

为什么把 Q-time 单独做一通道：  
它是最需要被“看到”的残差项，和 utilization 混在一起会让优势估计不稳定。

### 5.3 多头 critic

多头 critic 学：

```text
value(state) = {
    value_exec(state),
    value_qtime(state),
    value_util(state)
}
```

符号说明：

- `value(state)`：状态 `state` 的多头价值估计集合；
- `value_exec(state)`：exec 通道价值；
- `value_qtime(state)`：qtime 通道价值；
- `value_util(state)`：util 通道价值。

逐通道 GAE 为：

```text
td_residual(t,c) =
    reward(t,c) + gamma * value_c(state_next) - value_c(state_current)
```

符号说明：

- `td_residual(t,c)`：通道 `c` 在时刻 `t` 的 TD residual；
- `c`：reward/critic 通道，可取 `exec`、`qtime`、`util`；
- `reward(t,c)`：通道 `c` 的即时 reward；
- `gamma`：折扣因子；
- `value_c(state_current)`、`value_c(state_next)`：当前状态和下一状态的通道价值。

```text
advantage(t,c) =
    sum over offset >= 0:
        pow(gamma * gae_lambda, offset) * td_residual(t + offset, c)
```

符号说明：

- `advantage(t,c)`：通道 `c` 的 GAE advantage；
- `offset`：从当前时刻往后的偏移步数；
- `gae_lambda`：GAE 衰减参数，不是 Q-time Lagrangian 的对偶变量；
- `td_residual(t + offset, c)`：未来第 `offset` 步的 TD residual。

然后按通道归一化并加权：

```text
combined_advantage(t) =
    sum over c in {exec, qtime, util}:
        channel_weight(c) * normalize(advantage(t,c))
```

符号说明：

- `combined_advantage(t)`：PPO policy loss 使用的合成 advantage；
- `channel_weight(c)`：通道 `c` 的权重；若启用 Q-time Lagrangian，`qtime` 通道权重由对偶变量替代；
- `normalize(advantage(t,c))`：对通道 advantage 做均值方差归一化；
- 求和范围为当前多头通道集合。

为什么不用单头 value：  
因为 `exec / qtime / util` 的量纲、稀疏程度和终局属性完全不同，单头会把梯度混在一起。

### 5.4 PPO 目标

策略比率为：

```text
ppo_ratio(t) =
    policy_theta(action_t | state_t)
    / policy_old(action_t | state_t)
```

符号说明：

- `ppo_ratio(t)`：PPO 的新旧策略概率比；
- `policy_theta`：当前待更新策略；
- `policy_old`：采样 rollout 时的旧策略；
- `action_t`：时刻 `t` 实际采样并写入 buffer 的动作；
- `state_t`：时刻 `t` 的状态或 observation。

PPO clipped objective 为：

```text
policy_loss =
    - mean over minibatch:
        min(
            ppo_ratio(t) * combined_advantage(t),
            clip(ppo_ratio(t), 1 - clip_ratio, 1 + clip_ratio)
                * combined_advantage(t)
        )
```

符号说明：

- `policy_loss`：策略损失，代码中取负号做最小化；
- `combined_advantage(t)`：上一节得到的合成 advantage；
- `clip_ratio`：PPO clip ratio；
- `clip(...)`：把概率比限制在 `[1 - clip_ratio, 1 + clip_ratio]`；
- `mean over minibatch`：对 minibatch 样本取平均。

value loss 为：

```text
value_loss =
    sum over channel c:
        value_coef * squared(value_c(state_t) - return(t,c))
```

符号说明：

- `value_loss`：多头 value loss；
- `c`：通道索引；
- `value_coef`：value loss 权重；当前实现中各通道使用同一个值；
- `return(t,c)`：通道 `c` 的 return；
- `value_c(state_t)`：通道 `c` 的 critic 预测。

总损失为：

```text
total_loss = policy_loss + value_loss - entropy_coef * entropy(policy_theta)
```

符号说明：

- `total_loss`：最终优化的总损失；
- `entropy_coef`：entropy 系数；
- `entropy(policy_theta)`：策略熵，用于鼓励探索；
- 负号表示熵越大，总损失越小。

为什么仍然用 PPO：  
当前候选池是离散、可 mask、可截断的，PPO 的实现简单、稳定，而且和 rollout buffer 的轨迹一致性要求匹配。

### 5.5 Q-time Lagrangian

当启用 `use_qtime_lagrangian=True` 时，Q-time 权重由对偶变量 `lambda_qtime` 接管：

```text
lambda_qtime <-
    clip(
        lambda_qtime + lambda_lr * (qtime_cost - qtime_budget),
        0,
        lambda_max
    )
```

符号说明：

- `lambda_qtime`：Q-time Lagrangian 对偶变量；
- `lambda_lr`：对偶变量学习率，对应 `qtime_lambda_lr`；
- `qtime_cost`：当前 episode 估计的 Q-time cost；
- `qtime_budget`：可容忍的 Q-time cost budget；
- `lambda_max`：对偶变量上界；
- `clip(...,0,lambda_max)`：把更新后的 `lambda_qtime` 限制在 `[0,lambda_max]`。

其中 `qtime_cost` 的计算为：

```text
qtime_cost = - sum over t reward_qtime(t)
```

符号说明：

- `reward_qtime(t)`：第 `t` 步 Q-time reward，通常小于等于 0；
- `- sum reward_qtime(t)`：把负奖励还原为正的违约成本；
- 求和范围是一个 episode 内的所有步骤。

为什么要引入 Lagrangian：  
固定 `w_qtime` 容易在不同实例和不同噪声水平下失效，而对偶上升能把约束压力自动调到更合适的位置。

## 6. VC-MCTS：在线预留决策

### 6.1 为什么要用搜索而不是再学一个 RMA

预留动作本质上是一个反事实问题：

- 现在把机台派给当前 lot；
- 或者把它留给未来可能到达的高优先级 lot；
- 两条分支谁更好，只有把整条资源日历向前模拟之后才知道。

这种结构非常适合搜索，不适合直接靠稀疏 reward 端到端学习，所以当前方案把学习式预留替换成了 VC-MCTS。

### 6.2 ROP：预留机会点

ROP 只是门控，不是最终决策。它回答的是：

```text
exists reservation_pair(machine m, future_lot h)
```

符号说明：

- `exists`：存在；
- `m`：当前可预留的 idle machine；
- `h`：前瞻窗口内可见的 future lot；
- `(m,h)`：一个候选预留对；
- 该式表达的是 ROP 的门控语义：只要存在候选预留对，就允许进入 VC-MCTS 搜索。

若没有 ROP，就不建预留树，直接按 SAS 派工。

### 6.3 根动作空间

当前 root 动作空间是：

```text
root_actions =
    {no_op}
    union {dispatch(machine m) or delegate_dispatch(machine m)}
    union {reserve(machine m, future_lot h)}
```

符号说明：

- `root_actions`：VC-MCTS 在当前 root decision 上可选的动作集合；
- `no_op`：不派工、不预留，让时间继续推进；
- `dispatch(m)`：立即在 machine `m` 上派工一个候选 lot；
- `delegate_dispatch(m)`：由 rule 或 SAS delegate 为 machine `m` 选择具体派工动作；
- `reserve(m,h)`：把 machine `m` 预留给未来 lot `h`；
- `union`：集合并集。

其中：

- 若 `use_delegate_dispatch=False`，则 root 展开的是多个 `dispatch(m)` 边，具体 `(lot, ppid)` 直接由候选池中的高分动作给出；
- 若 `use_delegate_dispatch=True`，则 root 展开一个 `delegate_dispatch(m)` 边，具体 `(lot, ppid)` 由 `RuleDispatchDelegate` 或 `SASPolicyDispatchDelegate` 给出。

为什么要把 dispatch 委托出去：  
VC-MCTS 只需要决定“要不要留、留给谁”，派工细节交给已经训练好的 SAS，更符合模块分工，也能减少 root 分支。

### 6.4 目标比较

`VCMCTSObjective` 的字典序比较可写成：

```text
objective(action) =
    (
        qtime_violation_count(action),
        qtime_violation_total(action),
        priority_weighted_wait(action),
        -avg_utilization(action)
    )
```

符号说明：

- `objective(action)`：动作 `action` 的 VC-MCTS 目标向量；
- `qtime_violation_count(action)`：Q-time 违约数量；
- `qtime_violation_total(action)`：Q-time 违约严重度总量；
- `priority_weighted_wait(action)`：按 priority 加权的等待时间；
- `avg_utilization(action)`：平均机台利用率；
- `-avg_utilization(action)`：因为目标向量按越小越好比较，而 utilization 越大越好，所以取负号。

比较规则是 lexicographic：

```text
action_1 is better than action_2
    if objective(action_1) is lexicographically smaller than objective(action_2)
```

符号说明：

- `action_1, action_2`：两条待比较的 root action；
- `is better than`：优于；
- `lexicographically smaller`：按字典序更小；
- “字典序”表示先比较 `qtime_violation_count`，若相同再比较 `qtime_violation_total`，再比较 `priority_weighted_wait`，最后比较 `-avg_utilization`。

也可以写成一个用于 UCT 的标量评分：

```text
objective_score(action) =
    - qtime_penalty * qtime_violation_count(action)
    - qtime_total_penalty * qtime_violation_total(action)
    - priority_weighted_wait(action)
    + util_weight * avg_utilization(action)
```

符号说明：

- `objective_score(action)`：UCT selection 阶段使用的标量 exploitation score；
- `qtime_penalty`：Q-time violation count 惩罚系数；
- `qtime_total_penalty`：Q-time violation total 惩罚系数；
- `util_weight`：utilization 权重；
- `priority_weighted_wait(action)`：priority-weighted wait，越小越好，所以直接减去；
- `avg_utilization(action)`：平均机台利用率，越大越好，所以加上。

为什么先看 Q-time，再看 O2，再看 utilization：  
因为 Q-time 是硬约束，O2 是优先级等待的代理，utilization 只是最后才允许让步的软目标。

### 6.5 UCT / PUCT

每条边的选择分数是：

```text
uct_score(edge) =
    objective_score(mean_objective(edge))
    + exploration_c * prior(edge) * sqrt(log(total_visits) / visits(edge))
```

符号说明：

- `edge`：MCTS root 上的一条边；
- `uct_score(edge)`：边 `edge` 的选择分数；
- `mean_objective(edge)`：该边累计 rollout 的平均目标；
- `prior(edge)`：该边的 prior；
- `total_visits`：root 总访问数；
- `visits(edge)`：该边访问数；
- `exploration_c`：探索强度系数；
- `objective_score(mean_objective(edge))`：把平均目标向量转为标量 exploitation score；
- `log`：自然对数。

为什么这一步还需要 prior：  
搜索预算有限时，prior 用来引导“先把看起来有希望的边展开”，但最终仍由真实 rollout 目标决定。

### 6.6 final pick 和 no_op gating

最终选择按：

```text
selected_action =
    argmax over action:
        (
            -qtime_violation_count,
            -qtime_violation_total,
            -priority_weighted_wait,
            avg_utilization,
            visits
        )
```

符号说明：

- `selected_action`：VC-MCTS 最终选择的 root action；
- `argmax over action`：选择使五元组最大的动作；
- `-qtime_violation_count`、`-qtime_violation_total`、`-priority_weighted_wait`：违约和等待越小越好，取负后越大越好；
- `avg_utilization`：utilization，越大越好；
- `visits`：该边访问次数，只作为最后平局项。

但是 `no_op` 不是单纯按 visits 胜出；它必须在 Q-time count 或 Q-time total 上确实优于非 `no_op` 分支，才允许保留。

为什么要做 no_op gating：  
否则小预算搜索里 `no_op` 容易因为统计噪声被过度放大，导致“什么也不做”成为伪最优。

### 6.7 arrival probability weighting

若启用 arrival probability weighting，则对 reserve 边做期望折扣。设未来 lot 到达概率为：

```text
arrival_prob(eta,t) =
    exp(-arrival_prob_decay * max(0, eta - t) / lookahead_window)
```

符号说明：

- `arrival_prob(eta,t)`：未来 lot 到达并使 reserve 分支兑现的折扣概率；
- `eta`：future lot 的 ETA；
- `t`：当前决策时刻；
- `arrival_prob_decay`：到达概率衰减强度；
- `lookahead_window`：前瞻窗口宽度；
- `max(0, eta - t)`：ETA 距离，若 ETA 已不晚于当前时刻则取 0。

则 reserve 的有效目标为：

```text
effective_reserve_objective =
    arrival_prob * reserve_arrive_objective
    + (1 - arrival_prob) * reserve_miss_objective
```

符号说明：

- `effective_reserve_objective`：考虑到达概率后的 reserve 有效目标；
- `arrival_prob`：上一式得到的到达概率；
- `reserve_arrive_objective`：future lot 按预期到达并兑现 reserve 时的目标；
- `reserve_miss_objective`：future lot 未按预期兑现时，退回到最佳非 reserve 分支的目标；
- `(1 - arrival_prob)`：future lot 未兑现 reserve 收益的概率权重。

为什么要有这个项：  
远期 lot 的预留收益不应该和近端 lot 一样计分，否则会高估很远未来的 hold。

### 6.8 leaf value bootstrap

启用 `use_leaf_value=True` 时，叶子评估不必每次 rollout 到终点，而可由 multi-head critic 提供近似。其思想是：

- `qtime` 头估计剩余违约趋势，并映射回 `qtime_violation_count`；
- `util` 头估计终局利用率；
- `qtime_violation_total` 与 `priority_weighted_wait` 仍由 partial rollout 保持；
- `critic_to_objective_dims` 最终输出四个目标字段：`qtime_violation_count`、`qtime_violation_total`、`priority_weighted_wait`、`avg_utilization`。

为什么这是可选增强而不是默认路径：  
它能加速搜索，但毕竟是近似；默认仍以真实 rollout 为准更稳。

## 7. 为什么当前架构成立

### 7.1 下层为什么必须固定规则

因为下层只做“内部批处理排程 + 时间估计”，没有可学习的取舍。它的任务是把 `estimate` 和 `schedule_on_calendar` 统一起来，而不是再引入一个策略网络。

### 7.2 Q-time 为什么必须先于 priority

Q-time 是硬约束。priority 再高，也不能通过牺牲 Q-time 去交换。这个顺序是字典序，而不是加权和。

### 7.3 为什么 priority 用 filter 而不是 reward

priority 是强偏好，适合在候选池内做结构性收缩，而不是让 reward 稀释掉它的作用。

### 7.4 为什么 utilization 只能做软目标

因为当前问题中真正会发生冲突的，主要是“Q-time 安全”与“优先级等待”。utilization 只是在前两者满足后，才允许作为最后的折中项。

### 7.5 为什么预留用搜索而不是学习

因为预留的收益依赖未来 lot 是否到达、到达后是否真的需要这台机、以及当前派工对未来资源的挤占效应。这是典型的反事实比较问题，搜索比学习更自然。

## 9. 总结

如果把当前代码压缩成一句话，它就是：

```text
lower layer = accurate scheduling and timing
SAS         = current dispatch selection
VC-MCTS     = counterfactual reservation planning
```

符号说明：

- “下层”指 `lower_layer_estimator.py` 与 `lower_layer_scheduler.py`；
- “SAS”指 `phase2_*` PPO 派工策略；
- “VC-MCTS”指 `vc_mcts_planner.py` 中的在线预留搜索；
- “反事实比较”指在 clone 出来的 driver/ledger 上分别模拟 reserve、dispatch、no_op 后果。

这也是为什么这份报告不该写成 README。它需要说明每个决策点为什么这么定义、数学上怎么比较、以及为什么当前代码选择这条路线，而不是别的路线。

当前版本的核心价值，不在于再造一个“会 hold 的网络”，而在于把 hold 变成可验证、可回放、可比较的搜索分支。
