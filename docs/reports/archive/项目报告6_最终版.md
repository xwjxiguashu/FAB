## 0. 当前结论（2026-06-09）

当前项目已形成可运行的两层调度原型：

1. **下层** (`lower_layer_estimator.py` / `lower_layer_scheduler.py`)：给定 `(lot, machine, ppid)` 后，用共享核心 `schedule_deterministic` 完成子批组批、stage 资源选择和资源日历插入；是所有完成时间估计、dry-run、commit 的单一事实来源。

4. **实验闭环** (`vc_mcts_probe.py` / `evaluate_baselines.py`)：`late_hi` 实例 deterministic 轻预算验证中，VC+SAS delegate 将 Q-time 违规从 20→0、O2 从 1664→956（改善 −42.6%）。noise smoke (2 seeds) 通过——Q-time 清零、50/50 完成。

## 1. 统一符号

记：

- `l` 为 lot；`m` 为 machine；`p` 为 PPID；`t` 为当前时刻
- `deadline(l)` 为 lot `l` 的 Q-time deadline（阶段间队列时间约束）
- `arrival_time(l)` 为 lot 到达时刻；`priority(l)` 为优先级
- `mu_finish(l,m,p)`, `sigma_finish(l,m,p)` 为 `(l,m,p)` 的完成时间均值和标准差（由下层估时器输出）
- `epsilon_qtime` 为 Q-time 机会约束容许违规率；`z_eps` 为对应分位数（如 ε=2% → `z_eps≈2.05`）
- `candidate_set(t,m)` 为时刻 `t`、机台 `m` 的结构可行动作集合
- `policy_theta(a | obs_t)` 为 SAS policy 在观察 `obs_t` 下选择动作 `a` 的概率
- `value_c(state)` 为多头 critic 在通道 `c` 上的价值估计

调度状态：

```text
s_t = { 当前时刻, 机器/腔体日历, 已完成 lots, 在线可见 lots,
        前瞻窗内即将到达 lots (ETA/priority/qtime已知),
        预留 ledger (machine→future_lot 映射), … }
```

## 2. 总体算法流程

当前 episode 的主流程：

```text
while not done:
  1. 选出当前可派工机台 m* (最早空闲 → 最少候选(最受约束优先) → 最低ID)
  2. 构造 candidate_set(t, m*) → qtime_mask → priority_filter → TopK → pad
  3. 若 ROP 检测到预留机会点：
       VC-MCTS 建树搜索，比较 reserve / dispatch(or delegate_dispatch) / no_op
       选中 reserve → 登记 ledger, machine m* 不派工
       选中 dispatch → 进入步骤 4
       选中 no_op   → 推进时间
  4. SAS policy (或 rule delegate) 在候选池中选 (lot, ppid)
  5. dry-run 试算 → commit 到真实日历
  6. 记录 reward vector 和 next state
```

其中第 3 步 ROP 是宽触发式算力闸门——只在存在"可能有预留价值"的场景时建树，绝大多数决策点直接走第 4 步的贪心派工。

## 3. 下层：完成时间估计与资源日历排程

下层是固定规则调度器，负责给定 `(lot, machine, ppid)` 后的批处理、stage 资源选择和资源日历插入。**只做排程不做决策，无学习、无随机搜索。**

### 3.1 子批组批

一个 lot 的 wafer 数为 `N`，若 side capacity 为 `c`，则子批数为：

```text
B = ceil(N / c)
```

若 `c` 未设置，则退化为 `B = N`（每个 wafer 一个子批）。

满批优先策略：前面的子批装满 `c` 片，最后一个子批装余数。批处理时间与装载片数无关（批处理炉），非满批子批仍占整批的批处理时间。

### 3.2 共享确定性核心：`schedule_deterministic`

下层只有一个纯函数核心，estimate 和 schedule_on_calendar 共享——唯一区别是 `instance_free_init` 的来源（空字典 vs 真实日历）：

```text
For each sub-batch b (FIFO order):
  ready = lot_release_time              # 首阶段
  For each stage s (PPID process order):
    For each resource option (chamber, side) in stage s:
      key = (machine, chamber, side)
      cand_start = max(ready, free[key])
    selected = argmin cand_start        # 最早能开工的实例
    start = max(ready, free[selected])
    end   = start + process_time(b, s)
    free[selected] ← end
    ready ← end                         # 流水：下阶段 ready = 本阶段结束

lot_start = min over b,s start(b,s)
lot_end   = max over b,s end(b,s)
```

核心设计原则：

1. 实例选择用 `argmin max(ready, free)`（最早能开工）而非 `argmin free`（最早空闲）——多实例忙闲不均时两者结果不同，后者会把子批挤到同一实例。
2. 分配后**必须更新** `free[selected]`，否则所有子批被算到同一最早空闲实例上。
3. `estimate()` 以空 free 调用（状态无关，可缓存）；`schedule_on_calendar()` 以真实日历 free 调用（状态相关，不可缓存但非破坏性）。

### 3.3 蒙特卡洛完成时间分布

因多子批并行产生 `E[max] > max(E[·])` 偏置，仅用各阶段均值算 makespan 会系统性低估。用蒙特卡洛修正：

```text
For k = 1, …, K (default K = 50):
  For each (sub_batch b, stage s):
    p_actual(b,s) = μ(b,s) + ε,   ε ~ N(0, σ²), 各 (b,s) 独立
  Run schedule_deterministic with p_actual → makespan_sample(k)

mu_finish(l,m,p)    = (1/K) · Σ_{k=1}^{K} makespan_sample(k)
sigma_finish(l,m,p) = sqrt( (1/K) · Σ_{k=1}^{K} (makespan_sample(k) − mu_finish)² )
```

噪声模型（报告 §2.4.1）：纯工艺随机抖动——无偏、独立、估计时不可知、做完才揭晓。μ 和 σ 均作为工艺参数进入 agent 状态特征。

### 3.4 Q-time 机会约束

加工时间随机 → Q-time 不能写成确定性硬约束，改写为机会约束。因噪声无偏独立，`finish_time` 近似正态（CLT），违规概率有闭式：

```text
P(violation) = P(finish_time > deadline)
             = Φ( (mu_finish − deadline) / sigma_finish )

动作可行 ⇔ P(violation) ≤ ε
        ⇔ deadline(l) − mu_finish(l,m,p) ≥ z_eps · sigma_finish(l,m,p)
```

其中 `z_eps` 为容忍违规概率 ε 对应的分位数（如 ε=2% → `z_eps≈2.05`）。safety margin 不再是常数，而是自适应的 `z_eps·σ_finish(l)`——波动大的工件自动留厚垫、波动小的留薄垫。

> 诚实边界：正态噪声尾部无限长，违规概率恒 > 0。保证的是 **P(违规) ≤ ε**，而非"违规归零"。可控的是 ε（通过 `z_eps`），不是绝对的 0。

### 3.5 链式 Q-time (当前实现)

当前代码支持三种 Q-time mask 模式（`qtime_mask_mode`）：

- **aggregate**：单一聚合 deadline proxy，比较总完成时间 µ 与 deadline
- **chain**：每次 dry-run 后评估真实 per-stage `q_time_limits` 链（如 `(1,2)` / `(2,3)`）
- **chain_joint**：K 次带噪声 dry-run，估计联合机会约束 `P(any chain window violates)`

链式模式下，一次 dry-run 中任一链窗违约即标记为违规样本：

```text
chain_violation_prob(action) = (1/K) · Σ_{k=1}^{K} 1_{sample k has chain-window violation}

若 chain_violation_prob(action) > chain_threshold → 屏蔽该动作
```

已知发现：aggregate proxy 在特定实例上会漏掉链窗口违约（FIFO Q-time 违规 ~13.6），chain-aware mask 可将其降至 ~0.4。

## 4. 上层 SAS：候选池、动作选择与多头 PPO

SAS (Schedule-Action Selector) 负责在当前空闲机台上选择一个 `(lot, ppid)` 派工。SAS 自身不拥有 wait/reserve 决策——这些由 VC-MCTS 或上层时机层处理。

### 4.1 候选池构造流水线

对机台 `m`，候选池按**固定不可调换的顺序**生成：

```text
第0步：结构可行动作集
  candidate_set(t,m) = { (l,p) | lot l 已到达且未完成,
                                machine m 可加工 l,
                                p 是 (l,m) 可用 PPID,
                                PPID 步骤完整,
                                所需 Chamber/Side 存在 }

  ↓ ① qtime-safe mask（硬约束，删除会断送可见 lot 的动作）
  ↓ ② priority filter（强偏好，在 qtime-safe 集合内优先高优先级）
  ↓ ③ CandidateScore 排序（软目标启发式打分，仅排序不删除）
  ↓ ④ TopK + padding + valid_mask = candidate_pool（固定长度 K_action）
```

其中第三步启发式分数为：

```text
score(action) = due_urgency(action) + waiting_time(action)
              + 0.1 · qtime_slack(action)
              − 0.001 · total_process_time(action)
              − 0.001 · qtime_risk(action)

qtime_slack(action) = max(0, deadline(l) − predicted_completion(action))
```

此分数只做排序——删除已由 ①② 完成，valid_mask 只标记 padding 位（+防御性的结构不可行位）。喂给 SAS 的候选池**已是 qtime-safe 且 priority 过滤后**的。

### 4.2 priority filter

soft 模式下不删动作，仅排序。strict 模式保留：

```text
priority(action) ≥ max_priority_in_pool − priority_gap
```

Q-time 是硬约束，priority 是强偏好，utilization 是软目标——把它们混成一个分数会让"高优先级但高风险"和"安全但普通"混在一起。用 filter 实现字典序比 reward shaping 更稳定。

### 4.3 SAS 动作选择

policy 输出候选池上的 masked softmax：

```text
policy_prob(i | obs_t) = exp(logit_i) · valid_mask_i
                       / Σ_j [ exp(logit_j) · valid_mask_j ]
```

训练时采样，推理/greedy evaluation 取 argmax。多机同时在线的处理顺序为字典序（最早空闲 → 最少候选即最受约束优先 → 最低ID）：

```text
selected_machine = argmin over m in dispatchable_machines:
    (machine_available_time(m), valid_real_action_count(m), machine_id(m))
```

### 4.4 多头 Critic + 三通道奖励

奖励不再混合加权，改为三通道向量：

```text
reward_vector(t) = ( r_exec(t),  r_qtime(t),  r_util(t) )
```

各通道定义：

```text
r_exec(t) = +0.20 + w_pack · total_work/span    若插入成功
          = −0.40                                若插入失败
          = −0.50                                若 mask_invalid（防御性，正常不触发）
          =  0.00                                若 wait/noop (池空或全屏蔽)

r_qtime(t) = − new_qtime_violation(t) / num_lots   # 逐步密集，Σ_t = 终局总违反 (telescoping)

r_util(t)  = avg_machine_utilization               # 仅终局 (is_terminal=True)，唯一软目标
```

`r_exec` 中的 `packing = total_work / (lot_end − lot_start)` 是利用率向的逐步边际质量信号——腔体争用会拉长跨度 → packing 变小，agent 每步就能感知选得紧不紧凑。

### 4.5 逐通道 GAE

三个通道量纲、稀疏程度、终局属性完全不同，单头 critic 会把梯度混在一起。当前做法是对每个通道 `c ∈ {exec, qtime, util}` 独立计算 GAE：

```text
td_residual(t,c) = r(t,c) + γ · V_c(s_{t+1}) − V_c(s_t)

advantage(t,c) = Σ_{τ≥0} (γ·λ_gae)^τ · td_residual(t+τ, c)
```

然后逐通道归一化并加权：

```text
combined_advantage(t) = w_exec · norm(Â_exec)
                      + w_util · norm(Â_util)
                      + w_qtime · norm(Â_qtime)       # r_qtime 本身为负 (cost)，统一加法
```

若启用 PPO-Lagrangian（`use_qtime_lagrangian=True`），`w_qtime` 由自适应对偶变量 `λ_qtime` 取代：

```text
λ_qtime ← clip( λ_qtime + η_λ · (Ê[违规] − ε),  0,  λ_max )
```

### 4.6 PPO 目标

策略比率和 clipped PPO objective：

```text
ρ_t = policy_θ(a_t | s_t) / policy_old(a_t | s_t)

L_policy = − E_t[ min( ρ_t · A_t,  clip(ρ_t, 1−ε_clip, 1+ε_clip) · A_t ) ]

L_value  = Σ_{c} w_v · ( V_c(s_t) − R_t^c )²          # R_t^c = Â_t^c + V_c(s_t)

L_total  = L_policy + L_value − w_e · H(π)
```

## 5. Q-time 约束：两道防线

### 5.1 防线一：机会约束 mask（硬挡）

在决策时结构性屏蔽"会当场断送某可见 lot 的 Q-time"的动作（§3.4）：

```text
function qtime_safe_mask(candidate_actions, visible_lots, C_t, config):
    for i, (lot, ppid) in enumerate(candidate_actions):
        C_sim = dry_run(C_t, lot, machine, ppid)
        for l in visible_lots:
            if is_doomed(l): continue              # 注定违规者不作屏蔽依据（防死锁）
            mu, sigma = estimated_next_finish(l, C_sim)
            if deadline(l) − mu < z_eps · sigma:   # P(违规) > ε
                mask[i] = 0; break
```

### 5.2 防线二：PPO-Lagrangian（软压）

mask 挡不住窗外到达 + 随机尾部 + doomed lot 的残差违规。在训练目标层面用 CMDP 表述：

```text
max_π E[utilization]   s.t.   E[Q-time violation] ≤ ε

⇔ max_π min_{λ≥0}  E[util] − λ · (E[violation] − ε)       （拉格朗日对偶）

λ ← max(0, λ + η_λ · (Ê[violation] − ε))                 （对偶上升）
```

你只需设 ε（可容忍违规率，有物理含义），λ 自动停在约束边界。这修复了固定 `w_qtime` 的两难：太小压不住违规、太大牺牲利用率、最优值随训练进程变化。

> 诚实边界：收敛保证在深度非凸情形下打折（深度 RL 通病，非本方法独有）；保证的是期望约束 `E[violation] ≤ ε`，与"随机下无法绝对归零"自洽。

## 6. VC-MCTS：可见性感知的约束式 MCTS 在线预留

### 6.1 为什么用搜索取代学习式 RMA

预留动作本质是反事实问题——"现在派 vs 留给未来"谁更好，只有把整条资源日历向前模拟之后才知道。学习式 RMA 有三个结构性顽疾：

1. **奖励坍缩**：预留收益在终局才体现，稀疏 reward 难以 credit
2. **greedy under-hold**：贪心推理在"高优先级到达前那一下" `P(reserve)` 顶不过 0.5
3. **类别不平衡**：ROP 宽触发制造大量 `no_reservation` 正确样本，策略被推向"从不预留"

VC-MCTS 换一个范式：不学概率，而是把 `reserve(m,h)` 和 `dispatch` 都作为搜索边，用基策略 rollout 到底、按字典序目标比较，谁好选谁。没有梯度、没有 threshold、没有 advantage 归一化——三个失败模式在结构上不存在。

> 定位：VC-MCTS 是 **DyRo-MCTS (Chen et al., 2025)** 的原理性推广。DyRo 因完全看不见未来到达，只能用钝的机器空闲分布代理（结论是"避免早期空闲"）。本项目在有限前瞻窗内未来到达可见（ETA/priority/qtime 已知），于是"定向空闲（预留）"从浪费变为最优：窗内用显式仿真 + reserve 边取代钝代理，窗外退为 DyRo 式代理。当前瞻窗宽 `W=0`、无 Q-time 约束、加工确定时，本方案退化为 DyRo-MCTS。

### 6.2 ROP：预留机会点（算力闸门）

ROP 是宽触发式门控——只回答"是否存在候选预留对"，不替搜索决定"值不值得预留"：

```text
exists (machine m, future_lot h) such that:
  1. m 空闲 且 未被其他 reservation 冻结
  2. h 在前瞻窗 [t_now, t_now+W_lookahead] 内，尚未到达但已可见
  3. compatible(m,h) = True
  4. not is_doomed(h)
  5. reserve(m,h) 不会使其他已可见 lot 直接违反 Q-time 机会约束
```

若无 ROP，不建树，直接走 SAS 贪心派工（近乎免费）。若有 ROP，候选预留对按软机会分数 `S_res(m,h)` 排序截断 TopB 后进入搜索。

### 6.3 根动作空间

```text
root_actions = { no_op }
             ∪ { dispatch(machine m) }          # TopK 候选 (use_delegate_dispatch=False)
             ∪ { delegate_dispatch(machine m) } # rule/sas delegate (use_delegate_dispatch=True)
             ∪ { reserve(machine m, future_lot h) | (m,h) ∈ P_t }   # TopB 预留候选
```

delegate_dispatch 模式下，root 只有一个 dispatch 边，具体 `(lot, ppid)` 由 RuleDispatchDelegate 或 SASPolicyDispatchDelegate 给出。

### 6.4 目标向量与字典序比较

每个 rollout 末端计算目标向量：

```text
objective(action) = ( qtime_violation_count(action),
                       qtime_violation_total(action),
                       priority_weighted_wait(action),    # O2
                      −avg_utilization(action) )
```

其中 O2（优先级加权等待）为：

```text
O2 = Σ_l priority(l) · max(0, start_time(l) − arrival_time(l))
```

比较规则是字典序：先比 Q-time count → 再比 Q-time total → 再比 O2 → 最后比 utilization。Q-time 是硬约束门槛，O2 是优先级维度的优化主目标，utilization 是唯一可让步软目标。

### 6.5 UCT 选择

每条边的选择分数：

```text
uct_score(edge) = objective_score( mean_objective(edge) )
                + exploration_c · prior(edge) · √( log(total_visits) / visits(edge) )

objective_score = − qtime_penalty · qtime_violation_count
                  − qtime_total_penalty · qtime_violation_total
                  − priority_weighted_wait
                  + util_weight · avg_utilization
```

warm-up 保证小预算下每条 root edge 至少被 rollout 一次：`iteration_count = max(len(root_edges), n_iter)`。

### 6.6 Final Pick 与 no_op gating

最终选择按字典序：

```text
selected_action = argmax over action:
    ( −qtime_violation_count, −qtime_violation_total,
      −priority_weighted_wait, avg_utilization, visits )
```

但 `no_op` 有门控：若 `no_op` 胜出但存在非 no_op 替代边，`no_op` 必须在 `qtime_violation_count` 或 `qtime_violation_total` 上**严格优于**最佳非 no_op 边，否则降级为最佳 dispatch/reserve 边。

原因：早期完整轻预算中，无门控时 120 次决策里 no_op=89、episode 只完成 31/50。加入门控后 no_op→1、完成 50/50。

### 6.7 到达概率加权（可选）

若启用 `arrival_prob_weighting`，远期 lot 的预留收益不能和近端 lot 一样计分：

```text
arrival_prob(eta, t) = exp( −arrival_prob_decay · max(0, eta−t) / lookahead_window )

effective_reserve_objective = arrival_prob · reserve_arrive_objective
                            + (1−arrival_prob) · reserve_miss_objective
```

### 6.8 叶子 Critic 截断（可选 AlphaZero 增强）

启用 `use_leaf_value=True` 时，用多头 critic 提供截断估值而不必每次都 rollout 到终点：

```text
剩余 Q-time 违规估计:  remaining_violations ≈ max(0, −V_qtime · num_lots)
终局利用率估计:       V_util ∈ [0, 1]
O2 / qtime_total:    由 partial rollout (leaf_rollout_depth 步) 保持
```

这仍是可选增强——默认走完整 rollout 以保证准确性。

### 6.9 Rollout 基策略的三个要求（命门）

```text
rollout 内部基策略（FIFO/SPT/… 或训练好的 SAS）必须:
  ① 也走 qtime-safe 候选池（机制1硬挡在仿真里照样生效，否则 q̂ 乐观偏高）
  ② 尊重 reservation ledger（已 reserve 的机器不被基策略抢派，等 h 到优先上——
     否则 reserve 分支与 dispatch 分支的 q̂ 无差异，预留收益被仿真抹平）
  ③ 注入加工噪声 + 复用 CRN 种子（机制3，见下）
```

### 6.10 预留登记与兑现 (ledger)

```text
ledger 字段: reservation_id, machine_id, target_lot_id, start_time,
             target_eta, expire_time, status∈{pending,hit,miss,waste,released},
             idle_time, saved_wait_est, actual_saved_wait, blocked_lots
```

评估同上一版口径：`reservation_hit_rate`、`reservation_waste_time`、`saved_priority_wait`、O2 是否优于 SAS-only。若只降低旧 priority_violation 却拉高 O2 或 idle，视为失败。

## 8. 未完成边界与下一步

### 8.1 高优先级

1. **扩大 seeds**：当前 noise smoke 只有 2 seeds——需要更多 seeds 判断 SAS delegate 在噪声下是否稳定显著优于 rule delegate
2. **`late_hi` SAS checkpoint 噪声重训**：当前 checkpoint (`pressure_mh_hard.pt`) 可能在确定性数据上训练，噪声场景下性能不够稳定
3. **机制 2（优先级-能力鲁棒性 `ρ_pc`）**：当前 VC-MCTS 利用项只有 O2 价值项，DyRo 的 `α·q + (1−α)·ρ` 插值未完整落地
4. **机制 3（CRN 噪声 rollout）**：当前 rollout 无加工噪声注入——同一节点下各候选边应复用同一组噪声种子做方差缩减

### 8.2 中优先级

5. **多实例族验证**：不同到达强度、不同 qtime 紧度、不同 priority-到达相关性——确认 VC-MCTS 预留收益的边界条件
6. **叶子 Critic 截断的实验验证**：当前默认跑到底——若截断估值准确，可大幅降低单次决策时延
7. **HGNN 状态表示**：异型柔性下 `(lot, machine)` 多对多匹配关系复杂，手工候选池特征可能不够——HGNN 作为后期增强

### 8.3 低优先级

8. **公开基准 SMT2020**：即使项目最终用自有实例，也应至少在一个公开测试床（SMT2020 子集）上与派工规则做横向对比
9. **专家迭代**：拿 VC-MCTS 改进后的决策当标签回训 SAS prior（方法非原创，仅提速在线搜索）

## 9. 总结

本项目当前的实质是三层结构：

```text
下层 (fixed rules)     = 精确排程与估时（所有时间判断的单一事实来源）
SAS (learned PPO)     = 当前派工选择（qtime-safe + priority-filtered 候选池）
VC-MCTS (search)      = 反事实预留规划（reserve/dispatch/no_op 模拟比较）
```

核心价值不在于再造一个"会 hold 的网络"，而在于把预留变成可验证、可回放、可比较的搜索分支——每个数都是仿真出来的、可复算的。

从"已跑通"到"结论可信"，剩下的是工程上的更多 seeds、更多实例族、以及噪声下 checkpoint 的鲁棒性验证。
