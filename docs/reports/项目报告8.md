调度状态记为

```text
s_t = { t_now, 机器/腔体日历, 已完成 lots, 在线可见 lots, F_W, 预留 ledger, … }
```

其中 `F_W` 为前瞻窗 [t_now, t_now + W] 内**尚未到达但已可见**的 future lots（ETA / priority / qtime 已知）。网络只接收可计算摘要 z_t = featurize(s_t)，完整日历保留在环境中。

## 2. 总体算法框架与流程

> **以资源日历为环境，以候选池 + 约束 mask 控制动作空间，以多头 PPO 训练可学习的 SAS 派工策略，并在其上叠加 VC-MCTS 预留规划器（搜索式、非学习）处理"为未来高优先级 Lot 预留资源"的时机决策。**

### 2.0 主线统稿：一条主线、一个矩阵、一张合同表

> 本节是组织框架。核心原则：**每个组件在主线上有且只有一个位置、回答有且只有一个问题**；任何无法在矩阵和合同表中找到唯一格子的组件，要么降级为分支（分级见 §9.4.1），要么砍掉。

#### 2.0.1 一句话主线

```text
双重不确定性的字典序消解：
把加工随机性与到达随机性，沿「量化 → 约束化 → 学习 → 对冲 → 兜底」五级流水线
逐级转化为可计算的决策对象；每一级恰好执行字典序三层结构中的一层职责。
```

论文版表述：下层估时器以固定规则把**加工随机性量化**为完成时间分布 (μ_fin, σ_fin)（①量化）；机会约束 mask 把分布**约束化**为 qtime-safe 可行集，执行第 0 层硬约束（②约束化，机制 1）；SAS-PPO 在 priority 过滤后的可行集内**学习**当前派工的快策略，执行第 1 层的"当下"半边（③学习）；VC-MCTS 在可见性窗口内以 reserve 边**对冲**未来高优先级到达，执行第 1 层的"未来"半边——ρ_pc 给对冲定水位（机制 2，§7.12）、CRN+CVaR 给对冲定价与验收（机制 3，§7.13）（④对冲）；利用率作为唯一软目标吸收剩余自由度，λ_qt 对窗外黑盒残差统计性**兜底**（⑤兜底）。

> **快/慢双系统修辞**：SAS 是在每个决策点近乎免费的 System-1 快策略；VC-MCTS 是仅在 ROP 检测到对冲杠杆时才启动的 System-2 慢思考；ROP 是两个系统之间的算力闸门。

#### 2.0.3 组织核心：不确定性 × 处理阶段矩阵

机制 1/2/3 **不是三个并列模块，而是同一张矩阵中的三个格子**——这是把"散"变"整"的关键一句话：

```text
                    │ 准入（动作空间）        │ 评估（价值估计）        │ 兜底（残差）
────────────────────┼────────────────────────┼───────────────────────┼──────────────────
 加工噪声            │ 机会约束 mask           │ CRN 配对 + CVaR        │ —（无偏噪声
                    │ z_ε·σ 余量【机制 1】     │ 风险字典序【机制 3】     │  无系统性残差）
────────────────────┼────────────────────────┼───────────────────────┼──────────────────
 到达 · 窗内可见      │ ROP 硬过滤 + reserve 边 │ ρ_pc 对冲水位【机制 2】 │ 到达概率加权
                    │ 进入根动作空间 §4.3/7.3  │ + 配对 rollout 反事实   │ （支线, §7.7）
────────────────────┼────────────────────────┼───────────────────────┼──────────────────
 到达 · 窗外黑盒      │ —（信息上不可准入）      │ —（信息上不可估值）     │ λ_qt PPO-
                    │                        │                       │ Lagrangian §6.5
```

矩阵的两条检验规则：(i) **无空格遗漏**——每类不确定性在其信息上可处理的阶段都有恰好一个机制；(ii) **无重复覆盖**——没有两个机制处理同一格。此前学习式 RMA 被废除的深层原因可在此框架下重述为：学习式预留与 λ 兜底**重复覆盖了"窗内到达"格**。论文 Problem 节放这张矩阵，Method 各节开头声明"本节填充哪个格子"。

### 2.1 总体流程

一个 episode 的主循环：

```text
while not done:
  1. 选出当前可派工机台 m*      # 字典序: 最早空闲 → 最受约束(候选最少) → 最低 ID (§2.3)
  2. 构造 A(t, m*) → ① qtime_mask → ② priority_filter → ③ score 排序 → ④ TopK + pad
  3. 若 ROP 检测到预留机会点 (§4.3):
       VC-MCTS 建树，比较 reserve / dispatch(or delegate_dispatch) / no_op (§7)
       选中 reserve → 登记 ledger, m* 暂不派工
       选中 dispatch → 进入步骤 4
       选中 no_op   → 推进时间
  4. SAS policy (或 rule/sas delegate) 在固定长度候选池中采样/贪心选 (lot, ppid)
  5. dry-run 试算 (非破坏) → commit 到真实日历
  6. 记录 reward vector 和 next state
  7. (训练) 逐目标 GAE + 多头 PPO 更新
```

第 3 步的 ROP 是**宽触发式算力闸门**——只在"可能有预留价值"时建树；绝大多数决策点直接走第 4 步的贪心派工，近乎免费。

### 2.2 MDP 形式化定义

环境状态在通用 FJSP 基础上**补充两类信息**（加粗为本项目新增）：

```text
s_t = {
  当前时刻 t_now,
  未完成 Lot 集合: arrival / due_date / priority / wafer_count,
  **每个已可见 Lot 的 Q-time 窗口截止时刻 deadline(l)**,
  **前瞻窗 F_W 内即将到达的 Lot 及其 priority / qtime / ETA**,
  空闲 Machine 集合 M_idle,
  **每台 Machine 的能力: 可加工 recipe / 可用 ppid (异型柔性的关键信息)**,
  机器 Lot-level calendar, (machine,chamber,side) wafer-level calendar,
  已提交操作, 预留 ledger
}
```

策略与价值定义在特征摘要上：actor 给出候选池上的分布 `π_θ(· | z_t)`，多头 critic 给出 {V_c(z_t)}_c。奖励为**向量** r_t = (r_exec_t, r_qtime_t, r_util_t)（§6.2）。**机器分配本身不由策略学习**——VC-MCTS 决定"是否/为谁预留"、SAS 学"派哪个 `(l,p)`"，但"用哪台机器"由 §2.3 的固定规则按顺序给定；机器分配经"逐台顺序决策"**隐式涌现**。这是相对"同时学作业 + 机器分配"的主流 FJSP 工作的一处范围限制（诚实声明，见 §9）。

### 2.3 局部动作空间与机台选择

单台 Machine `m` 空闲时，动作为 a^m_t = (l, p)，`m` 由状态给定。多台同时空闲时**逐台决策**（避免组合动作空间），处理顺序为字典序：

```text
m* = argmin_(m ∈ M_idle)  ( avail(m),  N_cand(m),  id(m) )
```

**符号说明**：`avail(m)` 为 `m` 最早可用时刻（最早空闲优先）；`N_cand(m)` 为 `m` 的有效真实候选数（越少越先，即**最受约束者优先**）；`id(m)` 为机台编号平局项。三元组按字典序比较。

为什么这样选：先处理最早空闲、再处理选择余地小（异型下能力最受限）的机器，避免其候选被余地大的机器抢走而陷入空闲/违规；处理顺序由固定规则给定、不进搜索分支，避免徒增分支因子。同型退化为"最早空闲优先"。

### 2.4 三层字典序约束结构

本项目的目标不是"几个目标加权"，而是**严格分先后**的三层结构：

```text
第 0 层（硬）: 阶段间 Q-time 不可违(材料队列时间) —— 机会约束 mask(窗内) + cost/λ 兜底(窗外)
第 1 层（强）: 高优先级 Lot 优先              —— 候选池过滤 + VC-MCTS 预留, 但不得踩穿第 0 层
第 2 层（软）: 机台利用率                     —— 奖励通道, 唯一可让步目标
```

执行顺序「先 mask → 再过滤 → 再打分」天然实现字典序。形式化地，设 ≼_lex 为字典序，则两个调度结果的偏好为

```text
S₁ ≼_lex S₂  ⇔  (V_qt₁, O2₁, −U₁)  ≤_lex  (V_qt₂, O2₂, −U₂)
```

**符号说明**：`V_qt` 为 Q-time 违规量，`O2` 为优先级加权等待（§6.2、§10.2），`U` 为利用率（取负以统一为越小越好）。**关键前提（以代码为准）**：阶段间 Q-time（`q_time_limits`）是硬约束、目标严格趋零，归第 0 层；交付期拖期（due date）是**独立的量**，仅作评估指标、不进奖励。由此真正"可互相权衡的软目标"只有利用率一项——本项目本质是「**单软目标 + 两层约束**」，而非多目标帕累托问题，下文奖励与 critic 的三通道设计据此调整（§6），不作"多目标先进性"包装。

## 3. 下层：完成时间估计与资源日历排程

> **主线位置**：流水线第①级（量化）——把加工随机性转化为完成时间分布 (μ_fin, σ_fin)，是矩阵全部"准入/评估"格的输入；自身不执行任何字典序层、不做任何决策（合同表 §2.0.4）。

下层是**固定规则**调度器：给定 `(l,m,p)` 后完成子批组批、stage 资源选择、资源日历插入，是所有完成时间估计 / dry-run / commit 的**单一事实来源**。只做排程不做决策，无学习、无随机搜索。

### 3.1 两层架构与窄接口

```text
上层 (SAS 用 RL 派工 + VC-MCTS 搜索预留): 决定 哪个工件 / 哪台 machine / 哪个 ppid / 什么时机
  —— 存在真实取舍(利用率 vs 优先级 vs 不违 Q-time), 故用 RL/搜索
下层 (固定规则估时器): 一个工件被派到 (m, p) 后, 其一批 wafer 如何组批 / 进哪个 chamber 实例 / 各批时序
  —— 无取舍(wafer 对称、step 内 chamber 间无 Q-time、目标单一=尽早完成), 故用规则算
```

接口单向、窄：上层只向下层传 `(l,m,p)` 与各阶段 (μ_s, σ_s)；下层只返回完成时间分布等少数输出。术语澄清：下层的**排程逻辑**是确定性规则，但它作用在**随机的加工时间**上（§4.4），故**输出是分布** (μ_fin, σ_fin) 而非单值——"固定规则"与"分布输出"不矛盾。

### 3.2 子批组批

一个 lot 的 wafer 数为 `N`，side capacity 为 `c`，则子批数

```text
B = ⌈ N / c ⌉,      若 c 未设置则退化为 B = N.
```

**符号说明**：`B` 为子批数；⌈·⌉ 向上取整，最后不足一批也占一个子批。满批优先：前面的子批装满 `c` 片，最后一个装余数；批处理时间与装载片数无关（批处理炉），非满批子批仍占整批的批处理时间。每个子批每阶段占**一个** (chamber, side) 区间，其 wafer 同进同出。

### 3.3 共享确定性核心 `schedule_deterministic`

下层只有一个纯函数核心，`estimate()` 与 `schedule_on_calendar()` 共享——**唯一区别是实例空闲初值的来源**（空字典 vs 真实日历）。对每个子批 `b`（FIFO 顺序）、每个 stage `s`（PPID 工艺顺序）：

```text
start(b,s) = max( ready(b,s),  free(inst*) ),

inst* = argmin_(inst ∈ R(s))  max( ready(b,s),  free(inst) )
```

```text
end(b,s) = start(b,s) + p(b,s),       free(inst*) ← end(b,s)
```

完成时间

```text
completion(l) = max_b  end(b, s_last)
```

**符号说明**：`ready(b,s)` 为子批在前一阶段完成后的可开工时间（首阶段为 release）；`R(s)` 为 stage `s` 的可选资源实例集合 {(m, chamber, side)}；`free(inst)` 为实例当前空闲时刻；`p(b,s)` 为加工时长（确定性路径取 μ，噪声路径取采样值）；`s_last` 为工艺末阶段；`max_b` 表示 lot 须所有子批完成才算完成。

三条设计原则（直接关系正确性）：

1. 实例选择用 `argmin max(ready, free)`（**最早能开工**）而非 `argmin free`（最早空闲）——多实例忙闲不均时两者结果不同，后者会把子批挤到同一实例。
2. 分配后**必须更新** `free(inst*)`，否则所有子批被算到同一最早空闲实例上。
3. `estimate()` 以空 free 调用（状态无关、可缓存）；`schedule_on_calendar()` 以真实日历 free 调用（状态相关、不可缓存但**非破坏性**）。二者共享核心 ⇒ qtime mask 预判的 makespan 与 commit 实际 makespan **同口径**，修复了二者此前用两套算法导致的系统性偏差。

> **估计缓存（报告 §1.5 开销警示）**：makespan 分布只依赖 (l, m, p, n_mc)（静态编码器数据，不读 state），`start_offset` 仅在返回时加到 μ_fin。故 base 结果（offset 0）按 (l, m, p, n_mc) 缓存、offset 每次重新施加。`ResourceCalendarEnv` 在 `reset()` 时清缓存（base 时间/状态无关，整 episode 有效）。这把 50-lot pressure 实例的候选池构建从 ~0.49s 降到 ~0.086s。正确性依赖 `estimate` 状态无关；若未来改为读 state，缓存失效逻辑须移到 `advance_time`/`commit`。

### 3.4 蒙特卡洛完成时间分布

因多子批并行产生 E[max] > max(E[·]) 偏置，仅用各阶段均值算 makespan 会**系统性低估**。用蒙特卡洛修正：对 k = 1, …, K，每个 (b, s) 采样 p_actual(b,s) = μ(b,s) + ε, ε ~ N(0, σ²) 独立，跑一遍 `schedule_deterministic` 得 C_k，则

```text
μ_fin(l,m,p) = (1/K) · Σ_{k=1}^{K} C_k,

σ_fin(l,m,p) = √( (1/K) · Σ_{k=1}^{K} (C_k − μ_fin)² )
```

**符号说明**：`K` 为 MC 采样次数（默认 50；mask 用 20，doomed 判定用 10）；`C_k` 为第 `k` 次采样下 lot 的完成时刻。噪声模型为纯工艺随机抖动——无偏、独立、估计时不可知、做完才揭晓；μ, σ 均作为工艺参数进入 agent 状态特征。沿关键路径累加独立噪声，σ_fin ≈ √(Σ_s σ_s²)。

### 3.5 Q-time 机会约束（核心数学）

加工时间随机 ⇒ Q-time 不能写成确定性硬约束，改写为**机会约束**。因噪声无偏独立、finish_time 近似正态（CLT），违规概率有闭式：

```text
P(违规) = P(finish_time > deadline)
        = Φ( (μ_fin − deadline(l)) / σ_fin )
```

机会约束「屏蔽 P(违规) > ε 的动作」等价于：动作**可行**当且仅当

```text
┌─────────────────────────────────────────────────────────────┐
│ deadline(l) − μ_fin(l,m,p)  ≥  z_ε · σ_fin(l,m,p)          │
└─────────────────────────────────────────────────────────────┘
```

**符号说明**：Φ 为标准正态 CDF；`z_ε` 为容忍违规率 `ε` 对应的分位点（ε = 2% ⇒ z_ε ≈ 2.05）。左侧是"完成时间距 deadline 的安全余量"，右侧是"考虑随机波动后的安全边际"。

由此 safety margin **不再是常数**，而是自适应的

```text
qtime_safety_margin(l) = z_ε · σ_fin(l)
```

即 margin = "容忍违规概率"×"完成时间波动"——波动大的工件自动留厚垫、波动小的留薄垫。`z_ε`（或 `ε`）成为唯一需设定的、领域可解释的旋钮。

> **诚实边界**：正态噪声尾部无限长，违规概率恒 > 0。保证的是 P(违规) ≤ ε，**而非违规归零**。可控的是 `ε`，不是绝对的 0。为什么写成机会约束而非奖励项：Q-time 是硬约束，不应靠 reward shaping"劝一劝"；放进 mask 才能在搜索和 PPO 两边保持一致的可行动作空间。

**链式 Q-time（当前实现 `qtime_mask_mode`）。** 代码支持三模式（发表口径按 §9.4.1：`chain_joint` 为正文方法，`aggregate`/`chain` 降为消融行）：
- **`aggregate`**（默认）：单一聚合 deadline proxy，比较总完成时间 μ 与 deadline；
- **`chain`**：每次 dry-run 后评估真实 per-stage `q_time_limits` 链（如 (1,2)、(2,3)），用 `compute_q_time_violation`；
- **`chain_joint`**：`K` 次带噪声 dry-run，估计联合机会约束

```text
P_chain(a) = (1/K) · Σ_{k=1}^{K} 1{样本 k 任一链窗违约},

若 P_chain(a) > θ_chain  ⇒  屏蔽.
```

**符号说明**：1{·} 为指示函数；`θ_chain` 为 `qtime_chain_threshold`（默认 0，即任一样本出现链窗违约就屏蔽）；`K` = `qtime_chain_mc`。

> **实测发现**：`aggregate` proxy 在特定实例上会漏掉链窗口违约（纯 FIFO Q-time 违规 ~13.6），`chain`-aware mask 把它降到 ~0.4，`chain_joint` 残留清零。原因：聚合代理筛错对象——它比的是"总完成 vs 总 deadline"，而真正会断的是 stage 间的真实窗口 (1,2)/(2,3)。该实测正是论文消融叙事"代理为什么漏检"的素材（§9.4.1）。

## 4. 到达模型与前瞻机制

> **主线位置**：本节定义矩阵（§2.0.3）的"到达"两行——窗内可见（④对冲的对象）与窗外黑盒（⑤兜底的对象）的信息分区；这是本项目区别于通用 FJSP 的关键设定，也是 VC-MCTS 预留规划器存在的根本理由。

### 4.1 黑盒到达 + 有限前瞻窗

工件**随机到达、到达率未知**（不假设泊松或任何分布），但系统具备**有限前瞻**：lot 在真正进入待派工池前，会在前瞻窗 `W` 内变得可见（已 release / 在途 / ETA 已知），可见信息含 priority 与 qtime；窗外仍为完全黑盒。

> 基准实例**可用具体到达过程（如 Poisson）生成**到达序列（§10.1 的错峰到达），但这只是数据生成方式；**agent 状态/策略不得依赖到达率或分布形式**，窗外一律按黑盒处理。

### 4.2 违规来源的两分

```text
V_qt_total = V_qt_in-window(← 可预判, mask 挡) + V_qt_out-window(← 不可预判, λ 软压)
```

(a) 已可见 lot 间的相互挤占——其 deadline 已知，某动作是否害它们踩穿窗口当前可按完成时间分布估算 ⇒ 机会约束 mask 屏蔽。(b) 窗外到达的紧急 lot 抢占资源——信息上无法在当前时刻完全避免 ⇒ 残差通道（固定权重或自适应 λ）统计性兜底 + VC-MCTS 窗内选择性预留。随机到达只削弱 (b) 的可预判性，(a) 照常可 mask，且通常是违规大头。

### 4.3 高优先级资源预留与 ROP（预留机会点）

因前瞻窗内能看见即将到达的高优先级 lot，"为它预留资源"从**不可做**变为**可仿真、可预判**。但预留不能写成无目标的 `hold`——必须有对象、有责任、有验收。ROP 是**触发搜索的机会检测器与算力闸门**，回答"是否值得建树搜一搜"，不替规划器决定"值不值得预留"。

**第一层硬可行过滤。** 候选预留对 (m, h) 进入搜索前须满足：

```text
1. m 空闲且未被其他 reservation 冻结;
2. h 在前瞻窗 [t_now, t_now+W] 内, 尚未到达但已可见;
3. compatible(m,h) = 1 (m 结构上能加工 h);
4. not is_doomed(h);
5. reserve(m,h) 不会使其他已可见 lot 直接违反 Q-time 机会约束.
```

形式化的存在性门控：

```text
∃ (m,h): m ∈ M_idle, h ∈ F_W, compatible(m,h) = 1,
         ¬is_doomed(h), qtime_safe_after_reserve(m,h).
```

**第二层软机会评分（只排序、不决策）。** 对通过硬过滤的 (m, h) 算启发式分数

```text
S_res(m,h) = Δprio(h) + qtime_urg(h) + scarcity(h) + eta_overlap(m,h)
           − delay_cost(m) − idle_cost(h)
```

候选集为 P_t = TopB_{S_res}{(m,h) | 满足硬过滤}。**符号说明**：`Δprio` 为优先级增益、`scarcity` 为兼容机器稀缺度、`eta_overlap` 为"现在派工会不会覆盖 h 的 ETA"、`idle_cost` 为预留造成的空转成本。

> 同型 vs 异型：同型可弱化为"保留 k 台空闲额度"（数量预留）；**异型柔性下必须保留能加工 h 的特定机器 m（身份预留）**，故搜索必须显式建模 (m,h) 对。预留是**按机器粒度**——只冻结被选中机器，不冻结整组。若搜索选中 reserve(m,h)，ledger 登记 `(machine=m, target_lot=h, reserve_start, target_eta, expire, status=pending)`；h 到达后成功用上记 `hit`，超时/被替代/不可兑现记 `miss/waste` 并释放。

### 4.4 加工时间不确定性

每个 (b, s) 的实际加工时间 p_actual(b,s) = μ(b,s) + ε, ε ~ N(0, σ²) 独立无偏（§3.4）。该设定排除了"系统性差异（隐藏变量，应补特征）"与"长尾故障（需单独建模）"——本方案在纯噪声假设下成立。

**对训练与评价的要求**：训练时环境每步采样 p_actual，agent 状态用 (μ, σ)，Q-time 判定与 reward 用采样实际值（否则 agent 没见过波动、学不会留余量）；评价时 Q-time 违规率在多次随机 rollout 上统计（报告期望 + 分位数），而非单次确定性回放（`process_noise_enabled` + 每 seed 一个 `noise_seed`，§10）。

## 5. 约束与候选池：qtime 硬约束 + priority 强偏好

> **主线位置**：流水线第②级（约束化）——qtime mask 执行字典序第 0 层（机制 1 的准入侧，矩阵"加工噪声 × 准入"格），priority filter 执行第 1 层的资格准入；二者只收缩动作空间、不做选择。

### 5.1 候选池四步流水线（顺序不可调换）

```text
第0步 结构可行集: A(t,m) = {(l,p) | l 已到达且未完成, m 可加工 l, p 是 (l,m) 可用 PPID,
                              PPID 步骤完整, 所需 Chamber/Side 存在}
      ↓ ① qtime-safe mask    —— 硬约束: 屏蔽会断送可见 lot 的动作
      ↓ ② priority filter     —— 强偏好(在 ① 存活集合内)
      ↓ ③ CandidateScore 排序  —— 软目标启发式打分(仅排序、不删除)
      ↓ ④ TopK + padding + valid_mask = candidate_pool (固定长度 K_action)
```

数学上是集合的逐步收缩：

```text
A(t,m)  →[①]  A_safe  →[②]  A_prio  →[③④]  TopK_K(A_prio)
```

顺序不可调换的理由：Q-time 是硬约束、priority 是强偏好、utilization 才是软目标——混成一个分数会让"高优先级但高风险"和"安全但普通"混在一起。`valid_mask` 只标记 padding 位（+ 防御性结构不可行位）；喂给 SAS 的候选池**已是 qtime-safe 且 priority 过滤后**的。

### 5.2 qtime-safe mask（窗内可预判部分）

```text
function qtime_safe_mask(candidate_actions, visible_lots, C_t, config):
    for i, (lot, ppid) in enumerate(candidate_actions):
        C_sim = dry_run(C_t, lot, m, ppid)           # 非破坏
        for l in visible_lots:                        # 已到达 + 前瞻窗内即将到达
            if is_doomed(l): continue                 # 注定违规者不作屏蔽依据(防死锁)
            mu, sigma = estimated_next_finish(l, C_sim)
            if deadline(l) - mu < config.z_eps * sigma:   # ⇔ P(违规) > ε
                mask[i] = 0; break
```

两点必须说清：
1. **必须排除 `is_doomed` 的 lot**：若某 lot 即便等待也救不回（deadline(l) − 最早可能开始 < 0），它会让每个候选都触发屏蔽 → 候选池全空 → 强制等待 → 既排不了工、那个 lot 照样违规（死锁）。故其违规直接计入指标、调度继续推进。
2. **保证强度要诚实（随机版）**：对**被调度 lot 自身**保证 P(违规) ≤ ε；对**其他可见 lot**只是"显著降低"（真实开始时刻仍取决于后续决策）。本 mask 屏蔽的是"会当场断送某等待 lot 的动作"，而非"任何可能让其变紧的动作"。

控制流：mask 后候选池为空时视为"本次派工不可行"，交回上层时机层推进时间，**不计执行失败惩罚**（r_exec = 0，§6.2）。

### 5.3 priority filter（强偏好）

soft 模式不删动作、仅排序；strict 模式保留

```text
prio(l) ≥ max_{a ∈ A_safe} prio(a) − δ_gap
```

**符号说明**：δ_gap 为 `priority_min_gap`。strict 模式下，只有优先级接近最高的动作保留——RL **物理上不可能**选到低优先级候选；soft 模式只重排候选、RL 仍可探索。CLI：`--priority-mode soft|strict` / `--priority-min-gap`。为什么 priority 用 filter 而非 reward：priority 是强约束式排序条件，filter 实现字典序比 reward shaping 更稳定。建议先 `soft`（保留探索），稳定后视需要切 `strict`。

### 5.4 候选打分

对每个真实候选 a = (l, p)，启发式排序分

```text
score(a) = due_urg(a) + wait(a) + 0.1·qtime_slack(a)
         − 0.001·proc(a) − 0.001·qtime_risk(a)
```

```text
qtime_slack(a) = max( 0,  deadline(l) − predicted_completion(a) )
```

**符号说明**：`due_urg` 交期紧迫度、`wait` 已等待时间、`proc` 总加工时长、`qtime_risk` Q-time 风险特征、`predicted_completion` 来自 dry-run。它不试图"学全局最优"，只在 qtime-safe + priority 过滤后给 SAS 一个稳定的局部排序信号。候选池 18 维特征（index 0–17）：`is_real, is_wait, score, arrival_time, waiting_time, machine_slot_start, machine_load, total_process_time, predicted_completion, stage_count, qtime_risk, wafer_count, priority, due_slack, priority_rank_norm, due_slack_rank_norm, is_best_priority, is_most_urgent_due`。

## 6. 上层 SAS：动作选择、向量奖励与多头 PPO

> **主线位置**：流水线第③级（学习，System-1 快策略）执行字典序第 1 层·当下；§6.5 的 λ_qt 属第⑤级兜底，填充矩阵"窗外黑盒 × 兜底"格的第 0 层残差。

SAS（Schedule-Action Selector）在当前空闲机台上选一个 `(l,p)` 派工。**SAS 自身不拥有 wait/reserve 决策**——这些由 VC-MCTS 或上层时机层处理。两条策略/训练路径并存：单头（标量 reward，`Phase2SASActorCritic` + `Phase2PPOTrainer`）与多头（向量 reward，`Phase2SASMultiHeadActorCritic` + `MultiHeadPPOTrainer`），由 `--mode multihead` 选择。

### 6.1 SAS 动作选择（masked softmax）

actor 输出候选池上的 masked softmax：

```text
π_θ(i | o_t) = exp(ℓ_i) · m_i  /  Σ_j exp(ℓ_j) · m_j
```

**符号说明**：ℓ_i 为 actor 对第 i 个候选的 logit；m_i ∈ {0,1} 为 valid_mask（padding/无效位为 0）；分母只对有效动作归一化，无效动作概率为 0（实现上 padding 位 `masked_fill(-inf)`）。训练时采样，推理/greedy 取

```text
i*(t) = argmax_i  π_θ(i | o_t)
```

为什么用候选池而非对所有 lot 建模：动作本质是"在当前机台上选一个可执行 `(l,p)`"，固定长度候选池更易处理 mask、padding 和 PPO 轨迹一致性。观测由 `Phase2ObservationEncoder` 给出：候选级 18 维（§5.4）+ 全局 9 维（`lookahead=True` 时 13 维：补 upcoming count / max priority / min qtime / earliest ETA）+ 机器级特征作注意力 query。

### 6.2 向量奖励（三通道）

奖励不混合加权，改为三通道向量（固定顺序 `(exec, qtime, util)`）：

```text
r_t = ( r_exec_t,  r_qtime_t,  r_util_t )
```

**执行通道**（即时密集，含 packing 利用率质量）：

```text
r_exec_t =
  +0.20 + w_pack · total_work / span      if insertion_success
  −0.40                                    if insertion_failed
  −0.50                                    if mask_invalid (防御性，正常不触发)
   0.00                                    if wait / noop (池空或全屏蔽)
```

**符号说明**：`total_work` 为该 lot 所有 (子批×阶段) 加工时间之和，`span = lot_end − lot_start`；`packing = total_work / span` 是利用率向的逐步边际质量——腔体争用拉长跨度 ⇒ packing 变小，agent 每步就能感知选得紧不紧凑；`w_pack` 默认 0.10。

**Q-time 残差通道**（逐步密集，telescoping）：

```text
r_qtime_t = − ΔV_qt_t / N_lots,       Σ_t r_qtime_t = − V_qt_total / N_lots
```

**符号说明**：ΔV_qt_t = q_after − q_before 为本次 commit 新增的 Q-time 违约增量；N_lots 为 lot 总数（归一化）。逐步与终局总惩罚等价（telescoping、不双计），但 credit 大幅改善——agent 直接知道是哪一次派工害了 Q-time。

**利用率通道**（仅终局，唯一软目标）：

```text
r_util_t = 1{is_terminal(t)} · normalize(Ū)
```

**符号说明**：Ū 为完整日历下的平均机台利用率。注：旧 `progress` 通道已删除（在能完成全部 lot 的实例上恒为 1.0、是无梯度死重）；拖期不进奖励（仅评估指标）；SAS `wait_penalty=0`（主动等待成本归 VC-MCTS ledger）。

### 6.3 多头 critic + 逐目标 GAE

三个通道量纲、稀疏程度、终局属性完全不同，单头 critic 会把梯度混在一起。对每个通道 c 独立计算 GAE：

```text
δ_c_t = r_c_t + γ · V_c(z_{t+1}) · (1−done) − V_c(z_t),

Â_c_t = Σ_{τ≥0} (γ·λ_gae)^τ · δ_c_{t+τ}
```

然后逐通道归一化并加权合成：

```text
Â_t = Σ_{c ∈ {exec, qtime, util}}  w_c · norm(Â_c_t)
```

**符号说明**：γ 折扣因子；λ_gae GAE 衰减（**不是** Q-time Lagrangian 对偶变量）；`norm(·)` 在 **batch 层面**做均值方差归一化（逐通道，因 exec/qtime 每步非零、util 多为 bootstrap 传播值，尺度不同）；w_c 通道权重（典型 w_exec = 1.0、w_util = 0.5、w_qtime = 3.0 大值，体现硬约束优先；r_qtime 本身为负 cost，故统一加法）。逐通道由 `combine_channel_advantages()` 实现。

### 6.4 PPO 目标

策略比率与 clipped objective：

```text
ρ_t = π_θ(a_t | z_t) / π_old(a_t | z_t),

L_policy = −E_t[ min( ρ_t·Â_t,  clip(ρ_t, 1−ε_clip, 1+ε_clip)·Â_t ) ]
```

value loss（逐通道）与总损失：

```text
L_value = Σ_c  c_v · ( V_c(z_t) − R_c_t )²,    R_c_t = Â_c_t + V_c(z_t)

L_total = L_policy + L_value − c_e · H(π_θ)
```

**符号说明**：ε_clip PPO clip ratio；c_v value loss 权重（各通道同值）；c_e entropy 系数；H(π_θ) 策略熵（鼓励探索）；R_c_t 通道 c 的 return。

> **PPO 轨迹一致性（关键不变式）**：训练时若 `sas_step()` 返回 `insertion_failed`，采样动作**保留**在 buffer 中（r_exec = −0.40），**不得**替换为下一个可行动作——否则 log π 与 PPO 比率失真。替换 fallback 仅用于推理（`run_greedy_episode`）。

### 6.5 Q-time Lagrangian（PPO-Lagrangian / CMDP）

mask 挡不住窗外到达 + 随机尾部 + doomed lot 的残差违规。在训练目标层面表述为 CMDP：

```text
max_π  E[util]   s.t.   E[V_qt] ≤ ε

⇔  max_π min_{λ≥0}  E[util] − λ·( E[V_qt] − ε )
```

启用 `use_qtime_lagrangian=True` 时，w_qtime 由自适应对偶变量 λ_qt 取代（`qtime_weight()`），每 episode 做一次对偶上升（`update_lambda`）：

```text
┌───────────────────────────────────────────────────────────────────┐
│ λ_qt ← clip( max(0,  λ_qt + η_λ · (Ê[V_qt] − ε) ),  0,  λ_max ) │
└───────────────────────────────────────────────────────────────────┘
```

per-episode 违规率由 buffer 反推：qtime_cost = −Σ_t r_qtime_t（`episode_qtime_cost()`）。注意 r_qtime 是**逐步**的（§6.2），但 telescoping 使其和恰为终局总违规率，故 qtime_cost = V_qt_total / N_lots——逐步还是终局口径下该 cost 都相等。

**符号说明**：η_λ 对偶学习率（`qtime_lambda_lr`，须 ≪ 策略 lr）；ε cost budget（`qtime_cost_budget`）；λ_max 上界（`qtime_lambda_max`）。你只需设含义明确的 ε，λ 自动停在约束边界。CLI：`--qtime-lagrangian --qtime-budget 0.02 --qtime-lambda-lr 0.1`。固定权重与 Lagrangian 作用于**同一** qtime 通道、**不并用**。

> **诚实边界**：(1) 严格收敛保证在凸/表格情形成立，深度非凸下打折（深度 RL 通病）；(2) 保证的是**期望约束** E[V_qt] ≤ ε，与 §3.5"随机下无法绝对归零"自洽；(3) λ 可能震荡，必要时用 PID-Lagrangian 稳定。理论根基：拉格朗日对偶 + KKT（Boyd & Vandenberghe）；CMDP（Altman 1999）；CPO（Achiam 2017）、PPO-Lagrangian（Ray et al. 2019）、PID-Lagrangian（Stooke et al. 2020），属 Safe RL。**约束一致性升级（§7.12.3）**：该 λ_qt 同时作为在线 VC-MCTS 利用项中窗外残差风险的价格——离线学到的影子价格直接为在线搜索定价，prior / rollout / final pick 三处对 Q-time 口径统一。

## 7. VC-MCTS：可见性感知的约束式 MCTS 在线预留

> **主线位置**：流水线第④级（对冲，System-2 慢策略）执行字典序第 1 层·未来；机制 1/2/3 分别填充矩阵（§2.0.3）"加工噪声 × 准入"、"窗内到达 × 评估"、"加工噪声 × 评估"三个格子。

> **VC-MCTS = Visibility-aware Constrained Monte-Carlo Tree Search**：以显式 reserve(m,h) 为搜索动作的在线预留规划器，取代上一版的学习式 RMA。预留不再靠 PPO 从稀疏奖励里学，而是在每个 ROP 用一棵搜索树、以资源日历 clone 当忠实仿真器，把"现在派 vs 留给未来"两条分支模拟比一比。

### 7.1 为什么用搜索取代学习式 RMA

预留是反事实问题——"现在派 vs 留给未来"谁更好，只有把整条资源日历向前模拟之后才知道。学习式 RMA 有三处结构性顽疾：

```text
1. 奖励坍缩: 预留收益终局才体现, 稀疏 reward 难 credit
2. greedy under-hold: 贪心推理在"高优先级到达前那一下" P(reserve) 顶不过 0.5
3. 类别不平衡: ROP 宽触发制造大量 no_reservation 正确样本, 策略被推向"从不预留"
```

VC-MCTS 换范式：不学概率，而是把 reserve(m,h) 和 dispatch 都作为搜索边，用基策略 rollout 到底、按字典序目标比较——没有梯度、没有 threshold、没有 advantage 归一化，三个失败模式在结构上不存在。代价是把训练期成本搬到推理期（每个 ROP 要算若干次仿真，§7.9；该成本由 §7.13 的 OCBA 预算分配定理化地压缩）。

> **定位（写进 related work）**：VC-MCTS 是 **DyRo-MCTS (Chen et al., 2025)** 的原理性推广。DyRo 因完全看不见未来到达，只能用钝的机器空闲分布代理（结论"避免早期空闲"）。本项目在有限前瞻窗内未来到达可见，于是"定向空闲（预留）"从浪费变为最优：窗内用显式仿真 + reserve 边取代钝代理，窗外退为 DyRo 式代理。**当 W = 0、能力同质、无 Q-time 约束、加工确定时，本方案退化为 DyRo-MCTS**（退化链的完整刻画见 §7.12.2 性质 3）。

### 7.2 一棵搜索树，三个挂载点

三个机制不是平级模块，而是同一棵树上三个环节的挂载点（亦即矩阵 §2.0.3 的三个格子），执行有严格先后，复刻 §2.4 的字典序：

```text
节点 = 状态 s (资源日历 + reservation ledger + 窗内可见 future lot 作"确定性实体")
边   = 动作 a ∈ { dispatch / delegate_dispatch } ∪ { reserve(m,h) } ∪ { no_op }

挂载点 A —— 准入层 (机制 1: Q-time 约束)        对应字典序第 0 层 [✅ 已实现]
挂载点 B —— 选择/估值层 (机制 2: ρ_pc 优先级-能力对冲水位)  字典序第 1 层 [⬜ 深度设计 §7.12]
挂载点 C —— 评估层 (机制 3: CRN 配对 + OCBA + CVaR)    贯穿全树    [⬜ 深度设计 §7.13]
```

### 7.3 根动作空间

```text
R_root = {no_op}
       ∪ {dispatch(m) 或 delegate_dispatch(m)}
       ∪ {reserve(m,h) | (m,h) ∈ P_t}
```

- `use_delegate_dispatch=False`：root 展开多个 dispatch(m) 边，具体 `(l,p)` 由候选池 TopK 高分动作给出（`--dispatch-delegate topk`，旧行为/消融）；
- `use_delegate_dispatch=True`：root 展开**一个** delegate_dispatch(m) 边，具体 `(l,p)` 由 `RuleDispatchDelegate`（`--dispatch-delegate rule`）或 `SASPolicyDispatchDelegate`（`--dispatch-delegate sas`，非法/wait/padding 回退 rule）给出。

为什么把 dispatch 委托出去：VC-MCTS 只需决定"要不要留、留给谁"，派工细节交给已训练的 SAS，符合模块分工、减少 root 分支。`DispatchDelegate`（`dispatch_delegate.py`）是"选哪个候选"与"何时选/预留"的抽象边界，只返回动作索引、不改 driver/ledger 状态。

### 7.4 目标向量与字典序比较

每条 rollout 末端计算目标向量

```text
obj(a) = ( V_qt_count(a),  V_qt_total(a),  O2(a),  −Ū(a) )
```

其中优先级加权等待（O2）

```text
O2 = Σ_l  prio(l) · max( 0,  start(l) − arrival(l) )
```

比较规则是字典序：先比 V_qt_count → 再 V_qt_total → 再 O2 → 最后利用率。**符号说明**：V_qt_count Q-time 违约数（硬约束门槛）、V_qt_total 违约严重度总量、O2 优先级维度的优化主目标、Ū 平均利用率（取负因目标越小越好）。O2 在线到达下良定义、天然惩罚空转——这正是 §10.2 用它取代病态旧 `priority_violation` 的原因。

### 7.5 UCT 选择

每条边的选择分数

```text
UCT(e) = score( mean_obj(e) ) + c · prior(e) · √( log N / n(e) )
```

其中标量 exploitation score（与代码 `objective_to_score` 完全一致）

```text
score(obj) = −β₁·V_qt_count − β₂·V_qt_total − O2 + w_U·Ū
```

**符号说明**：`mean_obj(e)` 该边累计 rollout 的平均目标；`prior(e)` 边的先验（`edge.action.prior`）；`N` root 总访问数；`n(e)` 该边访问数；`c` 探索常数（`exploration_c`，默认 1.5）；β₁ = `qtime_penalty`（默认 10000）、β₂ = `qtime_total_penalty`（默认 1000）、w_U = `util_weight`（默认 1.0）。β₁ ≫ β₂ ≫ 1 使标量分数近似复刻字典序。warm-up 保证小预算下每条 root edge 至少被 rollout 一次：iteration_count = max(|R_root|, n_iter)。

### 7.6 Final pick 与 no_op gating

最终选择按字典序

```text
a* = argmax_a  ( −V_qt_count,  −V_qt_total,  −O2,  Ū,  n(a) )
```

访问数 n(a) 仅作最后平局项。但 `no_op` 有门控：若 `no_op` 胜出且存在非 no_op 替代边，`no_op` 必须在 V_qt_count 或 V_qt_total 上**严格优于**最佳非 no_op 边，否则降级为最佳 dispatch/reserve 边。

> **为什么要 no_op gating**：早期完整轻预算中，无门控时 120 次决策里 no_op=89、episode 只完成 31/50（小预算下 no_op 因统计噪声被过度放大，"什么也不做"成伪最优）。加门控后 no_op→1、完成 50/50。

> **目标升级（§7.13.4）**：final pick 的均值字典序将升级为**风险字典序**（分位数 + CVaR），no_op gating 门控条件同步改用分位数口径；当前实现等价于 β = 0（CVaR = 均值）的退化情形，作消融基线。

### 7.7 到达概率加权（可选，已落地）

启用 `arrival_prob_weighting` 时，远期 lot 的预留收益按 ETA 距离折扣：

```text
P_arr(η, t) = exp( −ρ_decay · max(0, η−t) / W )
```

```text
obj_reserve_eff = P_arr · obj_reserve_arrive + (1−P_arr) · obj_reserve_miss
```

**符号说明**：η future lot 的 ETA；ρ_decay 衰减强度（`arrival_prob_decay`）；obj_arrive future lot 按预期到达兑现 reserve 的目标、obj_miss 未兑现退回最佳非 reserve 分支的目标。远期 lot 的预留收益不应与近端 lot 一样计分，否则高估很远未来的 hold。

### 7.8 叶子 Critic 截断（可选 AlphaZero 增强，已落地、默认关闭）

启用 `use_leaf_value=True` 时，用多头 critic 提供截断估值而不必每次 rollout 到终点（`MultiHeadCriticLeafValue`）：

```text
V_qt_count  ≈  max( 0,  −V_qtime · N_lots )         (telescoping)
Ū            ≈  clip( V_util, 0, 1 )                (terminal)
```

V_qt_total 与 O2 两个无对应 critic 通道的维度由 partial rollout（`leaf_rollout_depth` 步，单调下界）填充。这仍是可选增强——默认走完整 rollout 以保证准确性（见 `docs/superpowers/plans/2026-06-07-vc-mcts-alphazero-prior-leaf.md`）。发表口径按 §9.4.1 列为支线：与 §7.13.3 的 OCBA 正交组合为 **compute-adaptive anytime planner**（critic 估值不确定时才深搜）。

### 7.9 Rollout 同源（命门）+ 机制 3 入口

rollout（叶子往下用基策略续跑到底）**必须和树策略用同一套规则**，否则用"无视 Q-time、无视 ledger 的钝策略"模拟未来，估计系统性失真：

```text
rollout 内部基策略必须:
  ① 也走 qtime-safe 候选池 (机制1硬挡在仿真里照样生效, 否则 q̂ 乐观偏高)   [✅ 已落地]
  ② 尊重 reservation ledger (已 reserve 的机器不被基策略抢派, 等 h 到优先上,
     否则 reserve 与 dispatch 分支 q̂ 无差异, 预留收益被仿真抹平)            [✅ 已落地]
  ③ 同节点各边复用同一场景库的随机数流 (CRN 同源, 机制 3 组件一)             [⬜ 深度设计 §7.13]
```

> **机制 3 已升级为三组件深度设计（CRN 配对比较 + OCBA 预算分配 + CVaR 风险字典序），完整数学与挂载点见 §7.13**。报告 7 原"同节点共享噪声种子做方差缩减、N_mc 取 3–8"的表述仅是其中组件一的特例。当前 rollout 无噪声注入——待落地。

### 7.10 统一选择公式（DyRo-UCT 目标架构）与三机制现状

目标架构把 PUCT 利用项扩为约束式 DyRo-UCT：

```text
a* = argmax_a  [ E(s,a) + c·p(s,a)·√n(s) / (1+n(s,a)) ]

E(s,a) = feasible(s,a)·[ α·q̂(s,a) + (1−α)·ρ̂_pc(s,a) ] − λ_qt·ĉ_qt(s,a)
```

**符号说明 + 落地状态**：
- `feasible(s,a) ∈ {0,1}`——机制 1 硬侧，窗内可预判违规的边置 0【✅ 已实现，qtime-first 字典序选择 + no_op gating】；
- `λ_qt·ĉ_qt`——机制 1 软侧，与 §6.5 PPO-Lagrangian **同一乘子**（约束一致性，§7.12.3）【目标设计，在线侧软压未单独接】；
- `ρ̂_pc(s,a)`——机制 2 优先级-能力**对冲水位**，已从报告 7 的"加权产能求和"升级为**二部匹配裕量**（深度设计见 §7.12）【⬜ 未落地，当前 UCT 利用项只有 −O2 + w_U·Ū】；
- `q̂(s,a)`——目标设计中升级为机制 3 的风险化估值（CRN 配对场景上的 CVaR/分位数口径，§7.13）【⬜ 当前为均值口径】；
- `p(s,a)`——先验，目标用训练好的 SAS（`prior_source="policy"` → `SASPolicyPriorProvider` 把候选 masked-softmax 当先验，reserve/no_op 边给固定注入先验）【✅ 已落地，默认 `heuristic`（`action.prior` 近似）】；
- `c, α`——探索常数 / 价值-对冲插值（DyRo 默认 α ≈ 0.5–0.6；消融方案 §7.12.5）。

> **当前实际口径**：`objective_to_score = −β₁·qtime_count − β₂·qtime_total − O2 + w_U·util`，p(s,a) 由 `action.prior` 近似（或 policy 先验）。ρ̂_pc 与在线 λ 软压未完整落地，是机制 2/3 的待办（§12）。

主循环伪代码：

```text
def vc_mcts_decide(env, decision_point, sas_prior, N_iter, N_mc):
    rop_pairs = env.detect_rop(decision_point)               # §4.3 宽触发 + S_res TopB
    if not rop_pairs:
        return sas_prior.greedy_action(env, decision_point)  # 无杠杆, 免费走 delegate
    root = Node(env.snapshot())                              # clone state+ledger (非破坏)
    for _ in range(max(len(root.edges), N_iter)):            # warm-up (机制3落地后改为 OCBA 两阶段, §7.13.3)
        node, path = select(root)                            # §7.5 UCT 下行
        child = expand(node)                                 # 仅 feasible=1 的边 (机制1硬侧)
        q, util, c_qt = evaluate(child, sas_prior, N_mc)     # reservation-aware rollout, 同源 §7.9
        backprop(path, q, util, c_qt)
    return objective_first_choice(root.edges)                # §7.6 字典序 + no_op gating (→ 风险字典序 §7.13.4)
# reserve(m,h) → 登记 ledger; delegate_dispatch → 交 delegate commit; no_op → 推进时间
```

### 7.11 可解释性输出

搜索天然可解释：对被选动作，可导出根节点各边的访问次数与 q̂/ρ̂/ĉ 分解、reserve 与 dispatch 分支的反事实 O2 差、预留命中/超时/浪费的事件记账（ledger）。每个数都是模拟出来的、可复算。决策写 JSONL（`selected_action`、`edges`、`mean_qtime`、`mean_qtime_total`、`mean_o2`、`mean_util`、diagnostics）；`vc_mcts_trace_summary.py` 自动统计 `reserve_available_decisions`、`reserve_selected_decisions`、`reserve_selection_rate_when_available`、`reserve_o2_gap_vs_best_non_reserve_avg` 等。机制 2/3 落地后扩展字段见 §7.12.4、§7.13.5。

### 7.12 机制 2 深度设计：优先级-能力对冲水位 ρ_pc

#### 7.12.1 动机：DyRo 的 ρ 在异型 FAB 下是错误尺度

DyRo-MCTS 的鲁棒性 ρ(s,a) 度量"状态对**任意**未来到达的容忍度"，用机器空闲分布作代理。该代理隐含**能力同质**假设：任何空闲机器都能承接任何到达。异型柔性下此假设失效——空闲产能 ≠ 可兑现产能：若空闲的机器都不能加工即将到达的高优先级 lot h，再多空闲也是零对冲。故本机制把鲁棒性从"空闲量"重定义到**（优先级类 × 加工能力）联合空间上的可兑现承接裕量**。

#### 7.12.2 定义：二部匹配裕量及其性质

t 时刻构造二部图 G_t = (M_t ∪ H_t, E_t)：

```text
H_t = { h ∈ F_W | prio(h) ≥ p_hi }                      # 窗内可见的高优先级 future lots
M_t = { m | m 空闲或将在 min_h ETA(h) 前空闲, 未被 ledger 冻结 }
(m,h) ∈ E_t  ⇔  compatible(m,h) = 1
              ∧  avail(m) ≤ deadline(h) − μ_proc(h,m) − z_ε·σ_proc(h,m)   # Q-time 可兑现
```

**符号说明**：`p_hi` 为高优先级类阈值（复用 priority filter 的类划分）；边的时间可行条件要求 m 不仅结构相容，且能在 h 的 Q-time 机会约束余量内承接（与 §3.5 同一 z_ε，口径一致）。

对冲水位定义为最大加权匹配及其归一化：

```text
ρ_pc(s) = max_{μ ⊆ E_t, μ 为匹配}  Σ_{(m,h) ∈ μ}  w_prio(h)

ρ̃_pc(s) = ρ_pc(s) / Σ_{h ∈ H_t} w_prio(h)  ∈ [0, 1]      # H_t = ∅ 时定义为 0
```

**语义**：ρ̃_pc = 1 表示"窗内每个可见高优先级 lot 都已有一台能力相容、时间可兑现的机器为其兜底"；ρ̃_pc < 1 的缺口即当前派工决策可能造成的不可逆挤占。

对根动作 a，令 s ⊕ a 为施加 a 的即时资源效果后的状态（dispatch(m)：m 在其占用区间内移出 M_t；reserve(m,h)：匹配被钉死包含 (m,h)；no_op：不变），定义

```text
ρ̂_pc(s, a) = ρ̃_pc(s ⊕ a),        Δρ_pc(a) = ρ̃_pc(s ⊕ a) − ρ̃_pc(s)
```

三条性质（论文中写作观察/引理）：
1. **单调性**：reserve 边 Δρ_pc ≥ 0（钉死可行边不减小最大匹配）；dispatch 边 Δρ_pc ≤ 0（移除节点不增大匹配）。
2. **杠杆刻画**：reserve(m,h) 有独立对冲价值 ⇔ Δρ_pc(dispatch(m)) < 0 且 (m,h) 在某最大匹配中——即"m 是 h 的稀缺承接者且当前派工会挤占它"。这把 §13.8 的经验判断（late_hi corr≈0.97 才有杠杆）升级为**可计算的结构条件**，并直接服务 §12.2 的相变扫描。
3. **退化链**：能力同质（compatible ≡ 1）且无 Q-time 时，ρ_pc 退化为空闲容量计数——即 DyRo 空闲代理的精神等价物；W = 0 时 H_t = ∅、ρ̃_pc ≡ 0，机制 2 自动失效，整体退回 q̂-only 搜索。与 §7.1 的"W=0 退化为 DyRo"声明闭环。

#### 7.12.3 UCT 集成与约束一致性

利用项即 §7.10 的目标架构：

```text
E(s,a) = feasible(s,a) · [ α·q̂(s,a) + (1−α)·ρ̂_pc(s,a) ] − λ_qt·ĉ_qt(s,a)
```

**符号说明**：q̂ 为（机制 3 风险化后的）目标估值经 `objective_to_score` 标量化并 batch 归一化到与 ρ̂_pc 可比的尺度；α 为价值-对冲插值（DyRo 经验值 0.5–0.6 作起点；α=1 即当前实现，作天然消融基线）；λ_qt **复用 §6.5 离线 PPO-Lagrangian 的对偶变量**——离线学到的 Q-time 影子价格直接为在线搜索的窗外残差风险定价，prior / rollout / final pick 三处对 Q-time 口径统一（**约束一致性**，可作加分项单独成段；若篇幅紧可降级为固定权重）。

#### 7.12.4 计算与挂载点

- 求解：匹配规模 |M_t| × |H_t| 在单个 ROP 处极小（机台组级别，通常 ≤ 10×10），匈牙利算法 O(n³) 或 scipy `linear_sum_assignment` 直接可用；仅根及一层子节点需要**增量重算**，rollout 内部不算 ρ_pc。
- 挂载点：新模块 `priority_capability_matching.py`（输入 ledger + 日历 + F_W，输出 ρ̃_pc 与边级 Δρ_pc）；`vc_mcts_planner.py` 在 edge 评分处把 `objective_to_score` 替换为 E(s,a)；trace JSONL 增加 `rho_pc_before / rho_pc_after / delta_rho_pc` 字段（沿 §7.11 约定，预留决策可回放为"对冲水位变化"）。
- 与 ROP 的关系：S_res（§4.3）仍只做 TopB 排序闸门不动；ρ_pc 是树内估值项，不替代 ROP。二者职责不混（合同表 §2.0.4）。

#### 7.12.5 消融与验收

| 实验 | 操作 | 预期/判据 |
|---|---|---|
| α 扫描 | α ∈ {1.0, 0.8, 0.6, 0.4}，late_hi + 能力稀缺梯度实例族 | 瓶颈紧时 α<1 显著降 O2 与预留 miss 率；瓶颈松时不劣于 α=1（无害性） |
| 对冲水位有效性 | 预留命中率 hit/(hit+miss+waste) 按 Δρ_pc 分桶 | Δρ_pc 大的预留命中率单调更高（机制按设计工作的直接证据） |
| 退化检验 | 同质能力实例 | α<1 与 α=1 无显著差异（退化性成立） |

### 7.13 机制 3 深度设计：CRN 配对 + OCBA 预算 + CVaR 风险字典序

#### 7.13.1 重新定位：这是排序与选择问题，不是估值问题

VC-MCTS 在每个 ROP 真正要回答的是 **reserve 与 dispatch 哪条边更好**（排序），而非各边目标值是多少（估值）。双重不确定性（随机到达 + 加工噪声）下，朴素做法是各边独立采样大量 rollout 求均值——方差大、预算浪费、且均值口径对零容忍的 Q-time 报废不忠实。本机制以三个相互咬合的组件重构评估层。

#### 7.13.2 组件一：CRN 配对场景

每个 ROP 节点固定一个**场景库** Ξ = {ξ_1, …, ξ_K}，（本节 K 专指场景数，即 §1 中 n_mc 的复用义，与候选池长度 K_action 无关。）每个 ξ_k 是一组完整随机数流，同时编码：(i) 各 (b,s) 加工噪声；(ii) 窗内到达时刻扰动（若启用）。同一节点下**所有候选边复用同一 Ξ** 做 rollout：

```text
obj^(k)(a) = rollout(s ⊕ a;  ξ_k),        k = 1, …, K
```

任意两边的比较使用**配对差**而非均值差：

```text
Δ^(k)(a, a') = O2^(k)(a) − O2^(k)(a'),
Δ̄(a,a') = (1/K)·Σ_k Δ^(k),
Var(Δ̄) = [ Var(O2(a)) + Var(O2(a')) − 2·Cov(O2(a), O2(a')) ] / K
```

**符号说明**：CRN 使两边 rollout 共享公共随机性 ⇒ Cov > 0 ⇒ Var(Δ̄) 远小于独立采样；要准的是 Δ̄ 的**符号**而非各自数值。Q-time 维度同理对 V_qt_count、V_qt_total 做配对。实现要点：rollout 接口增加 `scenario_id`，环境噪声流由 (node_id, scenario_id) 派生而非全局 RNG，保证"同 k 不同边"复用同一噪声实现——这是与 §7.9 ①②并列的**第三条同源纪律**。

#### 7.13.3 组件二：OCBA 顺序预算分配

不同 ROP 的难度悬殊：有的边一两次 rollout 即可分胜负，有的处在无差异区。均匀分配 K 是浪费。采用两阶段 OCBA：

```text
阶段一 (warm-up): 每条 feasible 边先跑 n₀ 条配对 rollout (n₀ = 3, 复用 Ξ 前 n₀ 个场景)
阶段二 (分配): 设当前最优边 b, 对 i ≠ b 以配对统计量分配剩余预算:

  N_i ∝ ( σ_i / δ_{b,i} )²,         δ_{b,i} = Δ̄(b, i),  σ_i = std({Δ^(k)(b,i)})

  N_b = σ_b · √( Σ_{i≠b} N_i² / σ_i² )

顺序停止: 当 Δ̄(b, second) 的 (1−γ) 置信下界 > 0, 或 |Δ̄| < δ* 且置信区间宽 < δ* (判无差异
          → 交给字典序下一层/访问数破平), 或预算耗尽 ⇒ 停止
```

**符号说明**：δ* 为无差异区宽度（领域可解释：O2 差小于 δ* 视为业务等价）；γ 为停止置信水平。OCBA（Chen et al. 2000）与 indifference-zone R&S 是仿真优化的标准理论，引用即可。**叙事价值**：§13.7"成本搬到推理期"的算力痛点由此转为贡献——预算被定理化地投到最可能翻转排序的边上，CRN 下 K = 3–8 即可给出稳定排序；并与 §7.8 叶子 critic 截断正交组合成 compute-adaptive anytime planner。**降级预案**：若工期紧，OCBA 可降级为"均匀预算 + future work"而不伤主线（组件一、三独立成立）。

#### 7.13.4 组件三：风险度量 final pick

K 个配对场景给出的是各边目标的**经验分布**，不止均值。半导体语义下 Q-time 报废零容忍、"宁可空转不可报废"，故 final pick（替换 §7.6 的均值字典序）改为**风险字典序**：

```text
第 0 层:  Q̂_β0( V_qt_count(a) )           # 违约数的 β₀ 上分位数 (β₀ = 0.95) — 尾部不是均值
第 1 层:  Q̂_β0( V_qt_total(a) )
第 2 层:  CVaR_β( O2(a) ) = mean( 最差 ⌈(1−β)·K⌉ 个场景的 O2 ),   β = 0.8
第 3 层:  −mean( Ū(a) );  访问数 n(a) 破平
```

**符号说明**：CVaR 经验估计直接对场景排序取尾均值，K 小时建议 β 取使 ⌈(1−β)K⌉ ≥ 2；β = 0（即 CVaR = 均值）退化回当前实现，作消融基线。no_op gating（§7.6）保留，门控条件同步改用第 0/1 层的分位数口径。**reserve 的风险语义**：预留胜出当且仅当其在尾部场景下也不劣——即 reserve 是对最坏到达/噪声实现的保险。这与机制 2 的"对冲"语义互为表里：**机制 2 给对冲的对象与水位，机制 3 给对冲的定价与验收**。

> **诚实边界**：(1) CVaR 的场景估计在 K 小时方差大——故必须与 CRN 配对联用（同场景排序，公共随机性在尾部比较中同样部分抵消）；(2) 本机制给出的是**经验分布意义下**的风险偏好排序，不主张分布鲁棒（DRO）保证；若审稿要求，可将场景库换为均值-协方差模糊集采样升级为 DR 版本，留作扩展声明而非本文主张。

#### 7.13.5 挂载点汇总

```text
NoiseScenarioBank      节点级场景库, (node_id, k) → 随机数流          [新增]
rollout(·, scenario_k) evaluate() 签名扩展, CRN 同源                  [改 vc_mcts_planner]
ocba_allocate()        替换均匀 iteration 循环 (warm-up 后逐边分配)    [改 §7.10 主循环]
risk_lexico_choice()   替换 objective_first_choice (含 no_op gating)  [改 final pick]
trace                  逐场景 obj 落盘 → 可复算 CVaR/分位数            [扩 §7.11 JSONL]
```

## 9. 与主流方法的对比及可借鉴点

### 9.1 主流技术地图（2023–2026）

1. **HGNN 状态表示**：析取图 + 异构 GNN（Song 2022/2023；Tang & Dong HGNNR 2024；HGT-Scheduler 2026）——把"工序流依赖"与"资源竞争"分开建模。
2. **注意力 / 双注意力 actor**（Wang 2023）——本项目双注意力 SAS 正属此线，**与主流对齐**。
3. **多智能体 / 协同 RL**——本项目上一版 RMA+SAS 属此线，**本版已把预留改为搜索式，故不主张此线为强项**。
4. **大规模 / 自监督 + 进化策略**（Tassel WSC 2023，SMT2020；Stöckermann 2025）——并指出学术 RL 缺真实 FAB 验证。
5. **Safe RL / CMDP**（PPO-Lagrangian）——本项目 Q-time 硬约束建模为 CMDP（§6.5）正属此线。
6. **在线前瞻 / MCTS**（Bertsekas 1997 rollout；He 2025；Saqlain 2023；**DyRo-MCTS Chen 2025**）——本项目 VC-MCTS（§7）是 DyRo-MCTS 的原理性推广。
7. **仿真优化 / R&S**（OCBA Chen 2000；indifference-zone）与**风险度量调度**（CVaR、分布鲁棒）——机制 3（§7.13）把这两线接进 MCTS 评估层。

> **严谨界定（避免夸大）**：向量奖励 + 多头 critic 是从多目标 RL 借鉴的成熟做法；双注意力 SAS 与主流对齐而非领先；PPO-Lagrangian / CMDP 是 Safe RL 成熟方法（非首创）；OCBA / CVaR 本身是仿真优化与风险度量的标准工具。本项目真正的差异点集中在 **可见性分区的约束式前瞻搜索（VC-MCTS）**：(i) 窗内可见到达 + 显式 reserve(m,h) 边取代钝鲁棒性代理；(ii) Q-time 报废硬约束作机会约束准入嵌进树（DyRo 纯软目标）；(iii) **能力感知对冲水位 ρ_pc**：二部匹配裕量取代空闲分布标量，含退化链与杠杆刻画（§7.12）；(iv) **风险鲁棒预留评估**：CRN 配对场景 + OCBA 预算 + CVaR 风险字典序（§7.13）。rollout/MCTS/OCBA/CVaR 本身成熟，创新在**组合方式与半导体 Q-time/异型能力约束的对接**。

### 9.3 可直接借鉴的点

1. **HGNN 增强状态表示**（异型柔性下推荐，后期增强）——HGNN 与双注意力互补：HGNN 在状态表示层编码高阶匹配/竞争关系，双注意力在决策层做候选选择。

## 10. 实验与实证

### 10.1 压力实例与 late_hi 实例

**pressure 实例**（`build_pressure_test_encoder`，50 Lot × 10 机台）：阶段间 Q-time 上限 `qtime_limit=3.0`（(1,2)、(2,3) 链），错峰 Poisson 到达 `arrival_mean_gap=0.6`。两个旋钮调区分度：`qtime_limit` 越小越紧（腔体争用 → 派工顺序影响 Q-time），`arrival_mean_gap` 越大利用率余量越多（wait-vs-dispatch 有意义）。

> **关键修复**：此实例此前从未设 `q_time_limits` ⇒ `compute_q_time_violation` 恒为 0 ⇒ Q-time 指标、`r_qtime` 通道、PPO-Lagrangian（`mean_violation` 恒 0 故 λ 永不启动）**全部静默失效**。补齐阶段间 Q-time 后三者同时复活。`small`（4-lot）实例**不区分策略**（所有规则 0 违规 / 利用率相同）。

**late_hi 实例**（`build_late_hi_encoder`）：高优先级 Lot 与到达时间高度正相关（corr ≈ 0.97）——这是预留/VC-MCTS 的 **go/no-go 实例**，因为只有"高优先级晚到"时，为其预留机器才有杠杆。

### 10.2 阶段 5a：规则预留与 priority 目标修正（VC-MCTS 的动机）

旧 `priority_violation`（全体按开工时间排序的逆序总量 Σ max(0, p_j − p_i)）对在线到达**病态**：晚到的高优先级 lot 必然晚开工，所有早到低优先级 lot 都计入其逆序——而 lot 不可能在到达前开工，这部分逆序物理不可避免。实测 strict 基线：可避免部分仅 220.8（**11.6%**），不可避免 1674.7（**88.4%**）。靠预留把该指标砸到 219 是用大面积空转买下不可避免部分 = gaming。故 §7.4 改用 O2（主）/ O1 可避免逆序（展示），旧量降级为参考。

**规则版预留结果**（pressure，FIFO，3 seed；除 util 外越小越好）：

| 配置 | O2 ↓ | O1 可避免逆序 ↓ | util ↑ | Q-time ↓ | 旧 priority_violation（参考）|
|---|---:|---:|---:|---:|---:|
| soft，无预留 | **862.5** | 220.3 | 0.844 | 10.7 | 1895.5 |
| soft，预留 w4 | 1888.8 (+119%) | 664.8 | 0.759 | 20.7 | 1846 |
| strict，无预留 | 874.6 | 220.3 | 0.830 | 8.7 | 1895.5 |
| strict，预留 w2 | 880.1 (+0.6%) | 243.1 | 0.827 | 19.3 | 1536 |
| strict，预留 w4 | 975.3 (+11.5%) | **11.7 (−95%)** | 0.748 | 24.0 | 219 |

**结论**：(1) 在真正要优化的 **O2** 上，钝规则预留**处处是负**（soft +119% 灾难、strict_w4 +11.5%）——靠空转重排把所有 lot 的等待拉长，得不偿失；这说明 O2 正确区分了"有用重排"与"浪费空转"。(2) 但 **O1** 显示 strict_w4 把可避免逆序 220→12（**−95%**），**优先级杠杆真实存在、可抓**，只是规则版抓得太糙（全有或全无地按阈值预留）。(3) 故规则版证伪了"钝预留改善正确目标"，同时为 §7 的 VC-MCTS 立下动机：需**外科手术式选择性预留**（仅当 高优先级少等×高权重 > 他人多等×低权重 才留），规则穿不过这针眼，VC-MCTS 靠对 reserve / dispatch 边的 rollout 精确算出 O2 之差来做。机制 2 的杠杆刻画（§7.12.2 性质 2）进一步把"何时有针眼"形式化。

### 10.3 阶段 6：oracle go/no-go（GREEN）

oracle = 信息完整（含窗外真实到达）+ 大 N_iter 的同一套 VC-MCTS，离线在 `late_hi` 上跑出 O2 下界。在 `late_hi`（corr≈0.97）上确认选择性预留**显著优于无预留基线（SAS-only / FIFO）**：Q-time 20→2、util 仅 −3%、O2 明显改善——这是开工 go/no-go 的唯一闸门，**已 GREEN**，解锁阶段 7。

> **跨实例口径提醒**：§10.2 的 O2≈862 是 **pressure** 实例上钝规则预留打不穿的地板，仅用于论证"需外科手术式选择性预留"；late_hi 与 pressure 是两个不同实例，O2 绝对值**不可直接横比**。本节 go/no-go 只比 late_hi 自身的「有预留 vs 无预留」。

实现优先级纪律：先证明预留有上界收益，再投入搜索调参；红灯就回去改实例生成器（§13.8），而不是加搜索深度。oracle 的进一步用法（最优性间隙曲线、相变扫描）见 §12.2。

## 13. VC-MCTS 一致性自查

**13.1 解决的原问题。** 上一版学习式 RMA 修掉了无目标 hold 与奖励重复计数，但未根除 greedy under-hold 与类别不平衡（学习范式结构性顽疾）。本版用 VC-MCTS 把"要不要为某未来高优先级 Lot 留这台机器"从"学一个概率"改为"模拟 reserve 与 dispatch 两条分支、按字典序比一比"，在结构上消除三个失败模式（§7.1）。

**13.2 与字典序结构是否冲突。** 不冲突且更直接：Q-time 仍是第 0 层硬约束（机制 1 准入层硬剪枝 + 字典序 qtime-first，§7.4/7.6），priority 第 1 层强偏好（reserve 边 + O2，机制 2 的 ρ_pc 已深度设计为匹配对冲水位 §7.12、待落地），utilization 唯一可让步软目标进价值估计。

**13.3 与 SAS 职责是否混淆。** 不混淆，已由 `dispatch_delegate.py` 落地：VC-MCTS 负责"是否预留/为谁/是否等待"，SAS（或 rule）负责"当前机器派哪个 `(l,p)`"；`topk`/`rule`/`sas` 三模式（§7.3；发表口径 §9.4.1）。

**13.4 ROP 是否过苛。** 仍宽触发、角色更纯——只做算力闸门（硬规则保留必要条件，S_res 只排序截断 TopB），是否真的预留交给树搜索（§4.3、§7.3）。

**13.5 分支空间是否过大。** 有风险但已控：宽触发 ROP + TopB reserve + no_op + 单个 delegate_dispatch，并强制 iteration_count = max(|R_root|, n_iter) 保证小预算下每条边至少 rollout 一次；机制 3 落地后由 OCBA 进一步把预算投向最可能翻转排序的边（§7.13.3）。

**13.6 是否仍有奖励重复计数风险。** 不再有——预留不训练、无奖励通道，reserve 分支的反事实收益由 rollout 直接精确模拟（reserve 分支尊重 ledger、dispatch 分支不尊重，两条 O2 之差即收益，§7.9）。命门改为"rollout 内部与树策略同源"——机制 3 落地后同源纪律扩为三条（qtime-safe、ledger、CRN 场景流，§7.9/§7.13.2）。

**13.7 是否可落地。** 可落地，成本从训练期搬到推理期（每决策点 clone driver/ledger 跑若干 rollout）。已通过 `rollout_max_steps`/`max_decisions`/trace summary/`--workers` 并行/轻预算参数把 late_hi 完整验证控制在可运行范围。缓解方向：Critic 叶子估值（§7.8）、子树复用、收紧 ROP/TopB、SAS delegate 减少 dispatch 分支、**OCBA 最优预算分配（§7.13.3，把算力痛点转为贡献）**。

**13.8 仍需实验验证的边界。** 预留收益依赖实例是否存在"晚到高优先级 + 兼容机器稀缺 + 当前派工挤占未来"的结构。`late_hi` 已证明存在杠杆（§10.4）；该结构条件已由机制 2 的杠杆刻画形式化（§7.12.2 性质 2），并将由 §12.2 的相变扫描系统验证。若某实例中 reserve 边长期输给 dispatch，应先检查该实例是否真有预留杠杆（oracle + Δρ_pc 验证），而非盲目堆搜索深度。

## 14. 总结

把当前代码压缩成一句话：

```text
lower layer (accurate scheduling & timing)
    → SAS (current dispatch selection)
        → VC-MCTS (counterfactual reservation planning).
```

- **下层**（`lower_layer_estimator.py` / `lower_layer_scheduler.py`）：共享 `schedule_deterministic` 核心的精确排程与估时，所有时间判断的单一事实来源（流水线①）；
- **SAS**（`phase2_*` PPO）：qtime-safe + priority-filtered 候选池中的当前派工选择，单头 / 多头两条路径 + PPO-Lagrangian（流水线②③⑤）；
- **VC-MCTS**（`vc_mcts_planner.py`）：在 clone 出的 driver/ledger 上分别模拟 reserve / dispatch / no_op 后果的反事实预留规划（流水线④，机制 1 已落地，机制 2/3 深度设计 §7.12/§7.13）。

本项目的核心价值不在于再造一个"会 hold 的网络"，而在把 hold 变成**可验证、可回放、可比较的搜索分支**——每个数都是仿真出来的、可复算的。当前已从"能跑通"走到 `late_hi` deterministic 与 process-noise smoke 通过完整轻预算验证；从"已跑通"到"结论可信"，剩下的是 §12 的统计协议、DyRo 基线 head-to-head、oracle 间隙与相变扫描，以及机制 2/3 按深度设计落地。

> **一句话定位（论文版）**：在能力异质、Q-time 硬约束与双重不确定性并存的 FAB 机台组调度中，提出**可见性感知的风险鲁棒在线预留规划器 VC-MCTS**：以二部匹配裕量 ρ_pc 将"为可见高优先级到达定向预留"形式化为能力瓶颈维度上的**对冲水位**（机制 2，§7.12）；以 CRN 配对场景、OCBA 最优预算分配与 CVaR 风险字典序完成对冲的**定价与验收**（机制 3，§7.13）；以机会约束准入将 Q-time 报废硬约束嵌入搜索树（机制 1）。下层以固定规则估算批处理完成时间分布、上层 SAS 以多头 PPO 学习当下派工，整体构成「量化 → 约束化 → 学习 → 对冲 → 兜底」的字典序消解流水线（§2.0）；当 W = 0、能力同质、加工确定时退化为 DyRo-MCTS，且其适用边界由可计算的结构条件（Δρ_pc 杠杆刻画 + 相变扫描）刻画而非事后观察。
