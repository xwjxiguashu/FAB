# 面向半导体 FAB 机台组调度问题的项目报告（完善版）

> 本版在初版《报告方案》基础上做了系统性完善，核心变化集中在六处：
> （1）补齐**到达模型与有限前瞻机制**，把「黑盒随机到达 + 可见即将到达」这一真实设定写进 MDP；
> （2）将 qtime/拖期、priority 从**奖励信号**升级为**机制约束**（机会约束 mask + priority 候选池过滤），确立「Q-time/拖期 > priority > 利用率」的字典序结构（R1：拖期=Q-time 违规）；
> （3）显式处理**加工时间不确定性**（纯工艺噪声），将 Q-time 改写为**机会约束**、safety margin 改为 z_ε·σ 自适应；
> （4）明确**两层架构**：下层固定规则估算工件内部批处理的完成时间分布，上层 RL 做派工/时机决策（1.5）；
> （5）将奖励由**加权标量**改为**向量**，Critic 由**单头**改为**多头**，GAE **逐目标独立计算**；
> （6）新增「**与主流方法对比及可借鉴点**」一章，并给出基准与验证建议。

## 0. 修订说明：相对初版的关键变化

| 维度 | 初版做法 | 完善版做法 | 解决的问题 |
|------|----------|-----------|-----------|
| qtime/拖期 | shaping 软惩罚 + 特征 | **机会约束 mask**（`deadline−μ<z_ε·σ`，违规概率≤ε）+ 残差通道（固定权重或自适应 λ） | R1：拖期=qtime 报废，硬约束不能靠"劝" |
| priority | shaping 软惩罚（-0.02） | **候选池过滤**（qtime-safe 集合内）+ DDT 资源预留 | 弱信号撑不起"高优先级优先" |
| 即时奖励 | 执行 + 质量混在一起 | **只剩执行合法性** | 消除成功奖励与质量信号抵消 |
| 奖励形态 | 加权标量 | **向量**，各通道独立 | 分量不丢、权重可解释 |
| Critic | 单头 V(s) | **多头** V_k(s) | 消除目标间梯度干扰 |
| 优势估计 | 单标量 GAE | **逐目标 GAE + 归一化加权** | 权重作用在无量纲量上，稳定 |

## 1. 算法升级总体框架

> **以资源日历为环境，以候选池 + 约束 mask 控制动作空间，以多头 PPO 训练可学习策略，并逐步升级为双智能体双注意力 DDA-PPO。**

### 1.1 总体流程

```text
事件触发或当前时刻推进
      ↓
读取资源日历状态 s_t（含前瞻窗内即将到达 Lot）
      ↓
按固定顺序选出当前决策的空闲 Machine m（多台空闲时逐台，见 1.3）
      ↓
DDT 决策时间智能体（针对 m）：立即派工 / 等待 / 为即将到达的高优先级 Lot 预留
      ↓（若 dispatch_now）
生成 Machine m 的结构可行候选动作池 A_t^m
      ↓
① qtime-safe mask → ② priority 过滤 → ③ Top-K/打分 → ④ padding/mask
      ↓
SAS Actor 在固定长度候选池中选择 action_index
      ↓
环境映射为真实动作 (lot, machine, ppid)，资源日历 dry-run / commit
      ↓
计算向量 reward 和下一状态
      ↓
PPO 用多头 Critic + 逐目标 GAE 联合更新
```

> 上图采用 1.3 的默认「逐台取法」（先选机台 m，DDT 再对 m 决策）。若选「全局取法」，则将"选机台 m"移到 DDT 之后；二者择一，全文据此统一。

各模块职责：

```text
下层估时器：算 (μ_finish,σ_finish)+占用区间，供 mask/打分/日历登记用（1.5，确定性启发式，非RL）
前瞻窗：让 DDT 能看见即将到达的 Lot，从而为高优先级 Lot 预留资源
候选池：控制动作规模
qtime-safe mask：硬约束，屏蔽会使未注定违规的可见 Lot 踩穿 qtime 的动作（详见 3.2）
priority 过滤：强偏好，在 qtime-safe 集合内优先高优先级 Lot
Top-K 打分：软目标启发式，仅用于排序与补足
Actor：学习候选动作选择概率
多头 Critic：分目标估计状态价值
资源日历：执行可行性校验与状态转移
向量 Reward：执行合法性 + 软目标，分通道提供训练信号
PPO：逐目标 GAE 更新策略与多个价值头
```

### 1.2 MDP 形式化定义

环境状态 `s_t` 在初版基础上**补充两类信息**（加粗为新增）：

```text
s_t = {
  当前时刻 t_now,
  未完成 Lot 集合：arrival / due_date / priority / wafer_count,
  **每个已可见 Lot 的 qtime 窗口截止时刻 qtime_deadline(l)**,
  **前瞻窗 [t_now, t_now + W_lookahead] 内即将到达的 Lot 及其 priority / qtime / ETA**,
  空闲 Machine 集合 M_idle,
  每台 Machine 的 Lot-level calendar,
  每个 (machine, chamber, side) 的 wafer-level calendar,
  已提交的 Lot / Wafer 操作,
  当前窗口内已完成 / 未完成 Lot 状态
}
```

网络输入仍为特征化摘要 `z_t = featurize(s_t)`，新增特征：

```text
z_t 新增:
  每个候选 Lot 的剩余 qtime（qtime_deadline - 预计开始时刻）
  前瞻窗内即将到达的高优先级 Lot 数量、最紧 qtime、最早 ETA
  "若现在排满，前瞻窗内可能无法服务的高优先级 Lot 估计数"
```

完整日历保留在环境中，网络只用可计算摘要。

### 1.3 局部动作空间

单台 Machine `m` 空闲时，动作为 `a_t^m = (lot, ppid)`，`machine = m` 由状态给定。多台同时空闲时按固定顺序（最早空闲 / 候选数 / 受限程度）逐台决策，每台决策后立即更新日历与 mask，避免组合动作空间。

> **决策粒度（需在实现时定死）**：在完整 DDA 中，DDT 的"是否派工/预留"与上述"逐台决策"如何嵌套有两种取法，本报告默认取后者：(i) 全局取法——DDT 全局决一次 dispatch_now，随后 SAS 对所有空闲机台逐台派工；(ii) 逐台取法（默认）——每选出一台空闲机台 m 作为一个决策点，DDT 先对 m 决"派/等/预留"，若 dispatch_now 再由 SAS 对 m 选动作。逐台取法语义更一致（预留本就是针对"某台机台是否留给未来"的决策），但决策点更多、DDT 训练样本更密；实现时二选一并在状态中标明"当前决策针对哪台机台"。

> **设计边界（须诚实声明）**：本框架中**机器分配本身不由策略学习**——DDT 学"何时"、SAS 学"派哪个 (lot, ppid)"，但"用哪台机器"由固定规则按顺序给定。机器分配是经由"逐台顺序决策"**隐式涌现**的：每台机器的候选池只含它能加工的 lot，SAS 为每台依次选择，最终分配由这一序列决定。其代价是分配质量**对固定机器排序敏感**，且非全局最优。这是相对"同时学作业+机器分配"的主流 FJSP 工作（如多指针网络、HGNN 的作业-机器双输出）的一个范围限制。若后续要消除该限制，可将机器选择也纳入动作（扩为 `(machine, lot, ppid)` 候选）或增设机器选择头，但会扩大动作空间、增加训练难度——建议作为后续工作，而非基础版目标。

### 1.4 三层字典序约束结构（新增）

本项目的目标不是"几个目标加权"，而是**严格分先后**的三层结构。**关键前提（来自问题定义）：due date 由剩余 Q-time 设定，工件一旦过 due date 即报废，故"拖期"与"Q-time 违规"是同一个硬事件**，目标严格为 0，归入第 0 层：

```text
第 0 层（硬）：Q-time / 拖期不可违（同一事件）  —— 机会约束 mask（窗内）+ cost/λ 兜底（窗外）
第 1 层（强）：高优先级 Lot 优先              —— 候选池过滤 + DDT 预留，但不得踩穿第 0 层
第 2 层（软）：机台利用率（进度为辅助代理）   —— 奖励通道，唯一可让步的目标
```

执行顺序「先 mask → 再过滤 → 再打分」天然实现字典序：会害 Q-time/拖期的高优先级动作在第 0 层被屏蔽，第 1 层无机会强推，**Q-time 永远优先于 priority**。

> 由此，真正"可互相权衡的软目标"只有利用率一项（进度是完成度代理，非独立目标）。这意味着本项目本质是「**单软目标 + 两层约束（Q-time/拖期硬、优先级强）**」，而非多目标权衡问题——下文奖励与 Critic 的多通道设计据此调整（4.5、4.7），不作"多目标帕累托"包装。

### 1.5 两层决策架构与下层启发式估时器（新增）

本项目的决策天然分两层，**只有上层用 RL，下层用固定规则（非 RL、非随机搜索）**。明确区分二者是整个方案能成立的前提——上层反复用到的 `finish_time`、`estimated_earliest_start` 等量，正是下层产出的。

> 术语澄清：下层的**排程逻辑**是固定规则（确定性的组批 + list scheduling），但它作用在**随机的加工时间**上（2.4），故其**输出是完成时间的分布** `(μ_finish, σ_finish)`，而非单一值。"固定规则"指算法不学习、不随机搜索；"分布输出"来自输入（各阶段加工时间）的随机性。二者不矛盾。

```text
上层（RL：DDT + SAS）：决定 哪个工件 / 哪台 machine / 哪个 ppid / 什么时机
  —— 存在真实取舍（利用率 vs 优先级 vs 不违 Q-time），故用 RL 学
下层（启发式估时器）：一个工件被派到 (machine, ppid) 后，其一批 wafer 在 step 内部
  如何组批、每批进哪个 chamber 实例、各批时序
  —— 无取舍（wafer 对称、step 内 chamber 间无 Q-time、目标单一=尽早完成），故用规则算
```

**下层问题结构（来自问题定义）。** 单个工件的一个 step：ppid 锁定 chamber 类型的经过顺序（A→B→C，工艺路径不可变）；每个 chamber 类型有多个物理实例可选（柔性）；每个 chamber 内多个 side 同进同出（批处理机，一次成批、整批同时开始结束）；工件不可拆批（**指作为一个派工单元整体派出、其 wafer 不与其他工件混入同一炉**，而非"机器被单块占用"），wafer 间无顺序约束。当 wafer 数 > side 容量时，工件内部分成多个**子批**，子批属同一工件、彼此可在阶段间流水重叠（故机器各阶段在重叠时段被不同子批占用，并非单块占用）。这是一个**带批处理机的柔性流水车间（hybrid flow shop with batching）**。

**下层启发式算法。**

```text
1. 组批: 满批优先, N 片 wafer → ⌈N/side容量⌉ 个子批
   （前提: 批处理时间与装载片数无关——批处理炉通常如此；若相关需另算）
2. 流水排程 (list scheduling):
   for 每个子批 b（固定发批顺序，如 FIFO）:
     for 每个阶段 s（按 ppid 顺序）:
       ready   = b 在上一阶段的完成时间          # 子批自身前序约束
       free    = min over 阶段s的实例(实例空闲时间) # 选最早空闲实例
       start[b,s] = max(ready, free);  end[b,s] = start[b,s] + 批处理时间(s)
   makespan = max_b end[b, 末阶段]
3. (可选) 用 NEH 优化发批顺序，子批少时收益有限
```
makespan 由**瓶颈阶段**（有效吞吐最低 = 实例数×1/批时间 最小的阶段）决定，组批服务于"喂饱瓶颈"。问题规模小、无取舍，贪心已接近最优，**不用 RL、不用重型优化（MILP/GA）**。

**下层交付物（上层的地基）。** 在加工时间不确定性（2.4）下，下层输出的不是单值而是分布刻画。**计算方式**：对固定的组批+排程规则，用各阶段加工时间的 (μ_s, σ_s) 做**蒙特卡洛采样**——多次采样各阶段实际时间、每次跑一遍 list scheduling、对得到的 makespan 取均值与标准差，即得 `(μ_finish, σ_finish)`（这天然解决下面易错点(1)的 E[max] 偏置）。规模小时蒙特卡洛开销可接受；若仍嫌慢，可用均值路径算 μ 再加偏置修正、用 `sqrt(Σσ_s²)` 近似 σ。

```text
estimate(O, machine m, ppid p) →
  (μ_finish, σ_finish)     # 完成时间均值与标准差；喂上层 estimated_finish 与机会约束(2.4.3)
  per_instance_occupancy   # 各 chamber 实例占用区间（按 μ 登记）；喂资源日历做冲突检测
  bottleneck_stage         # 瓶颈阶段；可作上层候选池打分特征
```
> 资源日历的不确定性处理：登记占用时用**期望区间（按 μ）**做冲突检测与后续 lot 的估时；待该工件实际加工完成、真实时间揭晓后，再用**实际值回填**日历。即"规划用 μ、执行后用实际值校正"。
> 两个易错点（2.4.2 已述，此处落到下层）：(1) makespan 常为"多并行子批取最晚完成"，`E[max] > max(E[·])`，**仅用均值路径会系统性低估** μ_finish，需上偏修正或（推荐）蒙特卡洛；(2) 沿关键路径累加独立噪声，`σ_finish ≈ sqrt(Σσ_s²)`。
> 衡量下层做得对的标准是**"估得准"而非"makespan 全局最优"**——一个估时偏 10% 的最优调度器，对上层 Q-time 判断的伤害大于一个估时精确的贪心调度器。

## 2. 到达模型与前瞻机制（新增）

这是本项目区别于通用 FJSP 的关键设定，也是 DDT 智能体存在的根本理由。

### 2.1 黑盒到达 + 有限前瞻窗

工件**随机到达，到达率未知**（不能假设泊松或任何分布），但系统具备**有限前瞻能力**：一个 Lot 在真正进入待派工池之前，会在前瞻窗 `W_lookahead` 内变得**可见**（例如已从上游工序 release、在途、ETA 已知），可见信息包括它的 **优先级** 与 **qtime 窗口**。前瞻窗之外仍为完全黑盒。

### 2.2 违规来源的两分

qtime 违规据此分为两类，可预判性完全不同：

```text
(a) 已可见 Lot 之间的相互挤占 —— 可预判
    已到达 + 前瞻窗内即将到达的 Lot，其 qtime_deadline 已知；
    某动作是否害它们踩穿窗口，当前时刻可按完成时间分布估算违规概率 → 机会约束 mask 屏蔽。
    对被调度 Lot 自身可保证违规概率≤ε；对其他可见 Lot 为显著降低（其真实
    开始时刻仍依赖后续决策）；已注定违规的 Lot 不作屏蔽依据（见 3.2）。

(b) 前瞻窗之外到达的 Lot 抢占资源 —— 不可预判
    现在把机器排满，窗外突然到来紧急 Lot 无机器可用 → 违规。
    信息上无法在当前时刻完全避免 → 用残差通道（固定权重或自适应 λ）做统计性兜底，
    并由 DDT 学习"留余量"。
```

随机到达只削弱 (b) 的可预判性，**(a) 照常可 mask**，且通常是违规的大头。

### 2.3 高优先级资源预留（DDT 的核心职责）

因为前瞻窗内能看见即将到达的高优先级 Lot，"为它预留资源"从**不可做**变为**可学、可预判**：

```text
若前瞻窗内存在高优先级 Lot L_hp，其 ETA 临近且 qtime 紧：
  DDT 可选择 wait_for_reservation：
    暂不把关键 Machine 排给低优先级 Lot，
    留出空档，等 L_hp 到达后立即服务。
```

这正是 DDT `wait_*` 动作的价值所在：机台空闲≠立即派工最优。预留决策落在前瞻窗内时**可预判**；窗外则退化为残差通道学到的"统计性留余量"。

### 2.4 加工时间不确定性与 Q-time 机会约束（新增）

> 本节确立一个贯穿全方案的设定：加工时间是**随机的**（来源：纯工艺噪声），估计时只知道分布、实际值加工完才揭晓。这把 Q-time 从"确定性硬约束"改写为**机会约束**，并让 safety margin 获得明确的概率含义。

**2.4.1 噪声模型。** 每个 (batch, stage) 的实际加工时间为标称均值加独立无偏噪声：

```text
p_actual(b,s) = μ(b,s) + ε,   ε ~ (mean 0, var σ²),  各 (b,s) 独立
设定: 噪声为"来源一"——纯工艺随机抖动，无偏、独立、估计时不可知、做完才揭晓。
      μ 与 σ 均已知（作为工艺参数），二者都进入 agent 状态特征。
```
> 此设定排除了"系统性差异（隐藏变量）"与"长尾故障"——前者应补状态特征而非当噪声，后者需单独的故障建模；本方案在纯噪声假设下成立，若实测数据呈多峰/长尾需另行处理。

**2.4.2 对完成时间的传导。** 下层估时器（见 4.x 与子问题说明）输出的不再是单一 `finish_time`，而是其**分布刻画** `(μ_finish, σ_finish)`：

```text
makespan 沿关键路径累加各阶段噪声 → 方差按路径长度增长（σ_finish ≈ sqrt(Σ σ_s²)）
但独立噪声部分抵消 → 变异系数(σ/μ)小于单段
注意 max 操作的偏置: makespan 常为"多并行批次取最晚完成"，E[max] > max(E[·])
  ⇒ 仅用各阶段均值算 makespan 会系统性低估完成时间；
     需对 μ_finish 做上偏修正，或对噪声做蒙特卡洛采样估期望。
```

**2.4.3 Q-time 改写为机会约束。** 因噪声无偏独立，`finish_time` 近似正态（CLT），故"违规概率"有闭式：

```text
P(违规) = P(finish_time > deadline) = Φ( (μ_finish − deadline) / σ_finish )
机会约束: 屏蔽 P(违规) > ε 的动作
  ⇔ 屏蔽满足  deadline − μ_finish < z_ε · σ_finish  的动作
其中 z_ε 为容忍违规概率 ε 对应的分位数（如 ε=2% → z_ε≈2.05）
```

**2.4.4 safety margin 获得概率含义（顺带成为一个理论贡献）。** 由 2.4.3，原先拍脑袋的固定 `qtime_safety_margin` 被替换为：

```text
qtime_safety_margin(l) = z_ε · σ_finish(l)     # 不再是常数
```
即 margin = "容忍违规概率"×"该工件完成时间波动"。波动大的工件自动留厚垫、波动小的留薄垫——这正是**自适应安全裕度**，且有明确的概率解释。`z_ε`（或等价的 ε）成为唯一需要设定的、领域可解释的旋钮。

**2.4.5 必须诚实的结论：随机下无法绝对归零。** 正态噪声尾部无限长，无论 margin 多大，实际加工时间暴长导致违规的概率恒 > 0。因此全方案中关于 mask 的保证一律表述为**"违规概率 ≤ ε"**，而非"违规归零"。可控的是 ε（通过 z_ε / margin），不是绝对的 0。

**2.4.6 对训练与评价的要求。**

```text
训练: 环境每 step 采样实际加工时间 p_actual；agent 状态用 (μ, σ)，
      Q-time 判定与 reward 用采样出的实际值 —— 否则 agent 没见过波动，学不会留余量。
评价: Q-time 违规率在多次随机 rollout 上统计（报告期望 + 分布/分位数），而非单次确定性回放。
```

## 3. 约束与候选池：qtime 硬约束 + priority 强偏好

### 3.1 候选池四步流水线（顺序不可调换）

```text
A_t^m = {(l,p) | l 已到达且未完成, m 可加工 l, p 是 (l,m) 可用 PPID}   # 结构可行
      ↓ ① qtime-safe mask     —— 硬约束
      ↓ ② priority filter      —— 强偏好（qtime-safe 集合内）
      ↓ ③ CandidateScore TopK  —— 软目标启发式（仅排序/补足）
      ↓ ④ padding + 结构 mask  —— 对齐到 A_fixed^m
```

结构性过滤（④的一部分）仍包括：Lot 未到达/已完成、Machine 不支持 Recipe、`(lot,machine)` 无可用 PPID、PPID 步骤缺失、所需 Chamber/Side 不存在。

### 3.2 qtime-safe mask（窗内可预判部分）

```text
function qtime_safe_mask(candidate_actions, visible_lots, C_t, config):
    mask = ones(len(candidate_actions))
    for i, (lot, ppid) in enumerate(candidate_actions):
        C_sim = dry_run_commit(C_t, lot, machine_m, ppid)
        for l in visible_lots:                 # 已到达 + 前瞻窗内即将到达
            if is_doomed(l, C_t, config):       # 已注定违规(连等待也救不回)的 lot
                continue                        # 不作屏蔽依据，否则会"全屏蔽→死锁"
            # 机会约束（见 2.4.3）：用完成时间分布判断违规概率
            mu, sigma = estimated_next_finish(l, C_sim)   # 见下方"等待 lot 估时口径"
            margin = config.z_eps * sigma                  # 自适应 safety margin
            if qtime_deadline(l) - mu < margin:            # ⇔ P(违规) > ε
                mask[i] = 0
                break
    return mask          # 若全部被屏蔽 → 交回 DDT 执行 wait（不罚，见 4.5）
```

> **等待 lot 的估时口径（须明确，否则 `estimated_next_finish` 无定义）**：等待 lot `l` 尚未派工，其将来用哪台机器/ppid、何时开始都未定。本 mask 取**乐观估计**——在 `C_sim` 下，假设 `l` 能用它**最早可用的可行 (machine, ppid)** 立即开工，由下层估时器（1.5）算其 `(μ,σ)`。
> **该口径的正确语义（勿误读为"保守"）**：乐观估计回答的是"提交候选 i 之后，`l` 是否**还有救**"。仅当 i 使 `l` 连最乐观情况都注定违规（即 i 断了 `l` 的最后生路）时才屏蔽 i；若 `l` 仍可救，则不屏蔽，信任后续决策步骤及时派 `l`（同一 mask 在后续步骤会再次判定）。因此本 mask 屏蔽的是"会**当场断送**某等待 lot 的动作"，而非"任何可能让其变紧的动作"——这也是为何对其他 lot 只能"显著降低"违规、而非绝对保证（真实开始时刻依赖后续决策）。
> 若希望更强的保护，可改用**悲观估计**（按 `l` 较晚的可能开工时刻判定），mask 会更激进地屏蔽，但需配合 `is_doomed` 与"全屏蔽→交回 DDT"防止过度限制导致空池。乐观/悲观口径二选一并在实现中固定。

两点必须说清，否则逻辑不成立：

1. **必须排除"已注定违规"的 lot。** 若某 lot 即便本步等待也无法在窗内被服务（`qtime_deadline(l) - 最早可能开始(均值) < 0`），它会让**每个**候选动作都触发屏蔽，导致候选池全空→强制等待→既排不了任何工件、那个 lot 也照样违规（死锁）。因此 `is_doomed` 的 lot 不计入屏蔽依据；它的违规已不可避免，应直接计入指标、并让 DDT/调度继续推进，而不是拖垮整池。
2. **保证强度要诚实（随机版）。** 加工时间随机下不存在"违规归零"（2.4.5）。本 mask 保证的是：对**被调度 lot 自身**，违规概率 ≤ ε；对**其他可见 lot**，由于其真实开始时刻仍取决于后续决策，仅是**显著降低**可预判的违规概率。`z_ε` 越大越保守（违规概率越低、利用率越低）。

> 控制流补充：在完整 DDA 中，DDT 选了 `dispatch_now` 但本步候选池经 mask 后为空时，视为"本次派工不可行"，控制权交回 DDT 重新决策（等待/推进时间），不计执行失败惩罚。

### 3.3 Q-time/拖期约束残差：固定权重 vs 自适应拉格朗日

mask 已结构性压低违规（窗内、机会约束），但仍有残差违规：**窗外到达的工件、机会约束的尾部、以及 doomed lot**。这部分由 `V_qtime` 残差通道（4.3/4.5）提供学习信号，其在策略优势中以减项出现。该通道的**权重**有两种取法：

```text
(i) 固定权重 w_qtime（大值，体现硬约束优先）：A 中减去 w_qtime·norm(Â_qtime)
(ii) 自适应拉格朗日 λ_qtime（更原则化）：把"期望违规 ≤ budget"作为约束
     策略优势减去 λ_qtime·norm(Â_qtime)
     对偶上升: λ_qtime ← max(0, λ_qtime + η_λ·(E[qtime违规] − budget))
```

`use_qtime_lagrangian` 默认关闭（先用固定 `w_qtime`）；仅当固定权重难以把违规压到目标时启用，`η_λ` 远小于策略学习率以防震荡。**两种取法二选一，不并用**（同一残差通道，区别只在权重是常数还是自适应）。

### 3.4 priority 候选池过滤（强偏好）

```text
function priority_filter(safe_actions, config):
    if len(safe_actions) == 0: return safe_actions
    max_pri = max(priority(a.lot) for a in safe_actions)
    if config.priority_filter_mode == "strict":
        return [a for a in safe_actions
                if priority(a.lot) >= max_pri - config.priority_min_gap]
    else:  # "soft"：不删动作，仅在 CandidateScore 中大幅加权，保留探索
        return safe_actions
```

建议先 `soft`（保留探索，避免策略坍缩），稳定后视需要切 `strict`。

## 4. 基础方案：候选池资源日历 SAS-PPO

DDA-PPO 不一步到位。基础版只学"当前 Machine 选哪个 (Lot, PPID)"，由规则触发，暂不学"是否等待"。

### 4.1 候选动作池生成

见 3.1。`CandidateScore` 中**移除 priority 项**（已上移到候选池过滤）：

```text
CandidateScore(l,m,p) = due_date_urgency(l) + qtime_slack(l) + waiting_time(l)
                        - estimated_process_time_mean(l,m,p) - resource_conflict_risk(l,m,p)
```

`K_action` 训练初期取 30–50，中等 50–100，大规模 100–200。

### 4.2 可变动作维度处理（保留）

不构造固定全局动作空间，采用**候选池索引动作空间**：Actor 输出 `action_index ∈ {0,...,K_action-1}`，环境映射 `candidate_actions[action_index] = (lot, ppid)`。不同时刻候选数经 Top-K / padding / mask 对齐到 `K_action`，Actor 输出固定维 logits，用 masked softmax；候选池为空或全被 mask 时不执行 softmax，改由 DDT/wait 推进时间，避免分母为 0。

### 4.3 SAS Actor-Critic（多头 Critic）

Actor 先用 MLP，再升级为机台条件化注意力（候选动作编码 `x_i = [lot_feat, ppid_feat, machine_feat, calendar_feat]`，机台为 query 做多头注意力，masked softmax 输出）。

**Critic 改为多头**，共享候选 embedding 与注意力表示：

```text
g_t = Pooling(h_m, e_1,...,e_K, global_features)
V_exec(z_t)     = MLP_exec(g_t)        # 即时执行（密集）
V_qtime(z_t)    = MLP_qtime(g_t)       # Q-time/拖期违规残差（硬约束 cost，见 R1 说明）
V_util(z_t)     = MLP_util(g_t)        # 唯一软目标：利用率
V_progress(z_t) = MLP_progress(g_t)    # 利用率/完成度辅助代理
```
> R1 下通道说明：因拖期 = Q-time 违规（同一硬事件），**不再有独立的 V_tardy 头**——它与 V_qtime 合并为单一"约束残差"头，估计 mask 未能拦住、最终仍违规/报废的部分。多头在此并非为"多目标权衡"，而是为分离三类**尺度与时间尺度迥异**的信号：执行（密集即时）、约束残差（稀疏、报废相关）、利用率（终局）。

### 4.4 资源日历执行与失败处理（保留）

`TryInsert(lot, machine, ppid, C_t)` 校验 Machine 可插入性、PPID 步骤完整性、Chamber/Side 可用时间、插入后时间表冲突。

**训练阶段轨迹一致性（关键）**：若插入失败，**不得**把执行动作替换为候选池中下一个动作——PPO 需要"采样动作与其 logπ"一致，替换会使概率比与梯度估计偏离真实策略。处理方式：保留原采样动作、返回失败惩罚、不提交任何日历修改、把环境推进到明确的下一状态。

推理阶段无梯度更新，可工程 fallback：按概率从高到低尝试，提交第一个可插入动作；全失败则 wait/no-op。

### 4.5 奖励通道设计（R1：拖期=Q-time）

即时 reward **只回答**"这一步动作合不合法、可不可执行"。质量/约束信号下沉：

**通道 EXEC（即时、密集）**
```text
r_exec = +0.20 if insertion_success
         -0.40 if insertion_failed
         -0.50 if mask_invalid   # 防御性分支，正常不应触发，见下
          0.00 otherwise
```
成功时**不再叠加** 约束/利用率分量。

> 关于 mask_invalid：在 masked softmax 正确实现下（mask=0 的位置概率恒为 0），训练时**采样不到** padding / 无效动作，此分支理论上不可达。因此 -0.50 仅作为**防御性断言**（捕捉实现 bug、数值下溢导致的异常采样），不应被当作"让 Actor 学会避开无效动作"的训练信号——避开无效动作由 mask 本身保证，不靠惩罚。

**wait 的处理（SAS 阶段）**
```text
SAS 是规则触发（池非空才激活），且在完整系统中"是否等待"由 DDT 决策，
故 SAS 自身从不主动选择 wait。SAS 唯一会遇到的 wait 是：
  候选池为空 / 经 qtime mask 后全被屏蔽 → 由环境/DDT 接管推进时间，r_exec = 0.0（不罚）
"主动等待"的时间成本属于 DDT 的 r_wait_cost（见 6.2.2），不在 SAS 奖励内。
```
> 修正：原先在 SAS 列出"主动 wait -0.02"与职责划分矛盾——SAS 不拥有 wait 动作。主动等待成本统一归 DDT。

**约束残差 + 软目标（终局向量，仅 episode 结束非零）**
```text
r_qtime    = -normalize(qtime_violation_count / num_lots)   # = 拖期违规（同一事件），硬约束残差
r_util     = +normalize(avg_machine_utilization)            # 唯一软目标
r_progress = +normalize(completed_lots / num_lots)          # 利用率辅助代理，计数制
```
> R1 关键变化：**取消独立的 r_tardy 通道**——拖期即 Q-time 违规，合并为单一 `r_qtime` 约束残差通道。该通道度量"mask 没拦住、最终仍报废"的工件数，目标趋零。
> 三处保留的改进：(1) 不加权求和，各通道独立；(2) normalize 用 running mean/std，消除量纲耦合；(3) progress 计数制，消除"偏好长加工时间 Lot"的偏置。
> `r_qtime` 与 mask 的关系：mask 已结构性压低违规（窗内、机会约束），`r_qtime` 只对**残差**（窗外 + 机会约束尾部 + doomed lot）提供学习信号，二者度量对象不重叠，不构成双重计数。
> 若终局过稀疏需补 shaping，**必须用 potential-based 形式** `F = γΦ(s') − Φ(s)`，保证不改变最优策略。

### 4.6 RewardConfig 与 compute_sas_reward（向量）

```text

```
> 约束残差的两种权重方式（二选一，不并用，避免重复计数）：(i) **固定权重** `w_qtime`（大值，体现硬约束优先）；(ii) **自适应拉格朗日** `λ_qtime`（对偶上升，把"期望违规 ≤ budget"作为约束，见 3.3）。后者更原则化，对应 `use_qtime_lagrangian=True`，此时 `w_qtime` 由 `λ_qtime` 取代。

### 4.7 多头 Critic + 逐目标 GAE + PPO loss

```text
for k in {exec, tardy, util, progress, (qtime)}:        # 逐通道独立 GAE
    delta_t^k = r_t^k + γ·V_k(z_{t+1})·(1-done) - V_k(z_t)
    Â_t^k     = delta_t^k + γ·λ_gae·(1-done)·Â_{t+1}^k
    R_t^k     = Â_t^k + V_k(z_t)

for k in {exec, qtime, util, progress}:        # R1: 逐通道独立 GAE（无独立 tardy）
    delta_t^k = r_t^k + γ·V_k(z_{t+1})·(1-done) - V_k(z_t)
    Â_t^k     = delta_t^k + γ·λ_gae·(1-done)·Â_{t+1}^k
    R_t^k     = Â_t^k + V_k(z_t)

A_t = w_exec·norm(Â_exec) + w_util·norm(Â_util) + w_progress·norm(Â_progress)
    - w_qtime·norm(Â_qtime)        # Q-time/拖期残差为 cost，做减项；
                                    # 若 use_qtime_lagrangian，则 w_qtime → 自适应 λ_qtime

L_actor = - E_t[min(ρ_t·A_t, clip(ρ_t,1-ε,1+ε)·A_t)]
L_value = Σ_k c_k·MSE(V_k(z_t), R_t^k)
L_total = L_actor + L_value - c_e·H(π)
```

> 两点务必注意，否则多头会"看着干净、实则学不动"：
> 1. **终局通道信号稀疏。** qtime/util/progress 整局只在最后一步非零，其 value 头要靠 bootstrap 把终局值沿轨迹回传，长 episode 下信号弱、方差大。缓解：这些通道的 value 头用较大 `c_k` 或单独更长的训练；必要时加 **potential-based shaping**（`F=γΦ(s')−Φ(s)`，不改变最优策略）提供密集梯度；先确认 EXEC 通道（密集）稳定后再依赖终局通道。
> 2. **归一化在 batch 层面做。** `norm(Â_k)` 用整个 batch 的 mean/std，而非单步；EXEC 通道每步非零、终局通道多为 bootstrap 传播值，两者尺度不同，**逐通道**归一化后再加权才能让权重表达真实重要度。


### 4.9 终止条件与失败处理（保留 + 补充）

终止：所有 Lot 完成；`t_now` 超 horizon 且无可调度 Lot；候选池空且未来无 Lot 到达 / 无 Machine 释放；`total_wait_steps > max`；连续插入失败 > `max_failed_actions`；日历不可恢复错误。

失败处理：采样到 padding/mask=0 → 正常不可达（masked softmax 保证），若发生则按实现 bug 处理，r_exec=-0.50 仅作防御；候选池空/全屏蔽 → 交回 DDT 推进时间，r_exec=0.0（不罚）；dry-run 失败 → 保留 (s,a,logπ)，r_exec=-0.40，不提交；commit 中途失败 → rollback，r_exec=-0.40；连续失败过多 → 终止，标 infeasible。

### 4.10 最终评价指标

```text
F(S) = [ tardy_count, total_tardiness,        # R1: 即 Q-time 违规数/总量（同一事件），硬约束，趋零
         -average_machine_utilization,         # 唯一软目标
         priority_violation ]                  # 强偏好的违反度
```
> R1 说明：因 due date 由 Q-time 设定、过期即报废，`tardy_count/total_tardiness` 与 `q_time_violation_count/total_q_time_violation` 是**同一组量**，不再分别列出（原版重复，已合并）。利用率取负以统一为最小化；展示时以正向报告。
> 评估口径：`tardy(=Q-time违规)` 作为硬约束，主要看其是否被压到接近 0（及违规概率是否 ≤ ε）；`utilization` 是唯一真正可权衡的软目标。因可权衡软目标仅一项，**不做多目标帕累托/超体积的强主张**（见 7.4 的相应调整）。训练用小尺度向量 reward，评估用上述指标在多次随机 rollout 上的统计。

### 4.11 基础 SAS-PPO 算法流程

```text
Algorithm: Candidate-pool Calendar SAS-PPO (multi-head)
1. Observe s_t, extract z_t (含前瞻窗信息)
2. Select idle machine m by fixed rule
3. Build structurally feasible pool A_t^m
4. ① qtime_safe_mask → ② priority_filter → ③ TopK → ④ pad/mask = A_fixed^m
5. If pool empty or all masked: execute wait (forced), store transition, advance time
6. SAS computes π(a|z_t, A_fixed^m); sample action_index (train) / rank (infer)
7. If padding/mask=0: r_exec=-0.50, store, continue
8. TryInsert(lot,m,ppid)
9. If success: commit, compute reward_vector
10. If fail (train): keep sampled action, r_exec=-0.40, no commit, advance
11. If fail (infer): try next candidate / wait
12. Store transition (with reward_vector). Repeat until window done
13. Per-objective GAE; update Actor + multi-head Critic with clipped PPO
```

## 5. 创新方案：DDA-PPO 双智能体双注意力资源日历调度

> **DDA-PPO：Dual-agent Dual-attention Calendar-constrained PPO Dispatching（双智能体双注意力资源日历约束 PPO 派工算法）**

### 5.1 双智能体协同（DDT 增加预留职责）

```text
资源日历状态 + 前瞻窗
      ↓
DDT 决策时间智能体: dispatch_now / wait_next_lot / wait_next_machine /
                    wait_delta / wait_for_reservation(为即将到达高优先级 Lot 预留)
      ↓ (dispatch_now)
SAS 派工动作智能体: 在 qtime-safe + priority 过滤后的候选池中选 (Lot, PPID)
      ↓
资源日历 dry-run / commit → 向量 reward → PPO 联合更新
```

DDT 状态特征（在初版基础上加粗为新增）：当前时刻、空闲 Machine 数、**前瞻窗内即将到达 Lot 数 / 最高优先级 / 最紧 qtime / 最早 ETA**、当前最高 qtime 风险、最高 due 紧迫度、平均利用率、候选池是否为空、下一次到达/释放时间、连续 wait 次数。新增动作 `wait_for_reservation` 用于 2.3 的预留。

### 5.1.1 未来窗工件的 Transformer 编码（创新点）

> **定位**：本创新点的实质不是"使用了 Transformer"，而是**用 set-attention 把有限前瞻窗内的未来工件（变长、无序、需建模相互竞争）编码进 DDT 决策时机智能体，服务于主动资源预留**。主流 RL 调度多为"事件触发即派工"，很少显式建模"可见但尚未到达"的工件，更少用其驱动预留决策——这是本项目相对主流的差异点。

**问题动机。** DDT 要做"是否为即将到达的高优先级 Lot 预留资源"，必须理解前瞻窗里那一组工件。该集合**变长**（每刻 10–20 个、有波动）、**无序**（到达先后不构成有意义的序列位置）、**需看整体竞争关系**（谁与谁抢同类机台、谁的 qtime 更紧）。把它手工聚合成几个标量（窗内数量 / 最紧 qtime / 最高优先级）会损失这些关系；用注意力编码集合则天然置换不变、可建模两两关系。

**编码结构（版本 A：self-attention，建议先实现）。**

```text
未来窗工件 {f_1, ..., f_n}, f_i = [priority, remaining_qtime, ETA,
                                   wafer_count, recipe_type, required_chamber_side ...]
   → Linear embedding (dim 64–128)
   → Transformer Encoder × (1–2 层, 2–4 heads, 无位置编码)   # 集合非序列，不加位置编码
   → Attention Pooling (可学习 query 或 [CLS] 汇总 token)     # 不用 mean，避免稀释少数紧急工件
   → future_ctx (固定长度向量)
   → 拼入 DDT 状态 → DDT 的 PPO 网络（Actor + 多头 Critic）
```

**编码结构（版本 B：cross-attention，进阶对比）。** 预留决策本质是"未来工件 × 当前资源"的匹配：是否为某即将到达的高优先级 Lot 留机台，取决于它所需机台类型当前与未来的占用情况。让未来工件去 attend 当前机台/资源状态，可直接建模该匹配关系：

```text
未来窗工件 {f_i} → self-attention → F          # query 来源
当前机台/资源状态 {m_j} → 编码 → M             # key / value 来源
future_ctx = CrossAttention(query=F, key=M, value=M) → Attention Pooling
   → 拼入 DDT 状态
```

**关键实现约束。**

1. **无位置编码**：未来工件是集合不是序列，加位置编码会引入"到达顺序有意义"的错误归纳偏置。
2. **变长 + padding mask**：pad 到固定上限（如 32），attention 中屏蔽 padding 位；窗内为空时 `future_ctx` 置零或用可学习 empty token。
3. **超上限截断**：偶发工件数超上限时，按"对预留最相关"排序（剩余 qtime 升序 + 优先级）保留前 N，与 SAS 候选池 Top-K 同理。
4. **与 SAS 注意力分离**：SAS 双注意力编码**当前候选池**（可立即派工）→ 决定"派哪个"；本编码器处理**未来窗工件**（尚不可派）→ 决定"派/等/预留"。两者输入集合与服务决策不同，**不共享注意力模块**（可共享底层 lot 特征 embedding 层以省参数）。
5. **轻量优先**：DDT 每步都要跑一次编码、决策频繁，层数与维度从小起步，控制推理延迟。

**消融实验（必做）。** 对 DDT 状态的未来工件表示做三档对比：

```text
手工聚合标量  vs  版本 A(self-attention)  vs  版本 B(cross-attention)
评估: 高优先级 Lot on-time 率 / qtime 残差违规率(看预留是否真起效) / DDT 推理延迟
推进顺序: 先 A 跑通并确认优于手工特征，再上 B 对比；不要一步到最复杂结构，否则归因困难
```

### 5.2 双注意力 SAS Actor（与主流对齐，保留）

Lot-PPID 自注意力分支建模候选 Lot 间的交期/qtime/优先级/等待竞争；Machine-Chamber 交叉注意力分支建模候选动作对 Machine/Chamber/Side 日历的占用压力。两分支融合后 `score_i = MLP(concat(E_lot_i, E_res_i, x_i, y_i))`，masked softmax 输出。`mask` 仍只屏蔽**结构不可行 + padding**；qtime 的机会约束 mask 在候选池生成阶段（3.2）已先行施加。Critic 共享双注意力表示，按 4.3 多头输出。

### 5.3 DDA-PPO 训练流程

共享环境、两套策略网络（DDT 与 SAS 各一，可共享底层 lot embedding，但注意力模块与 value 头分开）、各自多头 value、联合 PPO。分别记录 DDT 与 SAS transition，各自在自己的轨迹上逐目标算 GAE（终局软目标向量作为共享的团队信号，回填到两个 agent 各自的末步 transition），再用 clipped objective 更新两个 Actor 与对应多头 Critic。训练保持轨迹一致性（失败不替换采样动作）；推理可 fallback。

### 5.4 可解释性输出（保留 + 扩展）

DDT：为何派工/等待/预留；等待前后候选池质量变化；预留对 idle / qtime 风险 / 拖期风险的影响。SAS：Top-K 候选概率；两路注意力关注点；被 mask 动作及原因（区分结构 mask 与 qtime mask）；**向量 reward 各通道贡献**。

### 5.5 失败场景与替代方案

候选池常空 → 退化为规则触发，仅 SAS 选动作，暂不启用 DDT；双注意力不稳 → 退化单注意力，竞争信息作手工特征；qtime 残差难压 → 先只靠 mask，承认随机下残差违规不可完全消除；PPO 振荡 → 启发式 warm start 再微调；DDT 过复杂 → 先不引入，专注 SAS。

## 6. 实现路线与配置规格

### 6.1 分阶段实现路线

```text
阶段 1：资源日历环境 + 下层启发式估时器（1.5）+ 候选池 + 结构 mask（不训练，先把环境与估时做稳）
阶段 2：规则触发 SAS-PPO，仅即时执行通道（标量、单头）—— 验证闭环
阶段 3：加 机会约束 qtime mask + priority filter(soft) —— 验证 qtime 违规率 ≤ ε、总违规大幅下降（机制，无 reward 改动）
阶段 4：加终局向量 + 多头 Critic + 逐目标 GAE —— 验证软目标可学、权重可调
阶段 5：单注意力 → 双注意力 SAS（消融对比）
阶段 6：加 DDT（含 wait_for_reservation）；若残差违规明显，将 qtime 残差通道权重改为自适应 λ（3.3）
阶段 7：完整 DDA-PPO，small/medium/large 实例 + 公开基准评估
```

每阶段先稳定再进下一阶段；振荡时回退上一阶段，而非堆 reward 分量。

### 6.2 智能体配置规格（SAS 与 DDT 的状态/动作/奖励）

> 本节是可直接对照实现的配置规格。贯穿原则：**SAS 回答「派哪个」（以当前机台 m + 候选池为中心），DDT 回答「现在该不该派、要不要为未来预留」（以时机 + 前瞻窗为中心）**。两者职责不混，配置才干净。两个 agent 串行：DDT 选 `dispatch_now` 后才激活 SAS。

#### 6.2.1 SAS 配置

**状态（三层）**

```text
候选动作级 x_i（每个候选 (lot,ppid) 一行，共 K_action 行）:
  Lot:   due_date_urgency=(due-t_now)/H, remaining_qtime=qtime_deadline-预计开始,
         priority, waiting_time=(t_now-arrival)/H, wafer_count
  PPID:  step_count, process_time_mean(μ)/std(σ), required_chamber_side_count
  匹配:  estimated_earliest_start, finish_time_mu(μ_finish), finish_time_sigma(σ_finish),
         qtime_violation_prob=Φ((μ_finish−deadline)/σ_finish), insertion_slack,
         chamber_conflict_risk, resource_conflict_risk
机器级 machine_m_features（单行，作注意力 query）:
  next_available_time, short_window_util, scheduled_lot_count
全局级 global_features（单行，喂 Critic）:
  t_now归一化, 未完成lot占比, M_idle数, 平均利用率,
  最大qtime风险, 最高due紧迫度, 有效候选数/K_action

张量形状:
  candidate_features:[K_action, d_cand]  candidate_mask:[K_action]
  machine_features:[d_machine]           global_features:[d_global]
```
> `remaining_qtime` 与加工时间的 (μ,σ) 必须进特征（σ 让 Actor 学会对高波动工序更保守，见 2.4）。喂进 SAS 的候选池**已是 qtime-safe**（候选池生成阶段已按机会约束 mask），故 `candidate_mask` 只管结构不可行 + padding，不重复做 qtime。归一化方式与各维度数需按真实数据定。

**动作**

```text
action_index ∈ {0,...,K_action-1}  → candidate_actions[idx]=(lot,ppid) → execute(lot,m,ppid)
logits∈R^{K_action} → masked softmax（仅 candidate_mask=1 上归一化）
训练采样 / 推理 argmax 或按概率 fallback；多机同时空闲逐台决策；池空或全 mask 不跑 softmax，交回 DDT
```

**奖励（向量，R1：EXEC + QTIME/拖期残差 + UTIL + PROGRESS 四通道）**

```text
EXEC（即时）: +0.20成功 / -0.40插入失败 / -0.50 mask_invalid(防御性,正常不可达)
  wait: SAS 不主动 wait；空池/全屏蔽时 r_exec=0.0（主动等待成本归 DDT）
约束残差+软目标（终局）: r_qtime=-norm(qtime违规数/num_lots)[=拖期,硬约束残差,恒计],
  r_util=+norm(avg_util)[唯一软目标], r_progress=+norm(completed/num_lots)[利用率代理,计数制]
收口: 成功不叠加质量分量；qtime/priority 不进即时奖励；各通道独立不求和不clip；
  终局 normalize 用 running mean/std；R1 下拖期=qtime 合并为单一残差通道，无独立 tardy
Critic: 多头 V_exec/V_qtime/V_util/V_progress，逐目标 GAE，
  A=w_exec·norm(Â_exec)+w_util·norm(Â_util)+w_progress·norm(Â_progress)−w_qtime·norm(Â_qtime)
  (w_qtime 可由自适应 λ_qtime 取代，见 3.3)
```

#### 6.2.2 DDT 配置

**状态（三块）**

```text
当前态 current_features（单行）:
  t_now归一化, M_idle占比, 平均利用率, 候选池是否空, 候选有效数,
  已到达待派lot的[最高priority/最紧remaining_qtime/最高due紧迫度],
  consecutive_wait, 下一次machine释放时间
  [逐台决策粒度下] 当前目标机台 m 的特征: next_available, short_window_util,
                   支持的 recipe 类型, m 可服务的窗内高优先级 lot 数  # 让 DDT 知道在为哪台决策
未来窗态 future_summary（单行，手工聚合，消融基线）:
  窗内即将到达lot数, 窗内最高priority, 窗内最紧remaining_qtime, 最早ETA,
  窗内高优先级lot数, "若现在排满预计无法及时服务的高优先级lot估计数", 下一次到达时间
未来工件集合 → future_ctx（5.1.1 的 Transformer 编码，两个版本）:
  f_i=[priority, remaining_qtime, ETA, wafer_count, recipe_embed, required_chamber_side]
  版本A self-attention（先实现）:
    {f_i} → embedding → Encoder(1-2层,无位置编码) → attention pooling → future_ctx
  版本B cross-attention（进阶对比，建模"未来工件×当前资源"预留匹配）:
    {f_i} → self-attention → F (query)
    当前机台/资源 {m_j} → 编码 → M (key/value)
    future_ctx = CrossAttention(query=F, key=M, value=M) → attention pooling
  推进顺序: 先 A 跑通并确认优于 future_summary 标量，再上 B 对比（消融见 5.1.1）

张量形状:
  current_features:[d_cur]  future_summary:[d_fut]  future_items:[N_max,d_item]
  future_mask:[N_max]  → future_ctx:[d_ctx]
  DDT输入 = concat(current_features, future_summary, future_ctx)
```
> DDT 不看候选池逐个 (lot,ppid) 细节（那是 SAS 的活）。可共享底层 lot embedding 层，注意力模块分开。

**动作**

```text
d_t ∈ { dispatch_now, wait_next_lot, wait_next_machine, wait_delta, wait_for_reservation }
要点:
  dispatch_now 后才激活 SAS；wait_for_reservation = 有意识空出关键机台给窗内即将到的高优先级 lot
  防呆: consecutive_wait 超阈值 → mask 掉所有 wait 类，强制 dispatch_now
  可行性 mask: 未来无 lot 到达时 mask 掉 wait_next_lot 等无意义动作
```

**奖励（向量，含延迟项）**

```text
即时 TIMING 通道（防退化，注意：非"执行合法性"，与 SAS 的 EXEC 通道语义不同）:
  r_wait_cost = -0.01~-0.02 主动wait / 0.0 dispatch_now / 略松 reservation
延迟 RESERVATION 通道（核心，可先关）:
  r_reservation_hit = +R_hit 预留的高优先级lot按时被服务 / -R_miss 白等
  需环境记账并回填到当初 reservation 决策的 transition；第一版可先不做，靠 GAE 信用分配
终局: 与 SAS 共享同一套终局向量（Q-time/拖期残差 + 利用率 + 进度代理），但 DDT 有自己的多头 Critic
```
> 防双重计数：`r_reservation_hit` 与终局 Q-time/拖期残差通道会对"高优先级 lot 按时完成"这同一件好事重复奖励。二选一：要么 `r_reservation_hit` 保持**小幅 shaping** 量级（只起引导、不主导），要么干脆关掉延迟通道、只靠终局向量 + GAE 做信用分配（推荐的第一版）。
> 通道对齐：DDT 的即时通道（TIMING）与 SAS 的即时通道（EXEC）**语义不同、各自独立的 value 头**；两者仅共享终局软目标通道。联合 PPO 时按各自 transition 分别算 GAE。

#### 6.2.3 分阶段开关（配置层面）

```text
DDT 第一版（最小）: 动作只留 {dispatch_now, wait_delta}；奖励只留 r_wait_cost + 终局向量；
                    未来窗只用 future_summary 标量（不上 Transformer）
逐步加: + wait_for_reservation 动作 → + r_reservation_hit 延迟回填 → + future_ctx 编码（先 self-attention 版A，再 cross-attention 版B 对比）
注意: DDT 比 SAS 难训（核心奖励延迟稀疏），必须 SAS 稳后再加；
     大概率需规则 warm start；entropy 系数调高防坍缩到"永远 dispatch_now"
```

## 7. 与主流方法的对比及可借鉴点（新增）

> 本节基于近两年（2023–2026）公开文献，对照本项目定位，指出可借鉴之处。文献以作者/年份/出处标注，仅作技术参考。

### 7.1 主流技术地图

近年 RL 调度（含 FJSP / FAB）的主流范式可归纳为四条线：

1. **异构图神经网络（HGNN）状态表示**：把调度状态建成析取图（operation/machine 为节点，工序先后边与机器共享边为两类异构边），用 HGNN 提取特征。代表：Song et al.（2022/2023，析取图 + 异构 GNN 学习派工规则）、Tang & Dong（HGNNR，Machines 2024，关系子图分解 + 多头注意力融合）、HGT-Scheduler（异构图 Transformer，2026）。其核心论点：把"工序流依赖"与"资源竞争"两类边**分开建模**，比同质图更有表达力。

2. **注意力 / 双注意力 actor**：Wang et al.（2023）用自注意力分别抽取 machine 与 operation 特征，提出 dual-attention 网络，泛化性好。**本项目的双注意力 SAS 正属于这一线，与主流对齐。**

3. **多智能体 / 协同 RL**：把"选作业"与"选机器"或"多区域"拆给多个智能体协同（如 Nature Sci. Rep. 2025 的多目标协同 MARL、IJPR 2025 晶圆厂多区域协同 MARL）。**本项目 DDT + SAS 的双智能体结构属于此线。**

4. **面向真实 FAB 的大规模 / 自监督 + 进化策略训练**：Tassel et al.（WSC 2023）用自监督预训练 + 进化策略训练全局派工网络，在 SMT2020 测试床上优于传统层级派工规则，尤其对占多数的常规 Lot；Stöckermann et al.（2025）在真实工业数据上对比开源模型的可扩展性。该线还普遍指出**学术 RL 缺少在真实 FAB 的验证与部署**（tandfonline 文献综述 2025）。

### 7.2 本项目相对主流的定位

| 维度 | 主流常见做法 | 本项目 | 评价 |
|------|-------------|--------|------|
| 状态表示 | 析取图 + HGNN | 候选池手工特征 + 注意力 | 主流更强表达力；本项目更轻、与资源日历耦合更紧 |
| Actor | 双注意力 / 指针网络 | 双注意力 | 对齐 |
| 多目标 | 多为加权标量或单一 makespan | 向量通道 + 多头 Critic（分离尺度，非多目标权衡） | 工程更稳；R1 下软目标仅利用率，不主张多目标先进性 |
| 硬约束 | 多用惩罚，少数用 mask | qtime 机会约束 mask（z_ε·σ）+ 残差通道/λ | **本项目更严谨** |
| 动态触发 | 多为事件触发即派工 | DDT 学习"何时派 + 预留" | **本项目有创新** |
| 可行性 | 部分靠后修复 | 资源日历 dry-run/commit | **本项目工程更扎实** |
| 验证 | 缺真实/公开基准 | 待补（见 7.4） | **需补强** |

总体：本项目在**硬约束严谨性、动态决策时机、可执行性**上领先常见做法；在**状态表示的表达力**和**基准验证**上落后主流，是主要可改进项。（R1 下软目标仅利用率一项，故不把"多目标"列为强项。）

> 严谨界定（避免夸大创新）：表中"更先进/更严谨"是**相对调度领域常见做法**而言，不等于方法学上的原创。其中向量奖励 + 多头 Critic 是**从多目标 RL 借鉴的成熟做法**（合理的工程选型，非本项目首创）；双注意力 SAS 与主流（Wang 2023）**对齐而非领先**。本项目真正的差异点集中在 **有限前瞻 + set/cross-attention 驱动的主动资源预留（5.1.1）** 与 **加工时间噪声下的 Q-time 自适应机会约束处理（2.4，margin = z_ε·σ）**，正式发表前应再做一轮文献核实，确认这一组合的新颖性边界。

### 7.3 可直接借鉴的点

1. **用异构图增强状态表示（最值得借鉴）**：当前候选池手工特征难以充分表达"工序先后 × 资源竞争"的结构关系。可把 Machine / Lot / Chamber-Side 建成异构图节点，工序先后与资源共享建成两类边，用一层 HGNN 产生候选 embedding，再喂给现有双注意力打分。这是用 Song/Tang 的思路**增量增强**，而非替换框架——保留资源日历与候选池索引动作空间不变。

2. **自监督 / 启发式预训练做 warm start**：借鉴 Tassel 的自监督预训练 + ES，或更简单地用现有 NSGA-II / 派工规则生成的高质量轨迹做行为克隆预训练，再 PPO 微调，缓解稀疏终局奖励下的冷启动与振荡（对应 5.5 的 warm start 退化方案）。

3. **对标公开测试床 SMT2020**：主流 FAB 调度普遍以 SMT2020 + 传统层级派工规则为基线。即使本项目最终用自有实例，也应在 SMT2020（或其子集）上与启发式派工规则对比 on-time 率与 cycle-time，否则结果难以横向比较、可信度受限。

4. **评估口径（R1 调整）**：因可权衡软目标仅利用率一项，**不做多目标帕累托/超体积的强主张**。评估应分层报告：硬约束（Q-time/拖期违规数与违规概率是否 ≤ ε）作为首要门槛，利用率作为在满足约束前提下的优化指标，优先级违反度作为偏好满足度。若未来问题扩展出第二个真正可权衡的软目标，再引入帕累托/超体积分析。

5. **关注验证/部署 gap**：文献综述（2025）反复指出学术 RL 调度少有真实部署。本项目的资源日历 dry-run/commit 恰是面向可执行性的工程优势，应在报告中作为差异化卖点强调，并补一节"如何从仿真过渡到真实派工"。

### 7.4 建议的基准与验证

```text
基线对比：FIFO / EDD / CR / ATC 等派工规则；NSGA-II（现有）；纯启发式触发
公开测试床：SMT2020（HV/LM、LV/HM 场景），报告各优先级 Lot 的 on-time% 与 cycle-time
消融实验：单注意力 vs 双注意力；有/无 qtime mask；有/无 DDT；标量 vs 向量奖励 + 多头 Critic
分层评估：硬约束门槛（Q-time/拖期违规数、违规概率≤ε）+ 利用率 + 优先级违反度（R1 下不做帕累托/超体积强主张）
鲁棒性：不同到达强度、不同前瞻窗 W_lookahead、不同 qtime 紧度下的违规率与拖期
```

## 8. 最终结论

本项目本质上是面向 FAB 机台组的特殊并行机调度问题：在 Machine Group 内为待排产 Lot 决定加工 Machine、PPID、上/下机时刻，以及 wafer 在 Chamber/Side 上的顺序与时间，需在**工件随机到达（有限前瞻可见）与加工时间随机噪声的双重不确定性**下，满足 Q-time/拖期硬约束（随机下表述为机会约束 P(违规)≤ε）、优先级强偏好，并尽量提高机台利用率（R1 下唯一可权衡的软目标）。方案采用**两层架构**：下层用固定规则估算工件内部批处理的完成时间分布，上层用强化学习做派工与时机决策。

完善后的方案可概括为：

1. 显式建模**黑盒到达 + 有限前瞻**，把"为未来高优先级 Lot 预留资源"变为可学、可预判（窗内）+ 统计兜底（窗外）；并用 **set-attention（self-/cross-attention）编码前瞻窗工件**驱动 DDT 的预留决策（5.1.1），作为相对主流的差异点；
2. 确立 **Q-time/拖期 > priority > 利用率** 的字典序结构（R1：拖期=Q-time 违规，同一硬事件）：Q-time/拖期用**机会约束 mask**（随机加工时间下 `deadline−μ<z_ε·σ`，违规概率≤ε，+ 残差通道/λ 兜底）、priority 用候选池过滤（+DDT 预留）、利用率为唯一软目标；
3. 显式处理**加工时间不确定性**（纯工艺噪声，μ/σ 入状态）：完成时间为分布，Q-time 改写为机会约束，safety margin = z_ε·σ 自适应、有概率含义，训练时注入噪声、评价在多次随机 rollout 上统计；
4. 即时奖励**只管动作合法性**，质量信号下沉到**终局向量**，由**多头 Critic + 逐目标 GAE**分离学习，权重作用在归一化 advantage 上；
5. **两层架构**：下层用固定规则（满批组批 + list scheduling + 蒙特卡洛）估算工件内部批处理的完成时间分布 (μ,σ)，作为上层 Q-time 判断与候选池打分的地基；上层用 RL 做有取舍的派工/时机决策（1.5）；
6. Actor 用机台条件化 / 双注意力，资源日历 dry-run/commit 保证可执行性，训练保持 PPO 轨迹一致性；
7. 分阶段推进（环境+估时器 → 单头执行闭环 → mask/过滤 → 多头向量 → 注意力 → DDT → 完整 DDA-PPO）；
8. 借鉴主流的**异构图状态表示、自监督/启发式 warm start、SMT2020 基准、分层评估（硬约束门槛 + 利用率，不做帕累托/超体积强主张）**，补强状态表达力与验证可信度。

> **在不改变 FAB 机台组调度问题定义的前提下，构建一个面向"工件随机到达 + 加工时间随机"双重不确定性的两层调度框架：下层以固定规则估算工件内部批处理的完成时间分布，上层以有限前瞻、字典序约束（Q-time 机会约束 mask + priority 候选池）、双智能体动态决策（SAS 派工 + DDT 时机/预留）、双注意力动作选择与 PPO 训练为核心，并对标公开基准的智能调度原型。**
