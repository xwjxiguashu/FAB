# 面向半导体 FAB 机台组调度问题的项目报告

## 当前代码逻辑重构版

> 本报告根据仓库截至 2026-06-08 的本地代码与文档重新整理。有效实现以 `FAB_RL/FABenv/` 为准；根目录早期 NSGA-II、滚动调度、RMA/DDT 设想类文档仅作为历史背景，不作为当前功能完成状态的依据。

## 0. 报告重构说明

当前项目已经从早期“规则/启发式调度 + 未来学习式 RMA”的设想，推进为一个以资源日历为核心的两层调度系统：

1. 下层是固定规则调度器，负责 wafer 子批、stage、chamber/side 级别的真实排程与估时。
2. 上层是派工与预留决策层，当前包含 SAS-PPO 派工策略、规则基线、预留台账、ROP 候选触发器、VC-MCTS 在线预留规划器。
3. Q-time 不再只作为奖励惩罚处理，而是进入候选池 mask，且当前实现已经支持 `aggregate`、`chain`、`chain_joint` 三种 Q-time mask 模式。
4. VC-MCTS 已替代学习式 RMA，成为当前预留层的主线方案；SAS policy 可作为 dispatch delegate，也可作为 MCTS 的策略先验；多头 critic 可作为叶子价值估计，但这些 AlphaZero 式增强默认关闭。

因此，本报告不再按“规划中的 RMA 智能体”叙述，而是按当前代码真实数据流组织：问题建模、资源日历、候选池、SAS-PPO、Q-time 链约束、预留 ledger、VC-MCTS、实验验证与未完成边界。

## 1. 项目定位与问题定义

本项目研究半导体 FAB 机台组调度。给定一批 Lot、若干 Machine、每台机器可执行的 PPID、每个 PPID 对应的多 stage 工艺步骤，以及 stage 内可选择的 chamber/side 资源，系统需要在动态到达和加工时间噪声下决定：

- 当前哪个 Machine 应被处理；
- 在该 Machine 上派哪个 Lot；
- 使用哪个 PPID；
- 是否为了即将到达的高优先级 Lot 暂时预留 Machine；
- 派工后 wafer 子批如何在 chamber/side 上落入真实资源日历。

当前代码把约束和目标分成三层字典序结构：

| 层级 | 含义 | 当前处理方式 |
|---|---|---|
| Q-time | stage 间材料等待窗口，硬约束优先 | 候选池 qtime mask + terminal/Q-time reward channel + PPO-Lagrangian |
| Priority | 高优先级 Lot 应更早获得产能 | priority filter、O2 priority-weighted wait、VC-MCTS reserve |
| Utilization | 机器利用率与吞吐表现 | 候选评分、SAS policy、VC-MCTS tie-break |

代码中的关键结论是：Q-time 和 priority 不应只靠一个加权 reward 混合。Q-time 首先应从动作空间层面移除明显危险动作；priority 需要通过候选过滤和预留规划提前占产能；只有剩余软目标才交给评分或策略网络做权衡。

## 3. 数据建模

### 3.1 静态问题对象

`Phase1CalendarProblem` 继承 `ProblemDefinitionMixin` 与 `CalendarDecoderMixin`，是当前系统的统一 problem encoder。它包含：

- `num_lots`、`num_machines`；
- `feasible_machines[lot]`；
- `feasible_ppids[(lot, machine)]`；
- `ppid_steps[(lot, machine, ppid)]`，每个 stage 是若干 `[chamber, side, process_time]` 备选资源；
- `wafer_counts[lot]`；
- `arrival_times[lot]`；
- `due_dates[lot]`；
- `priorities[lot]`；
- `qtime_deadline[lot]`；
- `q_time_limits[(lot, machine, ppid, from_stage, to_stage)]`；
- `process_time_sigma[(lot, machine, ppid)]`；
- `machine_resources[machine]`；
- `side_capacity`。

### 3.2 实验实例

当前主要实例包括：

| 实例 | 构造函数 | 用途 |
|---|---|---|
| `small` | `build_small_encoder()` | 4 Lot、2 Machine，小规模回归测试 |
| `pressure` | `build_pressure_test_encoder()` | 50 Lot、10 Machine、3 stage、随机优先级压力实例 |
| `late_hi` | `build_late_hi_encoder()` | 与 `pressure` 同骨架，但 priority 与 arrival 高度正相关，即高优先级 Lot 倾向晚到 |

`pressure` 和 `late_hi` 的共同结构是：50 个 Lot、10 台 Machine、每台机器 5 个 PPID、每个 Lot 10 片 wafer、5 个 chamber、2 个 side、3 个 stage。到达时间由指数间隔累加生成；stage 间 Q-time 在 `(1,2)` 与 `(2,3)` 两条链路上登记。

`late_hi` 是验证预留机制的关键实例。只有当高优先级 Lot 可能较晚到达时，“当前机器空闲但暂时不派，等待未来高优先级 Lot”才有现实收益；若 priority 与 arrival 无关，预留收益会被实例结构稀释。

### 3.3 输出 schema

当前调度输出保持两个层级：

```text
lot_schedule:   (n, 5) [lot, machine, ppid, start_time, end_time]
wafer_schedule: (n, 9) [lot, wafer_id, machine, ppid, stage_id, chamber, side, start_time, end_time]
```

下层真实调度按 wafer 子批处理。同一子批内多个 wafer 共享同一个 `(chamber, side, start, end)`，但 `wafer_schedule` 仍保留逐 wafer 行，以保证输出 schema 不变。因此验证日历时必须对重复 wafer 行去重，避免把同一子批的同一资源区间误判为冲突；这一点已在 `validate_schedule()` 与 `validate_final_schedule_completeness()` 中处理。

## 4. 两层调度架构

### 4.1 下层：固定规则的 wafer 子批调度

下层负责在一个确定的 `(lot, machine, ppid)` 已经被上层选中之后，安排该 Lot 内部 wafer 如何通过多个 stage。

核心逻辑在 `lower_layer_estimator.py` 和 `lower_layer_scheduler.py`：

1. `compute_sub_batches(n_wafers, side_capacity)` 将 wafer 按 side capacity 分成子批。
2. `schedule_deterministic(...)` 是唯一确定性 list scheduling 核心。
3. `estimate(...)` 用空闲日历调用该核心，得到状态无关的完成时间分布。
4. `schedule_on_calendar(...)` 读取已提交 calendar 的 free time，再调用同一个核心，得到状态相关、绝对时间、非破坏式排程结果。

这一设计修复了早期环境内部重复实现 batch scheduling 的问题。现在 mask 估时和正式 commit 使用同一套 list scheduling 核心，减少了“预测可行但落地不可行”或“预测完成时间与真实完成时间不一致”的风险。

### 4.2 估时与真实排程的差异

| 接口 | 是否读真实日历 | 是否破坏状态 | 用途 |
|---|---:|---:|---|
| `estimate()` | 否 | 否 | Q-time mask、候选评分、规则基线估计加工时间 |
| `schedule_on_calendar()` | 是 | 否 | dry-run、commit 前可行性检查、正式插入前生成区间 |
| `commit_action_index()` | 是 | 是 | 将选中动作写入真实 state 与 schedule |
| `rollback_last_commit()` | 是 | 是 | 撤销最近一次 commit |

`estimate()` 支持缓存，键为 `(lot, machine, ppid, n_mc)`。由于基础 makespan 分布不读当前 state，`start_offset` 只在返回时平移 `mu_finish`，缓存可以跨同一 episode 复用。`ResourceCalendarEnv` 持有 `_estimate_cache`，在 `reset()` 时清空。

### 4.3 加工时间噪声

加工时间噪声来自 `process_time_sigma`。估时阶段通过 Monte Carlo 得到 `(mu_finish, sigma_finish)`；真实 commit 可在 `process_noise_enabled=True` 时使用 `noise_rng` 为每个子批、每个 stage 采样实际加工时间。评估时每个 seed 表示一次加工噪声实现。

## 5. 资源日历环境

`ResourceCalendarEnv` 是当前项目最核心的环境对象。它维护：

- 当前仿真时间 `current_time`；
- 未完成 Lot 集合与已完成 Lot 集合；
- `ScheduleState` 中的 machine/chamber calendar；
- 当前 `lot_schedule` 与 `wafer_schedule`；
- commit/rollback 所需日志；
- 候选池特征名、Q-time mask 模式、priority filter 模式等配置。

### 5.1 资源日历

`CalendarDecoderMixin` 提供：

- `add_calendar_interval()`：按 start/end 排序插入资源区间，若重叠则抛错；
- `remove_calendar_interval()`；
- `rollback_calendar_intervals()`；
- `find_earliest_slot()`；
- `validate_no_interval_overlap()`；
- `calendar_busy_time()`。

machine 日历记录整台机的占用；chamber 日历记录 `(machine, chamber, side)` 级资源占用。正式派工必须同时向 machine calendar 和 chamber calendar 写入区间。

### 5.2 dry-run、commit 与 rollback

`dry_run_action()` 在 state 副本上试算动作，不修改真实环境；`commit_action_index()` 使用同源调度器生成真实区间并写入 state、schedule 和 commit log；`rollback_last_commit()` 根据 commit log 撤销最近一次提交。

这一点是 VC-MCTS 可用的前提：MCTS 每条边都要复制 driver/ledger 做 rollout，任何评估分支都不能污染真实 episode。

## 6. 候选池与三层字典序约束

### 6.1 候选池结构

对每个待决策 Machine，`build_candidate_pool(machine)` 输出一个固定大小的 `CandidatePool`：

- `actions`：真实 dispatch 动作、可选 wait 动作、padding 动作；
- `action_mask`：布尔 mask，`True` 表示可选，`False` 表示 padding 或非法；
- `features`：形状为 `(top_k, 18)` 的候选特征矩阵；
- `invalid_reasons`：被排除动作的原因；
- `no_action_available`：当前无真实候选。

18 维候选特征包括：

```text
is_real, is_wait, score, arrival_time, waiting_time,
machine_slot_start, machine_load, total_process_time,
predicted_completion, stage_count, qtime_risk, wafer_count,
priority, due_slack, priority_rank_norm, due_slack_rank_norm,
is_best_priority, is_most_urgent_due
```

固定动作维度让策略网络不必处理可变长度动作空间；padding 由 mask 排除。

### 6.2 候选池流水线

当前流水线为：

```text
结构可行候选
    -> Q-time mask
    -> priority filter
    -> score 排序
    -> Top-K
    -> wait/padding
    -> rank features
```

顺序不可交换。Q-time 是硬层；priority 只能在 Q-time 安全集内起作用；score 只负责剩余软权衡。

### 6.3 Q-time mask

当前 `qtime_safe_mask()` 支持三种模式：

| 模式 | 逻辑 |
|---|---|
| `aggregate` | 使用 `estimate()` 得到总完成时间分布，判断 `deadline - mu_finish < z_eps * sigma_finish` 是否危险 |
| `chain` | 对每个候选做一次非破坏 dry-run，用真实 `compute_q_time_violation()` 检查 stage 链窗口 |
| `chain_joint` | 对每个候选做多次带噪 dry-run，估计“任一链路窗口违反”的联合概率 |

`aggregate` 是轻量近似，但可能漏掉 `(1,2)`、`(2,3)` 这种真实 stage 间窗口；`chain` 与 `chain_joint` 更贴近最终报废判定。代码中特别处理了 `is_doomed(lot)`：如果某 Lot 即使最乐观立即开工也已经无法满足 qtime deadline，则不因它被 mask 掉，避免全候选死锁。

### 6.4 Priority filter

`priority_filter()` 支持：

- `soft`：默认模式，不删除候选，只通过特征和排序给策略提供信息；
- `strict`：只保留 priority 接近最高值的候选，使策略物理上不能选择低优先级动作。

训练与评估入口都提供了 `--priority-mode soft|strict` 和 `--priority-min-gap`，确保规则基线与 RL 使用同样候选空间。

## 7. SAS-PPO 派工层

SAS 的含义是 Single-Agent Scheduling。在当前代码中，SAS 不负责主动等待或预留，它只在一个 Machine 的候选池已构造后，从有效候选中选择 dispatch action。

### 7.1 观察编码

`Phase2ObservationEncoder` 把候选池和全局上下文编码为 `Phase2Observation`。

基础全局特征为 9 维：

```text
current_time, completed_ratio, remaining_ratio,
machine_id_norm, machine_busy_time, valid_action_count_norm,
score_mean, waiting_time_max, due_slack_min
```

当 `lookahead=True` 时扩展为 13 维，额外包含：

```text
upcoming_count_norm, lookahead_max_priority,
lookahead_min_qtime, lookahead_earliest_eta
```

### 7.2 策略网络

`phase2_sas_policy.py` 实现两类策略：

1. `Phase2SASActorCritic`
   - 候选特征 18 维经两层 MLP 编码；
   - actor 对每个候选输出一个 logit；
   - critic 对有效候选做 masked pooling，再拼接全局特征输出标量 value。

2. `Phase2SASMultiHeadActorCritic`
   - actor 结构相同；
   - critic 分成 `exec`、`qtime`、`util` 三个 value head；
   - 每个 channel 单独估值，训练时再按权重或 Lagrangian 合并 advantage。

所有策略采样都通过 `MaskedCategoricalPolicy`，无效候选的 logit 被置为极小值，因此不会被采样。

### 7.3 Episode driver

`Phase2EpisodeDriver` 负责完整 episode：

1. 找到可派工机器；
2. 选择下一台机器，规则是 earliest available -> real candidate count -> machine id；
3. 构造候选池和观察；
4. 执行规则动作、policy 动作或 greedy 动作；
5. 在无可派工机器时推进到下一事件；
6. 检查终止条件。

支持的规则基线为：

```text
first_valid, FIFO, SPT, EDD, CR, ATC
```

所有规则都在同一个 qtime-safe + priority-filtered 候选池上排序，保证基线对比公平。

### 7.4 奖励设计

标量 reward 由 `RewardConfig` 控制，默认 wait penalty 为 0，因为 SAS 不拥有主动等待决策。执行层包括：

- 成功插入：`+0.20`；
- 插入失败：`-0.40`；
- mask invalid/padding：`-0.50`；
- wait/no-op：默认 `0.0`。

可选轻量 shaping 包含 tardy、qtime、priority、progress；可选 terminal reward 包含 tardiness、Q-time、utilization、priority violation。

多头 reward vector 由 `RewardVectorConfig` 与 `compute_sas_reward_vector()` 输出，当前核心 channel 为：

- `exec`：即时执行反馈；
- `qtime`：终局硬约束残差；
- `util`：终局利用率软目标。

### 7.5 PPO-Lagrangian

`MultiHeadPPOTrainer` 支持把 Q-time 残差视为 CMDP 约束。当 `use_qtime_lagrangian=True` 时，qtime channel 的权重由固定 `w_qtime` 改为自适应拉格朗日乘子 `lambda_qtime`：

```text
lambda <- clip(max(0, lambda + lr_lambda * (violation - budget)))
```

这样可将 Q-time 约束从“手调惩罚权重”改成“违反率超预算则自动提高约束价格”。当前这部分已实现并接入 CLI，但它是训练期约束压力，不替代候选池 qtime mask。

### 7.6 并行 rollout

由于瓶颈在 CPU 环境仿真而非神经网络，GPU 对端到端训练帮助有限。`parallel_rollout.py` 通过多个 worker 并行采集 multihead episode，然后合并到一个 `MultiHeadRolloutBuffer` 做一次 PPO 更新。每个 worker 会限制 BLAS/OpenMP 线程，避免 CPU 过度抢占。

## 8. 预留层：从 RMA 转为 VC-MCTS

早期报告中的学习式 RMA/DDT 预留智能体，当前已被搜索式 VC-MCTS 替代。这个替代是合理的：预留收益稀疏、类别不平衡、与派工奖励耦合，直接训练一个 reserve/no-reserve agent 容易退化为“不预留”。当前代码选择把预留变成可模拟比较的搜索分支。

### 8.1 Reservation ledger

`ReservationLedger` 是一个小型内存台账，记录：

```text
machine, future_lot, eta, created_at, expires_at, reason
```

它只负责登记、查询、释放、过期和兑现，不负责判断某次 reserve 是否值得。一个 future lot 不允许同时被多台机器预留。

### 8.2 ROP 机会检测

`detect_reservation_opportunities()` 是宽触发器，只负责提出值得搜索的 `(machine, future_lot)`：

- 只看 lookahead 窗内 upcoming lot；
- 跳过已预留 machine 和已预留 lot；
- 检查 future lot 是否能在该 machine 上加工；
- 比较 future priority 与当前候选池最高 priority；
- 用 priority gap、机器稀缺度、ETA 距离给机会点打分；
- 取 Top-B。

ROP 不直接做决策。真正的 reserve/dispatch/no-op 比较由 oracle 或 VC-MCTS 负责。

### 8.3 DispatchDelegate

`dispatch_delegate.py` 把“选哪个候选 dispatch”与“是否预留/等待”解耦。

当前有两种 delegate：

- `RuleDispatchDelegate(strategy)`：包装 FIFO/SPT/EDD/CR/ATC 等规则；
- `SASPolicyDispatchDelegate(policy, fallback_delegate)`：用训练好的 SAS policy 选动作，失败时可回退规则。

VC-MCTS 通过 `use_delegate_dispatch=True` 复用 delegate，因此不需要在 planner 内部写死派工策略。

## 9. VC-MCTS 在线预留规划器

### 9.1 根节点动作

`VCMCTSPlanner.build_root_actions()` 在每个决策点构造三类边：

| 动作 | 含义 |
|---|---|
| `dispatch` | 现在派一个候选 Lot/PPID |
| `delegate_dispatch` | 由 dispatch delegate 选一个派工动作 |
| `reserve` | 将当前 Machine 预留给某个 future Lot |
| `no_op` | 不派工，推进时间到下一事件 |

默认会始终加入一个 `no_op` 边；dispatch 边取候选池 top-k；reserve 边来自 ROP Top-B。

### 9.2 分支评估

每条边评估时都复制当前 driver 与 ledger：

```text
clone_driver_for_rollout()
clone_ledger_for_rollout()
apply_action()
run_rule_episode_with_reservations()
schedule_metrics_with_priority_wait()
```

因此搜索分支不会破坏真实 episode。reserve 分支会在分支 ledger 中登记预留；当目标 lot 到达且候选池允许时，优先兑现；过期则释放。

### 9.3 目标字典序

`VCMCTSObjective` 包含：

```text
qtime_violation_count
qtime_violation_total
priority_weighted_wait
avg_utilization
```

最终比较顺序是：

```text
Q-time 违反次数 -> Q-time 累计超时 -> O2 优先级加权等待 -> 平均利用率
```

其中 O2 定义为：

```text
O2 = sum_lot priority(lot) * max(0, first_start(lot) - arrival(lot))
```

越小越好。它把“高优先级 Lot 不应长时间等待”变成可比较的数值目标。

`no_op` 还有保护栏：只有当 no-op 在 Q-time 指标上严格优于最好非 no-op 动作时，最终才允许选择 no-op，否则避免 planner 退化为无意义空等。

### 9.4 PUCT 与先验

当前选择边时使用：

```text
objective_score(mean) + exploration_c * prior * sqrt(log(total_visits) / visits)
```

默认 prior 来自启发式 score。若 `prior_source="policy"` 且传入 `SASPolicyPriorProvider`，则 dispatch/no-op 的 prior 可由 SAS policy 的 masked softmax 改写；reserve 边会注入固定探索 prior，因为 SAS policy 不直接建模 reserve。

### 9.5 AlphaZero 式增强

`vc_mcts_alphazero.py` 提供两个默认关闭的增强：

1. `SASPolicyPriorProvider`
   - 将 SAS policy 在候选池上的概率分布作为 MCTS root prior。

2. `MultiHeadCriticLeafValue`
   - 用多头 SAS critic 的 `qtime` 与 `util` channel 估计叶节点剩余价值；
   - O2 和 qtime_total 仍来自部分 rollout 指标。

这两项目前是可选接口，不改变默认 VC-MCTS 行为。它们为后续从 root-level rollout 比较升级到更完整的 AlphaZero/MCTS 提供挂载点。

### 9.6 到达概率加权

`arrival_prob_weighting=True` 时，reserve 边的收益会按 ETA 距离折扣：

```text
p = exp(-arrival_prob_decay * (eta - now) / lookahead_window)
```

然后把 reserve 分支 objective 与最佳非 reserve 分支做概率混合。目的在于避免远 ETA 的 future lot 被过度乐观地预留。

## 10. 可解释性与 trace

VC-MCTS 每次决策可写 JSONL trace。`VCMCTSDecisionTrace` 会记录：

- 当前时间；
- 当前 machine；
- 被选中的 action；
- 每条 root edge 的 visits；
- 每条 edge 的 mean qtime、mean qtime_total、mean O2、mean util；
- reserve 是否可用、是否被选中；
- reserve/dispatch/no-op 各自最佳指标。

`vc_mcts_trace_summary.py` 可从 JSONL 中汇总：

- reserve 机会出现次数；
- reserve 被选中次数；
- reserve selection rate；
- reserve 对 O2 和 Q-time 的改善/退化；
- reserve 输给 dispatch/no-op 的原因。

这使当前预留层比学习式 RMA 更容易解释：每次 hold 都能回溯到“哪个 future lot、哪个 machine、哪条边的 rollout 指标更好”。

## 12. 当前实证结果

以下结果来自 `FAB_RL/FABenv/results/` 中近期归档结果。数值按本地文档记录整理。

### 12.1 SAS-PPO 派工层

`late_hi`，8 个加工噪声 seed，所有策略使用相同候选池：

| strategy | Q-time 违反数 | 总拖期 | priority violation | 平均利用率 | 完成率 |
|---|---:|---:|---:|---:|---:|
| FIFO | 12.75 ± 5.65 | 0.00 | 4304.01 | 0.83 ± 0.02 | 100% |
| SPT | 4.50 ± 1.66 | 0.00 | 4304.01 | 0.83 ± 0.04 | 100% |
| EDD | 12.75 ± 5.65 | 0.00 | 4304.01 | 0.83 ± 0.02 | 100% |
| CR | 33.50 ± 6.84 | 0.00 | 4304.01 | 0.88 ± 0.01 | 100% |
| ATC | 5.75 ± 2.73 | 0.00 | 4304.01 | 0.86 ± 0.02 | 100% |
| SAS-PPO | 1.75 ± 2.11 | 0.00 | 4304.01 | 0.87 ± 0.00 | 100% |

结论：SAS-PPO 在 Q-time 违反数上显著优于经典规则，同时保持较高利用率。但 priority violation 在该指标定义下对所有策略为常数，区分能力不足；当前更有价值的 priority 评价是 O2。

### 12.2 链感知 Q-time mask

阶段性汇报中的隔离实验显示，在 FIFO 派工、无 VC-MCTS 的条件下，把 aggregate proxy mask 改为链感知 mask，可将 Q-time 违反均值从 13.6 降到 0.4，最坏 seed 从 26 降到 2，累计超时均值从 11.25 降到 0.14，利用率从 0.821 轻微降到 0.812。进一步开启 `chain_joint` 可清掉剩余噪声尾部。

结论：当前 Q-time 问题的主要来源不是策略网络不够强，而是 mask 如果只看聚合 deadline，会漏掉真实 stage 链窗口。`chain`/`chain_joint` 让筛选口径与最终报废口径对齐。

### 12.3 VC-MCTS 预留层

`late_hi`，5 个加工噪声 seed，轻预算 VC-MCTS 参数：`n_iter=2`、`rollout_max_steps=60`、`max_decisions=120`、`top_b=3`、`top_k_dispatch=2`。

| 策略 | Q-time 违反 | O2 priority-weighted wait | O2 vs FIFO | 平均利用率 | 完成率 |
|---|---:|---:|---:|---:|---:|
| FIFO | 13.6 ± 6.9 | 1612.5 ± 83.9 | baseline | 0.820 ± 0.020 | 50/50 |
| ATC | 7.2 ± 3.2 | 1864.3 ± 47.0 | +15.6% | 0.870 ± 0.030 | 50/50 |
| SAS-only | 5.6 ± 2.3 | 1685.3 ± 23.6 | +4.5% | 0.860 ± 0.010 | 50/50 |
| VC-MCTS + rule | 0.0 ± 0.0 | 1245.6 ± 112.4 | -22.8% | 0.830 ± 0.027 | 50/50 |
| VC-MCTS + SAS | 0.4 ± 0.8 | 1335.8 ± 157.4 | -17.2% | 0.797 ± 0.038 | 50/50 |

在后续 `late_hi_mh.pt` 重新训练后，VC-MCTS + SAS 的 O2 进一步达到约 1240.4，基本追平 rule delegate；但方差更大，个别 seed 仍有 Q-time 残留。因此当前最稳的是 `VC-MCTS + rule`，完整学习融合版已经可行但仍需提升鲁棒性。

### 12.4 pressure 泛化验证

在随机 priority 的 `pressure` 实例上，VC-MCTS + rule 也有效：

| 指标 | baseline | VC-MCTS |
|---|---:|---:|
| O2 | 885.7 | 678.2 |
| Q-time 违反 | 13.6 | 1.2 |
| 平均利用率 | - | 0.803 |

结论：预留层收益并不只来自特意构造的 `late_hi`；在随机 priority 压力实例上，有限前瞻预留同样可降低 O2 与 Q-time。但 `late_hi` 更适合作为 go/no-go 实例，因为它更明确地暴露“高优先级晚到”带来的预留必要性。

- SAS prior 和 critic leaf value 已实现接口，但默认关闭，尚需系统性 ablation；

- priority 的旧 `priority_violation` 指标在当前实例上区分能力不足，O2 已更有解释力，但训练 reward 与 O2 尚未完全统一；
- 当前实例为合成实例，尚未接入真实 FAB 数据；
- GPU 训练不是瓶颈，性能提升主要依赖 CPU 并行和候选池/估时缓存优化；
- checkpoint 训练与恢复仍偏轻量，周期保存已支持，但 relaunch 不自动续训；
- `chain_joint` 更严谨但计算更重，需要与 VC-MCTS 预算共同调参；
- arrival probability weighting 目前是最小 A/B 机制，仍需在更多 seed 上验证。

## 15. 后续工作建议

1. 把 `chain_joint` 与 VC-MCTS 预算做联合消融：分别测试 aggregate、chain、chain_joint 对 Q-time、O2、运行时间的影响。

5. 把真实 FAB 数据或更接近真实的合成数据接入 `Phase1CalendarProblem`，验证模型在非均匀机台能力、复杂 recipe、不同 arrival/load pattern 下的泛化。

7. 统一 priority 评价体系：保留 O2 作为核心指标，并弱化或重定义当前常数化的 `priority_violation`。

## 16. 总结

当前项目已经形成一条清晰、可执行、可验证的技术路线：

```text
固定规则下层调度器
    -> 资源日历环境
    -> qtime-safe + priority-aware 候选池
    -> SAS-PPO 派工
    -> ReservationLedger + ROP
    -> VC-MCTS reserve/dispatch/no-op 在线规划
    -> 多 seed baseline / trace / O2 / Q-time 评估
```

项目的核心创新不在于简单把 RL 用到调度，而在于把半导体 FAB 调度中不同强度的要求拆开处理：Q-time 用候选空间硬筛和 Lagrangian 补尾，priority 用预留规划提前分配产能，utilization 留给评分和策略网络优化。当前代码已经证明，链感知 Q-time mask 可以把报废风险压到近零，VC-MCTS 预留可以在 `late_hi` 与 `pressure` 实例上显著降低 O2，并且与 SAS 派工层能够通过 delegate 和 prior/leaf value 接口融合。

从完成度看，当前最稳的可报告版本是“链感知候选池 + 多头 SAS-PPO + VC-MCTS 预留规划器”；从下一步研究看，重点应转向 O2-aware SAS 训练、AlphaZero 式搜索加速、真实数据接入和更系统的 ablation。
